# blinky_cli — флагманский пример МК (MCU_PLAN §6)

Суперцикл реального embedded-проекта: мигание LED по таймеру, кнопка,
командный интерфейс по UART. Логика — на EATLang со всеми гарантиями
(тотальность, Power of 10, отчёт §8), граница с железом — extern-шим
[mcu/Mcu.eat](../../mcu/Mcu.eat): `gpio_set`/`gpio_get`/`systick_ms`/
`uart_poll`. Один исходник собирается на **все** порты `mcu/boards/`
и на хост.

## Команды (байт в UART)

| Команда | Действие |
| --- | --- |
| `o` / `f` | LED вкл / выкл (выкл также гасит мигание) |
| `b` | переключить мигание (период 500 мс) |
| `s` | статус: режим мигания, счётчик команд, кнопка |
| `t` | аптайм в миллисекундах |
| `i` | номер такта суперцикла |
| `q` | выход (на плате — завершение через полухостинг) |

## Сборка

```sh
# хост (драйвер-эталон host_driver.c поверх stdin/часов):
make verify_mcu                       # соберёт и сверит всё сам

# любой порт mcu/boards/:
make mcu BOARD=microbit SRC="--lib . examples/blinky_cli/BlinkyCli.eat"
make mcu_run BOARD=microbit SRC="--lib . examples/blinky_cli/BlinkyCli.eat"

# реальная плата (см. mcu/README.md, нужен st-flash/picotool/nrfjprog):
make mcu_flash BOARD=bluepill SRC="--lib . examples/blinky_cli/BlinkyCli.eat"
```

## Автосверка

Вывод программы зависит **только** от последовательности команд:
мигание и время в канал не печатаются, поэтому QEMU-платы сверяются с
хостовым эталоном байт-в-байт (`make verify_mcu`, вход —
[cmds.txt](cmds.txt)). Команды `t` и `i` зависят от времени и в
автосверку не входят — это ручные замеры: `i`/`t` даёт латентность
такта суперцикла (числа — в tests/bench/FINDINGS.md).

Вход обязан заканчиваться `q`: суперцикл — единственный разрешённый
бесконечный `loop` в `main` (SPEC §5.1), других выходов у него нет.
