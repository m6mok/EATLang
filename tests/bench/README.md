# Нагрузочное тестирование EATLang

Запуск из корня репозитория:

```sh
make bench          # полный прогон (~2–5 минут)
make bench_quick    # быстрый прогон (~20 секунд)

# отдельные секции:
uv run python tests/bench/bench.py --only stress,selfhost
```

Артефакты (сгенерированные входы, бинарники, дампы) — в `build/bench/`.
Ненулевой код возврата — есть провалы; список печатается в секции ИТОГО.

## Секции

**pipeline** — скорость стадий Python-бутстрапа (`eatc lex/parse/typed/ir`)
на синтетических модулях ступенчатых размеров (S/M/L, `genprog.py`) плюс
многомодульная программа на 992 функции — рядом с пределом языка в 1024
функции (SPEC.md §6), больше программа физически не бывает. Ступени
позволяют увидеть асимптотику: время должно расти линейно с числом
токенов; сверхлинейный рост между ступенями — сигнал о квадратичном
алгоритме в компиляторе. Замер `ir` включает Rt.eat и llvmlite.

**runtime** — программы `programs/*.eat`: арифметика, вызовы функций,
массивы, структуры, строки, а с 2026-07-14 — метрики слоёв плана
оптимизаций (по одной программе на слой, см. docs/COMPTIME_PLAN.md §0
и TRACKS.md):

- `U64Bench`/`I64Bench` — 64-битная арифметика (виды 13/14), деление,
  широкие сдвиги; на ARM32 уходит в хелперы eabi64;
- `BitsBench` — битовое месиво с пре-маскированными `<<` и сдвигом на
  переменную величину (цель «ширина маски ↔ величина сдвига», трек 3);
- `TrapBench` — цена недоказанных проверок: индексы/делители/касты из
  пулов (крупнейший класс остатков каталога VERIFICATION_PLAN);
  должен ускоряться от прогресса верификатора без правок кодогена;
- `MatchBench` — enum + исчерпывающий match (диспетчер машин selfhost);
- `RetBench` — возврат агрегата (sret во временную alloca + memcpy —
  кандидат «sret прямо в слот назначения»);
- `CopyBench` — memcpy-копии массива при связывании (класс `lb_tp`
  selfhost, трек 4);
- `IoBench` — пропускная способность вывода: write_span, write_byte,
  print-интерполяция (метрика слоя «runtime.c с -O2» и порога батчинга).

Вторая волна (тоже 2026-07-14) — непокрытые пути:

- `ReadBench` — путь ввода read_byte/Result (половина профиля selfhost);
  отдельный контур: нагрузка — размер stdin (генерируется раннером),
  а не REPEAT;
- `StrCmpBench` — сборка строки интерполяцией + `==` с общим префиксом
  40 байт (скан Rt.eq — кандидат break-развёртки/векторизации);
- `BankBench` — банкованный пул `p[a/4096][a%4096]` (идиома памяти
  selfhost) с зависимым от acc адресом;
- `SortBench` — сортировка вставками: data-dependent свопы; результат
  потребляется взвешенной свёрткой (иначе LLVM оставляет min/max);
- `BranchBench` — 4-плечевой if/elif-диспетчер по битам acc (профиль
  эмулятора 6502);
- `MachineBench` — конечный автомат c var self (идиома машин selfhost;
  класс «var self убивает факты» верификатора, трек 3).

Каждая исполняется интерпретатором и
собирается в бинарник; вывод (контрольная сумма) обязан совпасть байт
в байт — дифференциальная проверка бэкендов. Нативная скорость меряется
на утяжелённом варианте: раннер подменяет `const REPEAT: u32 = 1` на
множитель, чтобы уйти от шума таймера и цены запуска процесса.
Контрольные суммы построены так, чтобы не вырождаться: свёртка
несимметрична, а проходов по массиву 127, потому что m^128 ≡ 1 (mod 2^16)
обнуляет пул при любом нечётном множителе.

**stress** — пары входов «на пределе / за пределом» для каждого лимита
SPEC.md §6: токены в файле, функции в программе, операторы в функции,
глубина блоков, глубина выражения. Компилятор обязан принять предельный
вход и быстро (< 60 с, без зависания) отказать с внятной ошибкой на
запредельном. Эмпирические пороги: 31 скобка проходит, 32 — уже нет
(глубину добавляет сам литерал внутри скобок).

**selfhost** — нативные `build/SelfLex` и `build/SelfParse` против
Python-эталона: пропускная способность (токенов/с) и байт-в-байт сверка
дампов. Вход — самая большая ступень пайплайна, которую принимают пулы
self-hosted компилятора: они меньше лимитов эталона (парсер отказывает
уже на ~65 536 узлах AST — L-ступень не проходит, M проходит); отказ
`err: ...` раннер фиксирует строкой отчёта и спускается на ступень ниже.
Если бинарники не собраны, полный режим собирает их сам (это тоже
нагрузочный тест бутстрапа), быстрый — пропускает секцию.

**compiler** — постоянная метрика скорости самого компилятора: все пять
фазовых бинарников (`SelfLex`, `SelfParse`, `SelfSig`, `SelfTyped`,
`SelfIr`) прогоняются на конкатенации собственных модулей компилятора —
том же входе, что у `make verify_bootstrap` (~385 КБ, ~90 000 токенов).
Для каждой фазы: время нативного бинарника, время Python-эталона на той
же стадии, отношение, размер вывода и байт-в-байт сверка дампов. Разница
времён между соседними фазами атрибуцирует стоимость фазы (parse − lex,
typed − parse, ir − typed). Устаревший относительно `selfhost/*.eat`
бинарник не замеряется (он молча печатает `err:` в stdout): полный режим
пересобирает его, быстрый — пропускает с подсказкой. Любая оптимизация
selfhost сверяется по этой секции «до/после».

## Калибровка

Шаблон синтетической функции (`genprog.gen_func`) — ~757 токенов,
54 оператора: с запасом внутри лимитов (60 операторов, глубина
выражения 32). Базовые порции бенчмарк-программ подобраны под ~2–3 с
в интерпретаторе; множители REPEAT — под ~0,5–5 с в бинарнике.
