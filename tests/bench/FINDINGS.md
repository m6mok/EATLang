# Выводы нагрузочного тестирования — задание на исправления и оптимизации

Дата замеров: 2026-07-12, Apple Silicon (darwin 25.3.0), Python 3.11.14,
полный прогон `make bench` (~65 с) + ручные эксперименты ниже. Каждый
вывод содержит доказательство, корневую причину с точным местом в коде
и рецепт проверки. Перед правками зафиксируйте базу: `make bench`
зелёный; после каждой правки он обязан оставаться зелёным (в нём
дифференциальные сверки interp == native и self-hosted == эталон).

## Приоритеты

## Статус исправлений (2026-07-12, вторая итерация)

- **F1 — сделано.** `compile_binary` гоняет O2-пайплайн нового pass
  manager'а llvmlite перед `emit_object` (`.ll` пишется до оптимизации —
  канон не тронут), target machine с `opt=2`; в Makefile `link` и
  `eatc-self` добавлен `clang -O2`. Бинарники: Arith 480→860M оп/с,
  Str 2.2→10.6M оп/с; self-hosted лексер из x0.8 стал x2.2 от эталона,
  парсер x2.5. Фикспойнт и все сверки зелёные.
- **F3 — сделано.** Пулы узлов парсера и nty/nrt тайпчекера банкованы
  по идиоме лексера (`[[u32; 65536]; 2]`, адресация `n / 65536`,
  `n % 65536`) — массив >65536 элементов запрещён самим языком, поэтому
  «поднять до 131072» = два банка. Контракты узлов подняты до 131072,
  циклы-обходы удвоены. Лимит узаконен: `MAX_AST_NODES = 131072`
  (SPEC §6, limits.py, счётчик аллокаций в Python-парсере). Вход L
  (122 721 токен) теперь парсится self-hosted'ом байт-в-байт с эталоном.
  Кадр main self-hosted компилятора 110 МБ — в стек 128 МБ влезает.
- **F4 — сделано** (×1.6–1.9 вместо надежды на 2–5): точечный
  `_copy_value` вместо deepcopy (скаляры без копии), диспетчеры
  `type(node) → метод`, реюз скоупов витка for (без 2 dict-аллокаций
  и Slot на виток), кэш (kind, cap) на узлах типов, operator.* для
  бинопов без trap'а, пропуск тривиальных `requires/ensures true`.
  Интерп: Arith 365→640K, Struct 142→246K, Str 579K→1.08M оп/с.
- **F6 — закрыто с поправкой методики.** Реальный emit_ir на L —
  2.2 с, а не 5.5 с (старая цифра снята под cProfile). Все 33 416
  trap-строк уникальны (line:col в тексте) — дедупликация без смены
  канонического формата невозможна. Добавлен кэш `declare_intrinsic`
  (24 К вызовов). Остаток времени — сериализация/именование llvmlite,
  без смены формата не устраним.
- **F5 — частично; остальное требует решения уровня SPEC.** В Rt.eat
  горячие циклы append_str/append_u32 переписаны на идиому «маска —
  доказательство» (`let j = (base & 255) + i` + guard `j < 256`):
  верификатор доказывает границы, per-byte trap'ы ушли из циклов
  (append_str под O2: 105→54 строки IR, тело — чистый guarded copy).
  На StrBench это не отразилось: узкое место — статические 256 итераций
  и memcpy 260-байтного параметра by-value, а не trap'ы. Дальше два
  пути (оба инвазивные): `break` в `for` (SPEC §4, оба парсера, Rt.eat,
  фикспойнт) или memcpy-идиома в обоих IR-эмиттерах зеркально.
- **F2, F7 — не тронуты** (F2 — отдельное решение по SPEC §7:
  аксиома exit/write_err_byte; F7 — низкий приоритет).

## Статус исправлений (2026-07-13, третья итерация — изменение языка)

Решения приняты: `break` допустим в любом цикле, но **не более одного
на цикл** (стиль MISRA); аксиома `exit(code: u32)` — **только в main,
единственный вызов** на программу (по аналогии с loop); аксиома
`write_err_byte(b: u8)` — канал stderr. Итого аксиом ОС стало пять
(SPEC §7, runtime.c).

- **F5 — закрыто.** Горячие циклы Rt.eat (write, append_str,
  append_u32, eq, read_line) переписаны на ранний выход `break` —
  строковые операции стали O(длины), не O(ёмкости 256). Верификатор
  доработан: ветка if, кончающаяся break, исключается из join'а
  (негация условия guard'а доступна после него), пост-состояние цикла
  с break консервативно (killed). **StrBench: 10.6 → 94.6M оп/с (×9;
  ×43 к исходной базе 2.2M)**; self-hosted лексер — x3.9 от эталона
  (1.97M ток/с), парсер — x5.0 (1.42M ток/с).
- **F2 — закрыто наполовину.** Все *Main.eat вызывают `exit(1)` на
  fail-пути: отказ self-hosted компилятора теперь возвращает код 1
  (успех — 0), проверено вручную. `err:`-строка пока остаётся в
  stdout — перенос в stderr требует eprint-хелпера в Rt.eat (аксиома
  write_err_byte уже есть) и правки fail()-путей; отдельный шаг.
  Потребители (bench_selfhost, verify_selfhost) можно упростить до
  проверки кода выхода.
- Зеркальность: оба чекера (typechecker.py, Check.eat) enforce'ят
  ≤1 break/цикл, exit-только-в-main-и-один-раз, недостижимость после
  exit (правило 10); оба эмиттера (codegen.py, Ir.eat) дают
  байт-в-байт тот же IR (break в for — через общий стек выходов
  loop_exits/lpx). Фикспойнт сошёлся, 27/27 кейсов suite, полный
  bench зелёный.

## Статус исправлений (2026-07-13, четвёртая итерация — break-развёртка компилятора)

Идиома раннего выхода `break` (принята в третьей итерации для Rt.eat)
развёрнута на **весь self-hosted компилятор**: ~130 горячих циклов в
Tok/Lexer/Parser/Check/Ir переведены со «спина до статической ёмкости»
(`for i in 0..CAP { if i < n { ... } }` — всегда CAP итераций) на
`if i >= n { break }` в голове цикла. Затронуты: сравнение имён
`span_eq` (было всегда 256 итераций), все линейные таблицы Check
(`find_func` 1024, `intern` 512, `lookup`/`declare` 256, `add_edge`
4096), насосы машин (`parse` 262144, машины выражений/инструкций/
resolve/const_eval), обходы связных списков AST (131072), дедупы имён
и глобалов Ir (`g_commit` 65536, финальные проходы 131072/65536),
литеральные append'ы `lb_lit`/`g_lit`/`tp_lit`/`nb_lit`, фолды
шунтинг-ярда (через флаг `more` — ≤1 break на цикл). Попутно
`runtime.c`: `fflush(stdout)` в `eat_read_byte` теперь только в
интерактивном режиме (isatty, кэш) — батч-фильтр не платит вызов libc
за каждый входной байт.

Постоянная метрика — новая секция `compiler` в bench.py: все пять
фазовых бинарников компилируют конкатенацию собственных модулей
(вход verify_bootstrap, 376 КБ / ~91 К токенов), дампы диффятся
с эталоном; устаревшие бинарники пересобираются (быстрый режим —
пропускает), что закрывает ловушку «старый бинарник молча печатает
err: с кодом 0» в замерах.

Числа (тот же вход, до → после):

| фаза | до | после | ускорение | к Python |
|---|---|---|---|---|
| lex | 0.05 с | 0.04 с | ×1.2 | ×5.3 |
| parse | 0.06 с | 0.05 с | ×1.2 | ×6.8 |
| sig | 0.05 с | 0.02 с | ×2.5 | ×18.0 |
| typed | 0.54 с | 0.14 с | **×3.9** | ×2.6 (было ×0.7 — медленнее эталона!) |
| ir (полный конвейер) | 0.98 с | 0.68 с | ×1.44 | ×1.6 |

Семантика не менялась: все дампы и IR байт-в-байт, фикспойнт
bootstrap сошёлся, `make check verify verify_suite verify_selfhost
verify_bootstrap` зелёные, рантайм-бенчи без регрессий (Arith 823M,
Str 88.8M оп/с).

Следующий резерв (не сделано): статические open-addressing хеш-таблицы
для таблиц имён Check и корзина к уже существующему хешу `g_hash` в
`g_commit` Ir — линейный поиск остался, теперь по фактическому числу
записей; на IR-фазе также вес имеют побайтовая эмиссия и вывод 29 МБ
через write_byte.

## Статус исправлений (2026-07-13, пятая итерация — профиль нативного бинарника)

Впервые снят профиль самого self-hosted бинарника (`sample` на macOS,
символы `eat_Ir__*` читаемы). Три находки и три этапа-коммита:

- **Блокировки stdio — 70 % времени SelfIr.** `putchar` на каждый из
  ~29 М выводимых байт тратил большую часть цены в
  `flockfile`/`funlockfile`. Программы EATLang однопоточны по
  построению → `runtime.c` переведён на `putc_unlocked`/`getc_unlocked`.
  **ir 0.68 → 0.22 с (×3.1)**, lex 0.04 → 0.01 с.
- **Вывод через интерполяцию.** Все `o_*`-хелперы и дампы писали байт
  как `write("{char(b)}")` — временный `str<256>` на каждый байт.
  Заменено прямым builtin'ом `write_byte(b)` (вывод байт-в-байт тот
  же). Плюс `lb_tp` (копия строки типа — пишется при каждой
  инструкции) переведён на блочную копию с одной проверкой ёмкости
  и масками-доказательствами вместо ветвления на байт.
  ir 0.22 → 0.19 с.
- **Алгоритмы.** (а) Дедуп глобалов `g_commit`: хеш-корзина `ghx`
  (open addressing, 131072 слота, нагрузка ≤ 1/2) вместо линейного
  скана — хеш `g_hash` уже считался. (б) Сортировка рёбер typed-дампа:
  было ~1.6 М посимвольных `cmp_key` (выбором по 1789 рёбрам);
  теперь ранги ключей считаются один раз (`rank_build`, порядок
  рангов ≡ порядку ключей — строгая монотонность), сравнение ребра —
  два целых. **typed 0.10 → 0.07 с, ir 0.19 → 0.18 с.**

Итог пяти итераций на входе bootstrap (376 КБ, ~91 К токенов),
`make bench --only compiler`:

| фаза | база 13.07 | сейчас | ускорение | к Python |
|---|---|---|---|---|
| lex | 0.05 с | 0.01 с | ×5 | ×18.5 |
| parse | 0.06 с | 0.01 с | ×6 | ×22.2 |
| sig | 0.05 с | 0.01 с | ×5 | ×22.0 |
| typed | 0.54 с | 0.07 с | **×7.7** | ×5.1 |
| ir (весь конвейер) | 0.98 с | 0.18 с | **×5.4** | ×6.0 |

Остаток профиля SelfIr: `lb_tp` (~35 %, копия инлайн-строк типов —
банкованные адреса не векторизуются), `eat_write_byte` (~25 % —
пол: вызов аксиомы на каждый выводимый байт; ниже — только
буферизованная аксиома вывода, решение уровня SPEC), обход строк
в `p_blocks`, остатки `g_commit` (хеш уже стоит, остался сам
`g_hash` по содержимому). Все сверки зелёные, фикспойнт сошёлся
после каждого этапа.

## Статус (2026-07-13, шестая итерация — хеш-индекс таблицы функций)

`fhx` в Check: open addressing на 2048 слотов по ключу
(владелец, имя), вставка в `add_func`; `find_func` и оба скана
методов (`xs_method_obj` в Check, `xmeth_obj` в Ir) — один probe
вместо линейного прохода по 1024. Семантика первого совпадения
сохранена (вставка в порядке роста индексов ⇒ в цепочке
пробирования первый — младший индекс, как у скана); дампы
байт-в-байт, фикспойнт сошёлся.

**Замер честный: на текущем масштабе (524 функции) нейтрально** —
typed остался 0.07 с: после break-развёртки линейные сканы с ранним
выходом по длине имени уже были дёшевы. Ценность — асимптотика:
у предела языка (1024 функции) сканы дорожают вдвое, probe остаётся
O(1); плюс единый паттерн с `ghx`. Декомпозиция typed-фазы
(дифференциальными сборками): тела ~0.04 с, дамп ~0.02 с,
`rank_build` ~0.01 с — профиль тел плоский, одиночного горячего
скана в нём больше нет.

## Статус (2026-07-13, седьмая итерация — LLVM-атрибуты в обоих эмиттерах)

Гарантии языка зашиты в IR как атрибуты — синхронно в `codegen.py`
и `Ir.eat`, канон изменился, фикспойнт пересобран:

- `eat_trap`: `cold noreturn nounwind`; `eat_exit`: `noreturn
  nounwind`; остальные аксиомы: `nounwind` (шим не разворачивает
  стек, trap завершает процесс);
- каждый `define`: `norecurse nounwind` — прямое следствие правил
  1 и 9 (рекурсии нет по построению, исключений нет);
- `willreturn` НЕ ставится осознанно: по LangRef выход через
  exit/trap «не возврат», а trap-пути — штатное поведение;
  `mustprogress` llvmlite не знает (и для main с `loop` он неверен);
- **только build-путь** (без зеркала — в `eatc ir`/SelfIr фактов
  верификатора нет): доказанные `+ - *` эмитятся с `nsw`/`nuw`,
  доказанный `<<` — `shl nuw`. Доказательства верификатора теперь
  работают не только на удаление проверок, но и на оптимизатор.

**Замер честный: на текущих бенчах нейтрально** (Arith 822M,
Str 88.3M оп/с — в пределах шума; компиляция компилятора — те же
0.18/0.07 с). Причина: `trap_if` всегда эмитил `unreachable` после
вызова — CFG-часть `noreturn` LLVM уже знал; горячие циклы бенчей
уже без проверок («маска — доказательство»). Ценность — корректная
база: `cold` для раскладки секций, `nsw`/`nuw` для IV/GEP-анализа
на будущих доказательствах, `norecurse`/`nounwind` — задокументи-
рованные в IR гарантии языка (важно для embedded-таргетов и
будущих версий LLVM).

## Статус (2026-07-13, восьмая итерация — атрибуты указательных аргументов)

Завершение слоя атрибутов (оба эмиттера, канон, фикспойнт
пересобран): слот агрегатного возврата — `noalias nofree nonnull`
(свежая alloca на каждый вызов, ни с чем не алиасится); получатель
`var self` и агрегатные параметры — `nofree nonnull` (аргументы —
всегда alloca вызывающего, кучи нет). Зеркало — три строки в
`p_args`.

Осознанные «нет» этой итерации:
- `noalias` на параметры и self **нельзя**: вызывающий передаёт
  указатель на оригинал, `s.append_str(s)` даёт самоалиас;
- `readonly`/`nocapture` — llvmlite их не знает (список
  ArgumentAttributes LLVM 14);
- `dereferenceable(N)` потребовал бы layout-математику размеров в
  Ir.eat — проект сознательно избегает её (sizeof — `gep(null, 1)`);
- `align`/`sret` — риск смены ABI-поведения, не добавлены.

Замер: нейтрально на бенчах и компиляции компилятора (те же
0.18/0.07 с, Arith/Str в пределах шума). Слой атрибутов исчерпан:
выигрыши, которые они обычно приносят, проект уже собрал через
`unreachable` после trap'ов и снятие доказанных проверок
верификатором. Дальше скорость двигают только решения уровня SPEC:
буферизованная аксиома вывода (пол ~25 % SelfIr — вызов
`write_byte` на байт) и модель передачи `str<256>` (260-байтные
копии).

## Статус (2026-07-13, девятая итерация — аксиома батч-вывода write_span)

Решение пользователя: шестая аксиома ОС. Новый builtin
`write_span(a: [u8; N], off: u32, len: u32)` (SPEC §7) — байты
`a[off..off+len)` в stdout одним вызовом; `off + len > N` — trap
(проверка без u64: `off > N or len > N - off`; sub при `off > N`
заворачивается, но тогда первое условие уже истинно). Шим:
`eat_write_span` — гибрид: короткие спаны (< 32 байт) —
`putc_unlocked` (fwrite берёт блокировку потока, ~десятки нс на
вызов — дороже побайтной записи имён), длинные — `fwrite`.

Реализация во всех слоях: typechecker/interpreter/codegen + зеркало
Check.eat (машина, код 110; арм exit вынесен в хелпер — правило 4)
и Ir.eat (машина, код 8: аргументы через ag-стек, эмиссия
байт-в-байт с llvmlite включая trap-блок); trap-вид 8; кейс
`tests/verify/28_write_span.eat`. Потребители: `Rt.write` (str —
одним спаном; все print/write/интерполяции батчатся), `o_lb`/`o_tp`/
`o_nb`/`o_sb` в Ir.eat (банкованные пулы — спанами по банкам),
`w_span` в Lexer.eat (спаны имён из sbuf — переиспользуется
Check/Parser/Ir).

**Числа** (вход bootstrap ~91 К токенов): **ir 0.18 → 0.13 с
(×9.0 от Python; ×7.5 от утренней базы 0.98)**, lex/parse/sig/typed
~без изменений (их вывод — короткие экранированные спаны).
StrBench без изменений (горячий цикл — in-memory). Все сверки
зелёные, фикспойнт с новой аксиомой сошёлся.

Урок: `fwrite` на коротких спанах МЕДЛЕННЕЕ побайтного
`putc_unlocked` — первый замер без гибрида дал регресс lex/parse
×1.5–2; гибридный порог 32 байта снял его.

## Статус (2026-07-13, десятая итерация — передача агрегатов по ссылке)

Трек 1 [STR_MODEL_PLAN.md](../../docs/plans/STR_MODEL_PLAN.md), направление
A (решение пользователя, SPEC §5.4): defensive copy у вызываемого
снята — не-`var` `self` и агрегатные value-параметры связываются
прямо по входному указателю (они неизменяемы; единственный писатель
в чужую память — `var self`). Наблюдаемая by-value-семантика
сохранена точечной страховкой: на месте вызова метода с `var self`
аргумент с общим корнем lvalue-пути (`s.append_str(s)`) копирует
вызывающий (`emit_call`/`lvalue_root`; зеркало — `xmeth`/`lv_root`/
`lv_overlap` в Ir.eat). Дифференциальный пример — `examples/alias/`.

Это сняло оба «осознанных нет» восьмой итерации: `noalias` теперь
корректен на всех указательных аргументах (перекрытие закрыто
caller-копией), `readonly`/`nocapture` добавлены расширением
whitelist'а llvmlite (шим в codegen.py, печать сортированная).
Исчезли 260-байтные копии всех 7 не-`var` методов RtStr (`write`,
`eq`, `parse_*`…) и копии сегментов интерполяции; кадры функций в
отчёте §8 ужались (`RtStr.eq` 270 → 10 Б).

**Числа** (до = d843e51, после = fbe71fa; min из прогонов; вход

Находка про размер (вход трека 2): `__const` вырос вдвое, потому что
раньше LLVM разворачивал memcpy-из-глобала в непосредственные store'ы
и выкидывал сами глобалы литералов; теперь указатель на глобал уходит
в вызов по ссылке — глобалы живут в данных. Кандидат-лекарство: эмитить
литеральные глобалы по фактической длине, а не 260 Б. Там же: trap-строк
в каноне 1446 шт. на 106.6 КБ текста — 70 % всех строковых глобалов.

Два урока охоты за регрессом (первый замер дал StrBench ×2.6 хуже,
36M):
- **`nocapture` обязателен при передаче по ссылке**: без него LLVM
  считает адрес `s` утёкшим в вызываемого, не промоутит агрегаты
  вызывающего и оставляет memcpy присваивания вызовом libc (раньше
  указатель уходил только в intrinsic-memcpy, который nocapture по
  определению). Диагноз — дизассемблер: векторные q-блоки «до»
  против побайтового цикла + `blr memcpy` «после».
- **ветвление в теле цикла убивает векторизацию LLVM 22** (llvmlite,
  путь `eatc build`): guard `if j < 256 { store }` внутри цикла
  append_str не if-конвертируется (Apple clang 21 справлялся, LLVM 22
  llvmlite — нет). Лекарство на уровне Rt.eat: оба условия слиты в
  единственный `break` (`i >= ol or (base & 255) + i >= 256`) — тело
  стало чистым копирующим циклом, guard остался доказательством
  границ. Тот же приём — кандидат для остальных циклов RtStr (`eq`,
  `p_lo`/`p_hi`) при следующем профиле.

Направление B плана (memcpy по фактической длине) по замеру не
требуется: после A и фикса векторизации остаточные копии связываний
не видны в профиле StrBench; вернуться при появлении новых данных.

## Статус (2026-07-14, двадцать третья итерация — слой 1а: llvm.assume из доказанных requires; попытка векторизации Rt.eq)

**llvm.assume (принято, build-only)**: доказанный верификатором
requires раньше просто не эмитился — LLVM терял факт диапазона.
Теперь codegen эмитит `call @llvm.assume(<requires>)` в прологе,
только для безопасных форм (сравнения имён/полей/литералов и их
and-конъюнкции: арифметика внутри assume без trap-проверок дала бы
poison). Канон `eatc ir` не меняется (requires_proven ставит только
верификатор в build-пути), зеркало selfhost не требуется — прецедент
nsw/nuw седьмой итерации. **Сегодня нейтрально** (Machine 0.35 с,
Sort 0.04 с — в шуме), и понятно почему: requires_proven требует
доказанности на всех вызовах, а loop-carried аргументы
(`acc = work(acc) % 65536`) верификатор не ускоряет — на
work-функциях бенчей флаг не встаёт; где встаёт (маскированный
аргумент на месте вызова) — факт после инлайна виден LLVM и так.
Ценность — канал: трек 3 (интервалы пулов) начнёт ставить
requires_proven там, где факт LLVM недоступен.

**Rt.eq OR-свёрткой (отвергнуто по замеру)**: замена раннего выхода
безветвевой масочной редукцией `d |= buf[i]^o.buf[i]` при `i < ln` —
StrCmpBench 0.03 → 0.13 с (×4 хуже): llvmlite НЕ векторизует и
безветвевую масочную редукцию (расширение урока десятой итерации),
цикл остался скалярным и потерял ранний выход (полные 256 итераций
против средних ~45). Правило 2 запрещает `for i in 0..ln` — маска
неустранима, ранний выход строго выгоднее. Откат.

опроверг premise: все три обещанных выигрыша уже сняты, параметризация
даёт суб-килобайт кадра против самой инвазивной правки репо.

- **копии str** — уже по длине `4 + ln`, не 260 (codegen.py:344-351);
- **struct-поля** — ноль полей `str<N>` в компиляторе, строковые поля
  это `[u8; N]` (Lexer.val `[u8;256]`, Ir.tp `[u8;65536]`);
- **str-параметры** — биндятся по указателю без alloca/копии в callee
  (codegen.py:538-539); почти все `str<N>` в коде — параметры; в
  отчёте кадров §8 str-тяжёлые функции (span_is/lb_lit/g_lit) не
  появляются (кадры малы).

Остаток — 3 слот-локали (`io`/`pe` str<16>, `pre` str<8>), ~240 Б
каждая на трёх разных мелких кадрах при стеке 128 МБ. Против —
семантика ёмкости (сейчас `compatible()` молча разрешает
str<256>→str<8>, рантайм-проверки нет), ABI RtStr (хардкод 256),
зеркало Check.eat+Ir.eat, верификатор, интерпретатор, SPEC, фикспойнт.

Итог в [TRACKS.md](../docs/plans/TRACKS.md) трек 1 (закрыт целиком: by-reference
сделано, len-копия в коде, `str<N>` отвергнуто). Код не менялся.

## Статус (2026-07-15, тридцатая итерация — дешёвый блок A: порог write_span, O(n²) в верификаторе, профиль тайпчекера)

## Статус (2026-07-14, двадцать седьмая итерация — §4 МК-метрика: цена проверок во флеше)

Замер, ради которого затевался весь верификатор: сколько снятые
проверки стоят там, где это важно (не суперскаляр). Тезис из
двадцать четвёртой итерации — «на Apple Silicon предсказанные
trap-ветки почти бесплатны» — здесь дополняется противоположной
стороной.

Метод: один и тот же `.ll` эмитится дважды — с верификатором
(снятие доказанных) и без (`compile_binary` без вызова `verify` —
все проверки остаются), оба в режиме `--trap-codes`, компилируются
для Cortex-M3 (`--target=thumbv7m-none-eabi -mcpu=cortex-m3 -O2
-ffreestanding`), сверяется `.text` (`llvm-size`). Программа —
mos6502 (канонический МК-пример, читает ROM → LLVM не сворачивает;
самодостаточная константная программа не годится — `work(7)`
сворачивается целиком в константу на -O2).

**Числа (mos6502, Cortex-M3):** trap-сайтов 322 → **61** (снято
81 %), `.text` 6156 → **5624 Б** (−532 Б, **−8.6 %**). Порядок цены
сайта мал (~2 Б: `cmp; bcs`), потому что холодные trap-блоки LLVM
сливает в общий хвост `eat_trap_code` (атрибуты cold/noreturn +
unreachable — седьмая итерация). В режиме trap-СТРОК экономия много
больше (каждый снятый trap уносит и свою строку-сообщение из
`__const` — тринадцатая итерация: строки доминируют в данных МК), но
МК-сборка идёт в trap-кодах, поэтому релевантна цифра .text.

Итого тезис замкнут: проверки рантайм-нейтральны на суперскаляре
(двадцать четвёртая) и стоят −8.6 % флеша + 81 % trap-сайтов на
in-order МК (эта). Верификатор — прежде всего инструмент размера и
детерминизма МК, а не хостовой скорости. Воспроизведение — таблица
в [mcu/README.md](../../mcu/README.md), раздел «Цена проверок».
Цикловый замер (QEMU `-icount`) требует TCG-плагина счётчика
инструкций — отдельный шаг, .text-метрика детерминирована и без него.

## Статус (2026-07-14, двадцать шестая итерация — §2.1 `!range` из инвариантов: ОТВЕРГНУТО замером)

Попытка передать инварианты пулов в LLVM метаданными `!range` на
чтениях (`load ..., !range !N`). Реализация рабочая и корректная:
верификатор аннотирует Index-узлы (`val_range = (lo, hi, sent)`,
build-only, проба цикла молчит), кодоген эмитит `!range`; дырка
`[0,hi] ∪ {typemax}` ложится на единый оборачивающийся интервал
`[typemax, hi+1)` (LLVM-каноника, проверено `ref.verify()`; пустой
случай `hi+1==typemax` отсеян). 562 сайта в SelfIr, канон `eatc ir`
чист (0 range), все 7 ворот зелёные, дифф-сверки interp==binary
зелёные — семантически безупречно.

**Но замер против политики §2.1 «оставить, только если нейтрально»:
`!range` — воспроизводимый регресс, не нейтраль.** TrapBench x512:
0.77 с без range → **0.83 с с range** (три раунда по best-of-6,
0.77/0.77/0.77 против 0.83/0.83/0.83 — не шум); BankBench без
изменений (0.39 с). Метаданные не живут в бинарнике, значит `!range`
меняет решения оптимизатора LLVM в горячем цикле пула к худшему
(вероятно, гасит какое-то преобразование или ведёт к иной выборке
инструкций). Выигрыша нигде не измерено. Решение — **откат** (как
Rt.eq OR-свёрткой в двадцать третьей итерации): при регрессе без
компенсации не коммитим. Урок: доказанный факт в IR ≠ ускорение —
на этом стеке (llvmlite=LLVM 22, Apple Silicon) `!range` из
пуловых инвариантов вредит; канал «инварианты → LLVM» уже закрыт
`nsw`/`nuw` и `llvm.assume`, `!range` не добавляет. Направление
закрыто, в OPTIMIZATIONS_PLAN §2.1 помечено «отвергнуто замером».

## Статус (2026-07-14, двадцать пятая итерация — трек 3: дизъюнктивный домен через циклы)

Продолжение §1.1 [OPTIMIZATIONS_PLAN](../../docs/plans/OPTIMIZATIONS_PLAN.md):
дырка-сентинел («плотное ∪ {NONE}») теперь проходит через циклы, а не
теряется на loop-join. Правка только verifier.py (canon/selfhost не
тронуты). Три механизма:

1. **Дырочное объединение выхода цикла** (`_loop_exit`): значение
   loop-carried пути на выходе = объединение входного значения и всех
   значений, присвоенных телу (сборщик `_exit_rec` пишет `(iv, hole)`
   на каждом присваивании). Сентинел даёт явная дырка записи либо
   точечный максимум типа — так `var f = NONE; в цикле f = fi` даёт
   `[0,1023] ∪ {NONE}`.
2. **hive-summary** (`_hive_summary` + `ret_holes`): функция, чьи
   return несут дырку (аккумулятор поиска `return f`), получает
   summary `("hive", dense, sent)`; на месте вызова
   `let x = find(...)` дырка садится на x, guard `x != NONE`
   схлопывает к плотной части — границы `pool[x]` доказываются.
3. **2-проходный инвариант тела** (probe без следов + real): проба
   прогоняет тело с подавлением отметок/учёта requires/return, собирая
   инварианты loop-carried путей (`slot = (slot+1) & M` → `[0,M]`,
   `c = pool[...]` → дырка) — они независимы от старого значения пути.
   Реальный проход сеет их на входе тела: чтения `hx[slot]`,
   `nk[c/65536][c%65536]` после guard'а получают точные границы.
   Отметки AND-сливаются, поэтому проба обязана молчать.

**Числа** (bootstrap-вход, 13 модулей): доказано 5356 → **5456 из
6945** за две итерации инвариантов (эта: +100 суммарно; чистый вклад
дизъюнктивного домена +14 к прошлому 5442). bounds 994 → 921,
overflow 294 → 287, requires 167 → 162, ensures 27 → 12. TrapBench:
133 → 138 из 217. Анализ верификатора — те же 0.35 с (2-й проход
тел не взорвал время: пробы дёшевы, глубина мала). Кейсы 42
(аккумулятор: bounds 3/3 — slot-инвариант + hive), 43 (негатив:
запись произвольного значения ломает сентинел, `dat[f]` в рантайме).
Ворота: suite 43/43, check/verify/selfhost/bootstrap/trapcodes —
зелёные, дифф-сверки бенчей зелёные.

**Остаток и следующий рычаг**: bounds 921 (Check.eat 471). Основное —
глубоко-вложенные чтения `nns[nfc[c]...]` без NONE-guard'а на
промежуточном узле (семантический инвариант «k==16 ⇒ ребёнок есть»,
интервалам недоступен) и assert-суммы полей (`self.ln + o.ln <= 256`,
103 шт — §1.2 плана, ждёт разностных инвариантов полей). Кодогенный
рычаг §2.1 (`!range` из инвариантов пулов) — независимый, дешёвый.

| `builtins.isinstance` | 41 007 119 | 1.5 с только на диспетчеризацию |
| `copy.deepcopy` | 2 001 006 | **2 deepcopy на итерацию цикла** — копируются в т.ч. скаляры |
| `push_scope`/`pop_scope` | по 2 002 004 | новый скоуп на каждый виток for |

Скорости сейчас: 130–580 К оп/с (StructBench медленнее всех — 132 К,
там deepcopy структуры на присваивание поля).

Что делать (по убыванию выгоды):
1. deepcopy только для составных значений (list/struct); int/bool/str
   неизменяемы в Python — копировать не нужно;
2. диспетчер `dict: type(node) → метод` вместо цепочек isinstance в
   `eval`/`exec_stmt`;
3. не пересоздавать скоуп на виток цикла (реюз кадра с очисткой).

Ограничение: интерпретатор — референс-семантика языка. После каждой
правки: `make check && make verify_suite && make bench` (в bench —
сверка вывода интерпретатора и бинарника на пяти программах) и
`make verify_selfhost` (interp == native на self-hosted исходниках).

## F5. Строковая машинерия: O(ёмкости) на каждую операцию

Доказательство: нативный StrBench — 2.2 М оп/с против 480 М оп/с у
арифметики; ускорение бинарника над интерпретатором ×4 при ×1300–2400 у
остальных программ. (Сравнение к тому же нечестное: интерпретатор
исполняет интерполяцию питоновскими строками — `interpreter.py:521`, а
бинарник — циклами Rt.eat.)

Корневая причина: `selfhost/Rt.eat:49` `append_str` — цикл
`for i in 0..256 { if i < o.ln { copy } }`: всегда 256 итераций с
guard'ом, потому что границы циклов статические, а `break` разрешён
только в `loop` внутри main (SPEC §4, правило 2). Каждая интерполяция
`"{s}ab"` копирует до 256 байт независимо от фактической длины. Тот же
паттерн — во всех append_* и `eq`.

Что делать (варианты, от дешёвого к дорогому):
- F1 уже даёт 2.6× на этом коде (LLVM векторизует guard-цикл);
- codegen: распознавать идиому «for 0..CAP + guard i<len» и опускать в
  `llvm.memcpy` по длине — локальная оптимизация без правки языка;
- язык: разрешить `break` в `for` (граница остаётся статической —
  правило NASA не нарушается, это ранний выход, а не бесконечность);
  потребует правки SPEC §4, обоих парсеров, Rt.eat и перегенерации
  фикспойнта — самое инвазивное.

## F6. Профиль emit_ir: trap'ы и сериализация

Доказательство (вход pipe_L_rt, 176 функций): emit_ir 5.54 с =
`generate()` 3.32 с + `str(module)` 2.21 с. Внутри generate:
`trap_if` — 33 416 вызовов, 2.05 с (62%); llvmlite `__str__` — 942 330

`llvm.*.with.overflow` + ветка + глобальная строка сообщения.

Замечания:
- в build-пути verifier уже устраняет доказанные проверки
  (`codegen.py:751 no_overflow`, `:448 proven`, `:775 div_safe`) — но
  `cmd_ir` (`__main__.py:150`) verify не зовёт, что консистентно с
  SelfIr (фикспойнт), менять осторожно;
- быстрые выигрыши без смены формата: кэшировать/дедуплицировать
  глобальные строки сообщений trap (сейчас их тысячи однотипных),

## Ось -O: элизия проверок верификатора (SELFHOST_VERIFIER_PLAN этап 5, 2026-07-17)

Проход №2 оси: `eatc ir -O` = fold → verify → эмиссия по аннотациям
(элизия доказанных trap-блоков, `nsw`/`nuw` на доказанных `+ - * <<`,
`llvm.assume` на доказанном requires); `SelfIrOpt` — байт-в-байт.
До этого элизию давал только Python-путь `eatc build`; теперь тот же
оптимизированный IR эмитит self-hosted компилятор.

Числа по каждому проекту (канон `eatc ir` → `eatc ir -O`; вход —
Rt + модули программы; trap-блоки — вызовы `@eat_trap`):

| Программа | строк IR | trap-блоков | plain `nsw`/`nuw` | `llvm.assume` |
| --- | --- | --- | --- | --- |
| hello_world | 2 396 → 1 580 (−34 %) | 126 → 34 (−73 %) | 31 | — |
| all (витрина) | 2 746 → 1 813 (−34 %) | 142 → 36 (−75 %) | 32 | 1 |
| async | 2 672 → 1 810 (−32 %) | 135 → 37 (−73 %) | 31 | — |
| lexer (проба) | 3 009 → 2 007 (−33 %) | 147 → 35 (−76 %) | 33 | 2 |
| async/Pipe | 3 569 → 2 473 (−31 %) | 166 → 41 (−75 %) | 37 | — |
| mos6502 | 6 585 → 4 350 (−34 %) | 321 → 60 (−81 %) | 89 | 16 |
| фронтенд (самоприменение) | 104 412 → 63 107 (−40 %) | 5 810 → 1 180 (−80 %) | 245 | 119 |

Картина ровная по всем проектам: **IR −31…−40 %, trap-блоков
−73…−81 %** — три четверти проверок каждой программы доказаны и
исчезают из эмиссии; остаток — принципиально недоказуемое
(loop-carried аккумуляторы, входные данные). Скорость эмиссии:
SelfIrOpt на самоприменении 0.26 с против эталона `eatc ir -O`
1.17 с (×4.5); кадр `main` SelfIrOpt 179,5 МБ, худшая цепочка
217,2 МБ — в `STACK_FLAGS` 256 МБ с запасом ~15 %.

Интерпретация «насколько быстрее работает программа»: на Apple
Silicon предсказанные trap-ветки почти бесплатны (двадцать четвёртая
итерация), поэтому хостовые рантайм-бенчи от элизии не двигаются —
выигрыш в размере кода и на in-order МК (двадцать седьмая итерация:
mos6502 на Cortex-M3 — `.text` −8.6 %, trap-сайтов −81 %). Новое в
этом этапе не сами проценты (у `eatc build` они были), а то, что
их теперь даёт self-hosted тулчейн — байт-в-байт с эталоном, под
гейтом `verify_selfhost_opt`.

Как повторить: `PYTHONPATH=src uv run python -m eatc ir [-O] <вход>`
на конкатенации Rt + модули программы; счётчики — `grep -c
'call void @"eat_trap"'`, `grep -cE ' (add|sub|mul|shl) (nsw|nuw)'`,
`grep -c llvm.assume` (минус declare).

## HTTP-парсер в crosslang (HTTP_PLAN этапы 0–3, 2026-07-17)

Новый бенч `http`: 2 000 запросов на REPEAT=1 через парсер
`lib/Http.eat` (4 профиля: GET с параметром пути и keep-alive, POST с
close, кривая request-line → 400, HTTP/1.0-мимо-маршрутов → 404) +
мини-роутер и `wants_close`. Порты C/Rust/Go/Python 1:1, сверка
9 вариантов байт-в-байт (`checksum`), сборка драйвером
(режим `"driver"` в BENCHES — общий с U128).

Полный прогон (медиана 5, native ×256): EATLang build 305 нс/запрос
(IPC 4,77), selfhost -O 304 (паритет), все-trap 376 (налог 1,23);
C -O2 226, UBSan 336 (1,48); Rust safe 232; Go 417; Python 7 834.

Декомпозиция EAT/C = 1,35 (замер по частям): первоначальная гипотеза
«интерполяция строк» опровергнута — сборка строк у EATLang в 5,8 раза
**быстрее** `snprintf` (8,7 против 50,2 нс/строка), `req_new` —
паритет с `memset` (99/95 нс, копии возврата элидированы). Разрыв
целиком в побайтовом цикле парсера: ×3,7 за вычетом общего memset
(8 291 против 3 074 инстр/запрос, вызовов в горячем пути нуль,
рантайм-проверки дают лишь +23 %) — llvmlite-конвейер не промотирует
поля 8,5-КБ структуры в регистры как clang (нет TBAA). Рычаги:
span-API парсера (состояние в локалах на диапазон, не на байт),
ленивое обнуление `raw[8192]` (~95 нс/запрос у обоих языков), TBAA
в эмиссии. Подробности — [CROSSLANG.md](CROSSLANG.md), «HTTP-парсер
(третий прогон)».

## U128: цена программного 128-бит (U128_PLAN этап 4, 2026-07-17)

Два бенча под замер-гейт этапа 4 (intrinsics — «только по замеру»):
`U128Bench` — микс без деления (LCG128 на константах PCG64 +
`mul_64`), `U128DivBench` — 400 полных `divrem` + 4000 `divrem_32` на
64-битные делители. В `bench.py` и crosslang — режим `"driver"`
(модуль с import-шапкой в cat-поток не встаёт, сборку ведёт драйвер
`--lib` / `stream`; в crosslang механизм переиспользован http).

Внутренний контур: микс — интерпретатор 4K оп/с, бинарник 74M оп/с
(×19 926); деление — 1K против 20M оп/с (×16 272). Crosslang (порты
C `__int128` / Rust `u128` / Go `math/bits` / Python `int`,
9 вариантов байт-в-байт): микс EAT/C = ×6,1 (×4,9 через
clang-конвейер — разрыв build/все-trap тот же llvmlite-против-clang,
что в 7.2), внутри вилки плана «5–15×»; деление blend ×8,4.
Раскладка деления отдельным замером: `divrem_32` ≈ 3,9 нс —
**паритет с C** (там 128/64 идёт тем же `__udivti3`), полный
сдвиговый `divrem` ≈ 490 нс (3,8 нс × 128 итераций) — ×30.

Вердикт этапа 4: **intrinsics не оправданы** — микс в вилке, горячих
клиентов нет; единственное узкое место (полное деление 128/128)
лечится алгоритмом — нормализация + по-лимбовое деление в духе
`divrem_32`, потенциал ~×10 **без правок языка** — и только по
появлению клиента с горячей точкой. Подробности —
[CROSSLANG.md](CROSSLANG.md), «128-битная арифметика U128».
