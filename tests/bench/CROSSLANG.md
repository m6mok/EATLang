# Кросс-языковое сравнение: EATLang против C, Rust, Go, Python

Дата: 2026-07-17. Машина: Apple M2, 16 ГБ, macOS 26.3.1.
Тулчейны: eatc (Python 3.11.14 + llvmlite, LLVM O2 + clang -O2 линк),
Apple clang 21.0.0, rustc 1.97.1 (`-C opt-level=2`), go 1.24.3,
CPython 3.11.14. Методика и оси — план
[CROSSLANG_BENCH_PLAN](../../docs/plans/CROSSLANG_BENCH_PLAN.md),
раннер — [crosslang/](crosslang/README.md). Прогон: `make
bench_crosslang`, 12 мин; полные таблицы всех вариантов — в
`build/bench/crosslang/results.json`.

Все порты сверены **байт-в-байт** с EATLang-бинарником (микро — единый
REPEAT, макро — два лёгких ROM); стена — медиана ≥ 5 прогонов;
инструкции/циклы — `/usr/bin/time -l` (instructions retired / cycles
elapsed).

## Скорость исполнения (нс/оп; меньше — лучше)

| Нагрузка | EATLang | C -O2 | Rust | Go | Python | EAT/C |
| --- | --- | --- | --- | --- | --- | --- |
| arith — u32-арифметика | 1,23 | 1,17 | 1,16 | 1,45 | 32,7 | 1,05 |
| sort — вставки, data-dependent | 0,39 | 0,41 | 0,41 | 0,61 | 33,6 | 0,95 |
| branch — диспетчер по битам | **1,10** | 2,47 | 2,43 | 1,21 | 84,7 | **0,44** |
| bank — банкованная память | 4,06 | 4,16 | 4,40 | 5,27 | 154,5 | 0,97 |
| strcmp — сборка+сравнение строк | **25,8** | 67,8 | 46,2 | 43,1 | 129,5 | **0,38** |
| u64 — 64-битная арифметика | 4,45 | 4,42 | 4,65 | 4,93 | 168,5 | 1,01 |
| mos6502 — эмулятор, нс/шаг | 3,63 | 3,25 | 4,35 | 5,58 | 514,8 | 1,12 |

Геометрическое среднее EAT/C по семи нагрузкам — **0,79** (EATLang на
21 % быстрее); без двух выбросов (branch, strcmp) — **1,02**: паритет
с C в пределах шума. Rust — та же LLVM-лига (±5 %), Go стабильно
+15…70 %, Python — в 26–40 раз медленнее на микро и в 142 раза на
эмуляторе.

Оба выброса — не артефакт замера, а различие кодогенов, проверено
дизассемблированием:

- **branch ×2,2**: clang и rustc сворачивают if/elif-диспетчер в
  безветвенную цепочку `csel` — считают все четыре плеча каждую
  итерацию и завязывают их в длинную цепь зависимостей; EATLang (и Go)
  оставляют живые ветвления, которые предсказатель M2 обрабатывает
  быстрее (14,5 инстр/оп против 23; IPC 3,8 против 2,8).
- **strcmp ×2,6 к C**: интерполяция `"{x}"` компилируется в
  специализированный `Rt.append_u32` (без разбора format-строки и без
  аллокаций); у соперников — их естественные идиомы: C `snprintf`
  (парсинг формата каждый вызов), Rust `format!` (аллокация String),
  Go `strconv` + конкатенация. Меряется цена идиомы языка, не
  качество бэкенда — это осознанный выбор методики.

## Эффективность кода для процессора

Инструкций на операцию (плотность кода) и IPC — по базовым вариантам:

| Нагрузка | EAT инстр/оп | C | Rust | Go | EAT IPC | C IPC |
| --- | --- | --- | --- | --- | --- | --- |
| arith | 6,0 | 6,0 | 6,0 | 7,0 | 1,50 | 1,50 |
| sort | **4,8** | 6,3 | 6,3 | 10,9 | 3,82 | 4,60 |
| branch | **14,5** | 23,0 | 23,0 | 14,5 | 3,82 | 2,84 |
| bank | 21,1 | 19,1 | 19,1 | 26,2 | 1,56 | 1,36 |
| strcmp | **492** | 1208 | 932 | 771 | 5,94 | 5,50 |
| u64 | 12,0 | 11,1 | 11,1 | 12,1 | 0,78 | 0,70 |
| mos6502 | **54,8** | 58,3 | 56,2 | 94,4 | 4,44 | 5,31 |

EATLang-код по плотности не уступает C/Rust (на sort — заметно
плотнее: банкованные границы уже доказаны и LLVM получает чистые
guarded-циклы), Go тратит на 20–70 % больше инструкций на ту же
работу. Низкий IPC у arith/bank/u64 — у всех языков сразу:
loop-carried зависимость (следующая операция ждёт предыдущую),
процессору нечем заполнить такты; высокие IPC (4–6) — там, где есть
независимая работа. Python исполняет 726–7969 инстр/оп при IPC 5–7:
интерпретатор занят собой, а не задачей.

## Налог на безопасность (внутри каждого языка)

Дельта времени «вариант с проверками ↔ без» на одной и той же
нагрузке:

| Нагрузка | EAT: все-trap ↔ build | C: UBSan ↔ -O2 | Rust: safe ↔ unchecked | Go: bounds ↔ -B |
| --- | --- | --- | --- | --- |
| arith | −3 % (шум) | 0 % | — (нет массивов) | 0 % |
| sort | −3 % (шум) | **+70 %** | +0 % | **+34 %** |
| branch | **+41 %** | −5 % (шум) | — | −4 % |
| bank | +1 % | +7 % | +6 % | −4 % |
| strcmp | −4 % (шум) | +2 % | — | +1 % |
| u64 | +5 % | +15 % | — | −3 % |
| mos6502 | −3 % (шум) | +14 % | — | — |

Главный вывод оси: **у EATLang налог на безопасность почти нулевой** —
верификатор доказал и снял проверки там, где они стоили бы дорого
(sort/bank: маска и границы доказаны статически), а оставшиеся trap'ы
почти бесплатны (предсказуемые никогда-не-взятые ветки; на u64 +5 % —
цена недоказанных переполнений). Единственный крупный выигрыш элизии —
branch (+41 %: доказанные сдвиги и переполнения убирают инструкции из
цепи зависимостей). Для сравнения: Rust платит ~0–6 % (LLVM сам снимает
bounds-check по маске), Go — до +34 % на сортировке, UBSan для C —
до +70 %. Обратная сторона: у C/Rust-вариантов «без проверок» при
ошибке — UB/тишина, у EATLang все недоказанные проверки остаются
trap'ами всегда.

Оговорка: у EATLang «все-trap ↔ build» проходит через разные конвейеры
(clang -O2 по .ll против llvmlite O2 + линк объектника), поэтому
дельты ±3–5 % — шум конвейера, а не эффект проверок; сигнал только
branch и u64.

### Сборка селфхост-компилятором (второй прогон, 2026-07-17)

Вариант «EATLang selfhost -O» — цепочка `SelfIrOpt` (компилятор,
написанный на EATLang; ось `-O`: свёртка + элизия) + `clang -O2` +
runtime.c; IR сверен с `eatc ir -O` байт-в-байт перед замером.

| Сборка | микробенч | эмулятор mos6502 |
| --- | --- | --- |
| eatc build (Python + llvmlite) | 0,18–0,19 с | 0,70 с |
| **SelfIrOpt + clang** | **0,11–0,13 с** | **0,23 с** |
| — из них чистая эмиссия SelfIrOpt | 0,01 с | 0,02–0,03 с |
| clang -O2 (порт C) | 0,05 с | 0,10 с |
| rustc -C2 | 0,09–0,69 с | 0,65 с |
| go build | 0,05–0,22 с | 0,05 с |

Сам компилятор на EATLang эмитит IR эмулятора за **20–30 мс** — в
~25 раз быстрее Python-эталона (который, справедливости ради, ещё
гоняет verify и test-блоки); почти всё время цепочки — это clang.

По скорости исполнения строка «selfhost -O» легла в тот же коридор
±5 %, но вскрыла нюанс: на branch (1,52 против 1,09 нс/оп у build) и
эмуляторе (3,96 против 3,79/3,58) она медленнее обоих других
EATLang-вариантов при меньшем числе trap'ов. Первичная гипотеза —
«`llvm.assume`-факты оси `-O` съедают выигрыш элизии» — замером
этапа 7.1 **опровергнута** (2026-07-17): в branch-варианте оси `-O`
assume нет вообще (0 вызовов — IR совпадает с build с точностью до
trap-строк), а снятие всех 15 assume эмулятора не сдвинуло
инстр/шаг (56,32 → 56,29; стена в коридоре шума ±2,5 % канона).
Реальные причины разрыва: на branch — конвейер сборки (llvmlite у
build против clang по .ll, OPTIMIZATIONS_PLAN §7.2), на эмуляторе —
сама элизия оси `-O` против канона при том же clang (инстр/шаг
56,3 против 54,6: снятый trap-блок уносит и range-факт ветки,
профиль — §7.3). `llvm.assume` как инертный снят из эмиссии
(OPTIMIZATIONS_PLAN §7.1).

Замер §7.2 (2026-07-17, `backends.py`: оба бэкенда на
байт-идентичном .ll build-пути) закрыл вопрос конвейера: clang-путь
нигде не быстрее штатного — branch +28…+31 % к build (кодоген),
mos6502 +4,9…+6,8 % (мид-энд), остальное паритет; llvmlite 0.48
несёт LLVM 22.1.0 против Apple clang 21.0.0. Маршрут `eatc build`
остаётся llvmlite; `-flto` — ось размера (33К против 50К), не
скорости. Числа — FINDINGS, 39-я итерация.

Между полными прогонами дрейф ±5 % по стене — нормален; таблицы выше
из первого прогона, селфхост-числа — из второго
(`results.json` — всегда последний прогон).

### HTTP-парсер (третий прогон, 2026-07-17)

Бенч `http` (HTTP_PLAN этапы 0–3): 2 000 запросов на REPEAT=1 — тот же
парсер, что `lib/Http.eat` (request-line + заголовки в фиксированные
буферы, срезы-Header, OWS-трим, регистронезависимый поиск, мини-роутер
c параметром пути, решение keep-alive); порты C/Rust/Go/Python 1:1,
9 вариантов сходятся байт-в-байт. Сборка — драйвером (`--lib`):
HttpBench → `lib/Http.eat` → `lib/Fmt.eat`.

| EATLang build | 305 | 4 775 | 4,77 |

## Честные оговорки

1. **EATLang, C и Rust делят бэкенд LLVM** — сравнение микробенчей
   во многом меряет фронтенды/семантику, а не три независимых
   компилятора. Go (свой бэкенд) и Python (интерпретатор) — настоящие
   внешние точки.
2. **Порты 1:1, а не идиоматичные программы**: C-программист написал
   бы strcmp через ручной itoa и обогнал бы `snprintf`; сравнивается
   «та же программа», а не «лучшая программа на языке».
3. **Сравнивается только выразимое на EATLang** подмножество задач:
   без кучи, рекурсии, неограниченных циклов. Задачи, где C/Rust/Go
   используют динамические структуры, в сравнение не попадают
   по построению.
4. Микробенчи чувствительны к капризам планировщика LLVM (см. branch);
   макробенч-эмулятор — самая представительная строка таблицы:
   паритет EATLang с C в пределах 12 % при полностью доказанной
   памяти (все обращения к mem[65536] — без проверок в рантайме,
   маска доказана верификатором).
5. Python-порту вычтен старт интерпретатора (~18 мс); REPEAT у него
   меньше (нормировка на операцию).
