## 0. Идея: тотальность как оптимизация

Центральное следствие Power of 10 ([POWER_OF_10.md](POWER_OF_10.md)):
язык не Turing-полный, каждая функция доказуемо завершается
(правила 1+2: граф вызовов — DAG, границы циклов статичны).
Значит, **вычисление любого вызова с константными аргументами
разрешимо** — не эвристика с произвольным лимитом, как `constexpr`
в C++ (где язык Turing-полный и предел — костыль), а прямое
следствие дизайна языка. EATLang — редкий язык, где partial
evaluation корректен по построению.

Что это даёт на практике:

- **таблицы, посчитанные компилятором**: CRC32/SHA-константы
  ([CRYPTO_SUITE_PLAN.md](CRYPTO_SUITE_PLAN.md)), таблицы опкодов
  6502, синусы/гаммы для МК — в бинарнике готовый `__const`-глобал
  вместо кода инициализации на старте (для МК — данные во флеше,
  минус код, минус время старта);
- **константы из выражений предметной области**: сегодня
  `const_eval` (`src/eatc/typechecker.py:383`, зеркало —
  `selfhost/Check.eat:573`) умеет только `+ - * / %` над литералами
  и `const`-именами; любую формулу сложнее приходится считать руками
  и вписывать литералом;
- **свёртка веток**: вызов с константными аргументами внутри
  функции складывается в литерал → верификатор получает точный
  интервал → снимаются trap-проверки ниже по потоку (синергия с
  треком 3 [TRACKS.md](TRACKS.md)).

Честная оговорка по профилю: selfhost-компилятору comptime даёт
немного (его таблицы кодировок уже литералы) — главные выгодники
crypto, эмуляторы и МК. Для selfhost-профиля основной выигрыш —
косвенный, через верификатор.

## 1. Семантика: два яруса

### Ярус A — язык: вызовы в const-выражениях (решение по SPEC)

Разрешить в контекстах, где сегодня требуется константа
(верхнеуровневый `const`, границы `for`, размеры `[T; N]` /
`str<N>`), вызов **comptime-годной** функции (§2) с константными
аргументами:

```eat
const CRC_POLY: u32 = 0xEDB88320
const CRC_STEP7: u32 = crc_entry(7)      # вычисляется компилятором

func crc_entry(i: u32) -> u32
    requires i < 256
    ensures true
{
    var c: u32 = i
    for _ in 0..8 {
        ...
    }
    return c
}
```

Семантика ошибок — строже рантайма, в духе «всё, что не доказано, —
trap», только здесь trap случается на компиляции и потому
становится **ошибкой компиляции**:

- вычисление trap'ит (переполнение, деление на ноль, нарушенный
  `requires`/`ensures`/`assert`) → ошибка компиляции с координатами
  вызова и текстом trap'а;
- превышение бюджета шагов (§3) → ошибка
  `comptime: превышен предел шагов (предел N)` — штатная ошибка
  класса «capacity exceeded», как пределы SPEC §6.

### Ярус B — оптимизация: свёртка в теле функций (семантика не меняется)

В build-пути: вызов comptime-годной функции, все аргументы которого —
известные константы, вычисляется компилятором и заменяется
литералом. Три исхода:

- вычислилось → подставить результат (наблюдаемое поведение
  идентично: функция чистая, результат тот же);
- trap → **не сворачивать**, оставить рантайм-вызов — программа
  trap'нет там же и с тем же текстом, что и раньше;
- бюджет исчерпан → не сворачивать.

Ярус B не меняет ни одну наблюдаемую программу — это чистая
оптимизация, но требует того же вычислителя, что и ярус A.
Порядок внедрения: сначала A (меньше поверхность, явная семантика),
B — вторым шагом по замерам.

## 2. Comptime-годность: чистота без аннотаций

Функция comptime-годна ⇔ транзитивно не вызывает аксиом ОС
(`read_byte`, `write_byte`, `write_span`, `write_err_byte`, `exit`,
`trap` — сам trap годен: это ошибка компиляции, §1) и `extern func`.
Граф вызовов статичен (правило 9: указателей на функции нет) и уже
строится для DAG-проверки правила 1 — годность вычисляется одним
обходом без какой-либо разметки в языке. Никаких `comptime`-ключевых
слов: годность — свойство, а не декларация.

`loop` в `main` не мешает: `main` не вызывается из выражений.

## 3. Бюджет шагов

Правило 2 даёт статическую верхнюю границу числа шагов любого
вызова, но она произведение границ вложенных циклов по глубине DAG —
может быть астрономической. Поэтому практический предел в духе §6:

- **шаг** — одно вычисленное выражение/statement вычислителя
  (определить точно и одинаково в обоих компиляторах — входит
  в паритет!);
- предел — константа `MAX_COMPTIME_STEPS` в `limits.py` и зеркало
  в selfhost (кандидат: 1 000 000 — таблица CRC 256×8 ≈ 20 К шагов,
  запас ×50);
- превышение: ярус A — ошибка компиляции, ярус B — отказ от свёртки.

Бюджет — на **один** const/вызов, не суммарный: иначе порядок
объявлений влиял бы на успех компиляции.

## 4. Паритет — главный ценник этапа

Требование прежнее ([MODIFYING.md](MODIFYING.md)): оба компилятора
байт-в-байт. Для comptime это означает **два одинаковых вычислителя**:

- **Python-эталон**: переиспользовать интерпретатор
  (`src/eatc/interpreter.py`) — он уже реализует точную семантику
  (trap-тексты, переполнения, касты). Обернуть в comptime-вход:
  счётчик шагов + запрет аксиом. Дёшево;
- **selfhost**: интерпретатора нет — нужен вычислитель AST на
  EATLang. Идиомы проекта готовы (парсер — автомат с явным стеком,
  обходы — итеративные машины, память — пулы): вычислитель — та же
  машина по типизированному AST, стек кадров — пул с курсором,
  глубина ограничена DAG (правило 1 гарантирует конечность).
  Это крупнейший модуль этапа — фактически мини-интерпретатор
  в selfhost (новый файл, кандидат `selfhost/Eval.eat`).

Паритет вычислителей проверяется жёстко: одинаковые значения —
через фикспойнт (`verify_bootstrap`) и дампы, одинаковые тексты
ошибок — байт-в-байт, включая координаты. Малейшее расхождение
семантики (например, порядок проверки переполнения и деления)
ломает бутстрап — поэтому определение «шага» и порядок trap-проверок
фиксируются в SPEC, а не в коде.

Опасный узел бутстрапа: как только selfhost-код **сам** начнёт
использовать const-из-вызовов, его обязан уметь собирать
Python-эталон → обычный порядок MODIFYING (сначала эталон, затем
selfhost, затем использование).

### 9.1 Семантика вычисления — семантика бинарника (строгая по-операционная)

Интерпретатор Python считает промежуточные значения в произвольной
точности (fit только в точках связывания) — это НЕзеркалируемо:
u64×u64 = 2¹²⁸ живёт в Python и невыразимо в EATLang-вычислителе.
Поэтому comptime-режим — **строгий по-операционный** (= семантика
скомпилированного кода, что обязательно и для яруса B):

- `+ - *` и унарный `-`: результат вне диапазона типа узла →
  trap `переполнение {kind}` (текст кодогена, без значения);
- `/ %`: делитель 0 → `деление на ноль`; знаковые INT_MIN/(-1) →
  `переполнение {kind}` (и для `/`, и для `%` — как в кодогене);
- `<< >>`: сдвиг ≥ ширины → `сдвиг ≥ ширины {kind}`;
- касты: вне диапазона → `переполнение при {target}()`;
- `~` — маска ширины (не переполняет), сравнения/and/or/not — bool.

Обычный прогон интерпретатора (`eatc run` рантайм-путь) не меняется —
строгий путь только под `_comptime_mode`.

### 9.2 Шаг и бюджет

Шаг: вход в вычисление выражения (eval), вход в инструкцию
(exec_stmt), виток цикла for (и loop). `requires true`/`ensures true`
не вычисляются (0 шагов). Границы for-диапазона и узлы самих
const-выражений — const_eval-уровень, шаги не считаются. Бюджет
`MAX_COMPTIME_STEPS = 1_000_000` — на каждый **верхнеуровневый
comptime-вызов** в const-выражении; ленивые резолюции const и
вложенные вызовы внутри тела копятся в бюджете корня (счётчик не
сбрасывается). Превышение → `comptime: превышен предел шагов
(предел 1000000)`. Глубина comptime-вызовов ≤
`MAX_COMPTIME_CALL_DEPTH = 64` → `comptime: превышена глубина
вызовов (предел 64)`.

Const-выражения comptime-констант (вне тел): узлы не типизированы,
поэтому операции фитятся к **типу объявляемой константы** (строго,
тексты кодогена), деление/остаток — trunc (семантика тел, не floor
классического const_eval). Обычные const'ы без вызовов идут прежним
путём (floor, произвольная точность до финальной проверки) — их
поведение не меняется.

### 9.3 Годность A1 — скалярное подмножество

Функция comptime-годна (ярус A1) ⇔ транзитивно по графу вызовов:
(а) не достигает нечистых встроенных (read_byte, write_byte,
write_span, write_err_byte, exit, arg_count, arg_len, arg_byte,
print, write) и не extern; (б) тело в скалярном подмножестве:
инструкции block/let/assign/if/elif/else/for-range/return/break/
assert/exprstmt/discard; выражения int/bool/name/binop/unary/call;
типы let/параметров/возврата — целые и bool. Вне подмножества (char,
str, self/методы, member/index, struct/array-литералы, match, loop,
Result/Option и конструкторы) — не годна в A1 (расширение — A2/A3).
Тексты (Python): `{f} не comptime-годна: транзитивно вызывает аксиому
ОС или extern (§5)` и `{f} не comptime-годна: тело вне скалярного
подмножества (ярус A)`. Ошибки — с координатами const-декларации.

### 9.4 Ленивая резолюция и порядок

Отложенные const'ы (значение содержит вызов) собираются в фазе
деклараций с типом и значением-заглушкой; вычисляются в фазе 3.5
(после типизации тел) в порядке объявления; Name на ещё не
вычисленную const резолвит её инлайн; цикл const→func→const →
`цикл в comptime-константе {name}`. trap/бюджет при вычислении →
ошибка компиляции с координатами обрабатываемой const-декларации
(включая случай, когда trap случился в ленивой зависимости). Вызовы
в размерах массивов/границах циклов/ёмкостях строк — ошибка яруса A
(`…comptime-вызовы разрешены только в const-декларациях…`).

### 9.5 Дампы и место в конвейере

`eatc sig` печатает `const … = {значение}` → sig-путь обязан
вычислить comptime-const'ы: Python — dump_signatures догоняет тела и
фазу 3.5 при наличии отложенных; selfhost — SigMain при отложенных
догоняет 3b (тела) + 3.5. typed/ir-пути получают 3.5 в run_typed.
Байт-паритет гейта обязателен на успешных программах (значения в
sig/typed/ir дампах); neg-пути verify_selfhost не сверяет (`if eatc
… ; then` — отказ эталона пропускает стадию), поэтому тексты
selfhost-ошибок упрощены (без имён — в selfhost-ошибках интерполяции
имён нет нигде), но детерминированы.

### 9.6 Архитектура зеркала: методы Check, НЕ отдельный Eval.eat

Отдельный `Eval.eat` невозможен: методам чужого struct'а пришлось бы
передавать `Check` (пулы Parser + таблицы) параметром, а вызов из
var-self-метода Check с self-аргументом → caller-копия всего Check
(алиасинг-правило SPEC §5.4) — неприемлемо для пулов. Вычислитель —
секция полей и методов `struct Check` (как const_eval): стеки машины
(work-стек узел/фаза, стек значений kind+u64+i64, кадры локалей
64×66), драйвер — один ограниченный цикл с ev_step-диспетчером.
Представление значений: kind + пара (u64 для беззнаковых/bool,
i64 для знаковых) — native-арифметика с пре-проверками переполнения
(как trap-условия кодогена), никаких 128-битных промежуточных.
