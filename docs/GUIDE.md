# Руководство по EATLang

Практический учебник языка. Формальная спецификация — в
[SPEC.md](../SPEC.md); здесь — как на языке писать.

## 1. Философия за одну минуту

EATLang построен вокруг одной идеи: **10 правил NASA (Power of 10)
нельзя нарушить, потому что в языке нет средств их нарушить**.

Отсюда следует всё остальное:

- нет кучи → размер массива и ёмкость строки — часть типа;
- нет рекурсии → граф вызовов — DAG, глубина стека известна точно;
- нет `while` → каждый цикл имеет границу, каждая программа завершается;
- нет исключений → ошибки — значения (`Result`), их нельзя «забыть»;
- нет warnings → всё либо корректно, либо ошибка компиляции.

Язык **тотальный**: любая программа доказуемо завершается (кроме
одного разрешённого `loop` в `main` — модель реактивной системы).
Компилятор печатает верхнюю границу памяти программы как артефакт
сборки — это возможно только благодаря перечисленному выше.

## 2. Быстрый старт

Нужны [uv](https://docs.astral.sh/uv/) и clang.

```sh
uv sync                                # окружение (единственная
                                       # зависимость — llvmlite)
make check                             # проверить все примеры
make run_hello_world                   # запустить интерпретатором

make eatc                              # компилятор EATLang из Python-бутстрапа
make eatc_self                         # компилятор, собранный самим собой
make binary SRC="examples/math/Math.eat"   # компиляция + линковка
make run    SRC="examples/math/Math.eat"   # запустить build/Math

# по шагам: .eat → LLVM IR → бинарник (модули через пробел, последний — главный)
make compile SRC="Mod.eat Main.eat"    # → build/Main.ll
make link    SRC="Mod.eat Main.eat"    # → build/Main
# рантайм-модуль selfhost/Rt.eat подставляется первым автоматически
# библиотека lib/ (Ascii, Buf, Const, Fmt, Hex, Io, Num, Parse) подключается тем же списком:
make run SRC="lib/Ascii.eat lib/Num.eat examples/modules/Main.eat"

# либо импортами (docs/MODULES_PLAN.md §2): драйвер сам строит DAG
# модулей и подставляет Rt.eat и lib/; пути — от корня (--lib .)
#
#   import {
#       is_digit,
#       digit_value as digit,
#   } from "lib/Ascii.eat"
#
# модуль публикует имена export-блоком (первым блоком файла):
#   export { fmt_u32, Dec }
# каждое имя — явно (wildcard нет); коллизия и неиспользуемый
# импорт — ошибки, переименование — as с любой стороны
PYTHONPATH=src uv run python -m eatc run --lib . examples/modules/Main.eat
```

Три команды компилятора:

| Команда | Что делает |
| --- | --- |
| `eatc check <файлы>` | парсинг, проверки Power of 10, типы, исполнение test-блоков |
| `eatc run <файл>` | check + запуск `main` интерпретатором |
| `eatc build selfhost/Rt.eat <файлы> [-o out]` | check + верификация + LLVM → нативный бинарник + отчёты |

Для микроконтроллеров у `build`/`ir` есть флаг `--trap-codes`:
вместо текстов trap-сообщений во флеш попадают числовые коды
(`trap N` в stderr), а таблица `; trap <код>: <сообщение>` уходит
комментариями в хвост `.ll` и в файл `<out>.trapmap` рядом с
бинарником (у Mos6502 это −98 % константных данных).

Первая программа:

```text
func main() {
    print("Hello, world!")
}
```

## 3. Переменные и типы

Переменная объявляется с типом и инициализатором — оба обязательны.
По умолчанию она неизменяема; мутабельность — осознанный выбор:

```text
func main() {
    let a: i32 = 10        # неизменяемая
    var b: i32 = 5         # изменяемая
    b = b + a
    print("b = {b}")
}
```

Типы:

| Тип | Описание |
| --- | --- |
| `i32`, `u32`, `u16`, `u8`, `u64`, `i64` | целые; переполнение — trap, а не молчаливый wrap |
| `bool` | `true` / `false` |
| `char` | один байт-символ |
| `[T; N]` | массив ровно из N элементов |
| `str<N>` | строка ёмкостью до N байт |
| `Result<T, E>`, `Option<T>` | встроенные sum-типы для ошибок |

Неявных преобразований **нет**: `i32 + u32` — ошибка компиляции.
Явные преобразования — `i32(x)`, `u32(x)`, `u16(x)`, `u8(x)`, `u64(x)`,
`i64(x)`; выход за диапазон — trap (или статически доказанная
безопасность, см. §9).

Целые литералы — десятичные (до 2^64−1) и шестнадцатеричные
(до 16 hex-цифр):

```text
const CRC_POLY: u32 = 0xEDB88320
const SHA_K0: u64 = 0x428A2F98D728AE22
let masked: u32 = flags & ~0x0F    # ~ — битовое NOT (только беззнаковые)
```

Битовые операции (`& | ^ << >> ~`) определены только для беззнаковых
типов; у сдвигов величина ограничена шириной типа (иначе trap), `~`
из типа не выходит и trap'а не имеет. `u16` добавлен для МК
(ADC/DMA-буферы, таблицы регистров): `[u16; N]` вдвое плотнее `[u32; N]`.
`u64`/`i64` — для SHA-512, длин в битах и времени без заворота; на
32-битных МК их деление/умножение стоят вызова `__aeabi_*`
(mcu/common/eabi64.c) — в горячих циклах M0 предпочитайте u32.

Мутабельного глобального состояния не существует. На верхнем уровне
допустимы только `const`, `func`, `struct`, `enum`, `test`:

```text
const LIMIT: u32 = 5
```

Строки поддерживают интерполяцию — выражение в `{}` типизируется
компилятором:

```text
print("итого {a + b}, флаг {flag}")
```

## 4. Функции и контракты

Контракты — часть сигнатуры. Правило 5 требует: у функции
с параметрами — `requires`, с возвращаемым значением — `ensures`.
Отказ от контракта только явный: `requires true` видно на ревью.

```text
func clamp(x: i32, lo: i32, hi: i32) -> i32
    requires lo <= hi
    ensures result >= lo and result <= hi
{
    if x < lo {
        return lo
    }
    if x > hi {
        return hi
    }
    return x
}
```

В `ensures` доступно имя `result`. Нарушение контракта — trap
с координатами. Но сначала верификатор попробует доказать контракт
статически — доказанные проверки не попадают в бинарник (§9).

Результат вызова **нельзя молча отбросить** (правило 7):

```text
let m: i32 = clamp(15, 0, 10)   # использовать
discard clamp(15, 0, 10)        # или отбросить явно
clamp(15, 0, 10)                # ошибка компиляции
```

Функция не длиннее 60 statements, не больше 6 параметров — иначе
ошибка компиляции (правило 4 и пределы §6 спецификации).

### extern: граница с C (микроконтроллеры)

Периферию МК (GPIO, таймеры, SPI) язык видит через `extern func` —
объявление C-функции без тела:

```text
extern func gpio_set(pin: u32, on: bool)
    requires pin <= 29

extern func systick_ms() -> u32
    ensures true
```

Реализация пишется на C рядом с проектом и линкуется по имени
(см. mcu/). Правила границы:

- аргументы и возврат — скаляры; в аргументах допустимы массивы
  `[u8|u16|u32|u64; N]` (C получает указатель read-only);
- `requires` проверяется у вызывающего как обычно; `ensures` и
  завершаемость C-кода — на вере (та же граница, что у аксиом ОС);
- интерпретатор extern не исполняет — программа с такими вызовами
  проверяется бинарником (для МК — QEMU).

## 5. Управление

`if` / `elif` / `else` — как везде. `while` в языке **нет** — цикл
обязан иметь границу, известную до запуска:

```text
for i in 0..10 {          # константный полуинтервал [0, 10)
    print("{i}")
}

let data: [i32; 3] = [1, 2, 3]
for x in data {           # число итераций — из типа массива
    print("{x}")
}
```

Границы диапазона — литералы или `const`. Переменная в границе —
ошибка компиляции («не константа времени компиляции, правило 2»).
Идиома «пока не найдём» выражается через `for` + ранний `return`.

Единственная бесконечная конструкция — `loop`, разрешён только
в `main` и не более одного (модель главного цикла реактивной
системы); внутри допустим `break`:

```text
import {
    read_line,
} from "lib/Io.eat"

func main() {
    loop {
        match read_line() {
            Ok(line) {
                print("echo: {line}")
            }
            Err(_) {
                break
            }
        }
    }
}
```

Недостижимый код после `return`/`break` — ошибка компиляции
(правило 10). Неиспользуемая переменная — тоже (заглушка — `_`).

## 6. Структуры и методы

Структуры живут на стеке и передаются **по значению** — испортить
данные вызывающего невозможно: мутирует только `var self`. Физическую
копию компилятор элидирует — параметры читаются по ссылке, а при
перекрытии аргумента с `var self`-получателем (`s.append_str(s)`)
копию делает вызывающий (SPEC §5.4):

```text
struct Point {
    x: i32
    y: i32

    func add(self, other: Point) -> Point
        requires true
        ensures true
    {
        return Point { x: self.x + other.x, y: self.y + other.y }
    }
}

func main() {
    let p: Point = Point { x: 1, y: 2 }
    let q: Point = p.add(Point { x: 3, y: 4 })
    print("({q.x}, {q.y})")
}
```

`self` неизменяем. Указателей и ссылок в языке нет — методы не могут
мутировать вызывающего, «алиасинг» невозможен по построению.

## 7. Enum, match и ошибки

```text
enum Mode {
    Idle,
    Active,
    Failed
}

func describe(m: Mode) -> str<16>
    requires true
    ensures true
{
    match m {
        Idle   { return "idle" }
        Active { return "active" }
        Failed { return "failed" }
    }
}
```

`match` обязан быть исчерпывающим — пропущенный вариант это ошибка
компиляции, а не сюрприз в рантайме.

Исключений нет. Ошибающиеся операции возвращают `Result` — и правило 7
не даст его проигнорировать (`read_line`/`parse_i32` — из lib/,
шапка import опущена):

```text
func main() {
    print("Введите число: ")
    match read_line() {
        Ok(line) {
            match parse_i32(line) {
                Ok(n) {
                    print("получено {n}")
                }
                Err(_) {
                    print("это не число")
                }
            }
        }
        Err(_) {
            print("ошибка ввода")
        }
    }
}
```

Создавать `Result`/`Option` умеют и ваши функции — конструкторами
`Ok(x)`, `Err(e)`, `Some(x)`, `None`. Они **контекстные**: пишутся
только там, где ожидаемый тип известен точно — в `return` функции
с типом возврата `Result`/`Option`, в `let` с такой аннотацией и в
payload вложенного конструктора (`return Ok(Some(x))`):

```text
enum DivError {
    ByZero,
}

func checked_div(a: u32, b: u32) -> Result<u32, DivError>
    requires true
    ensures true
{
    if b == 0 {
        return Err(DivError.ByZero)
    }
    return Ok(a / b)
}
```

В аргументе вызова или выражении без аннотации `Ok(1)` не
скомпилируется — свяжите значение `let r: Result<...> = Ok(1)`.

Встроенные функции: `print(s)`, `read_byte() -> Result<u8, IoError>`,
`len(x) -> u32`. Сами `read_line() -> Result<str<256>, IoError>` и
`parse_i32(s) -> Result<i32, ParseError>` — библиотечные
(`lib/Io.eat`, `lib/Parse.eat`): в программе с import-шапкой их
подключает драйвер (`eatc run --lib . Main.eat`), в cat-режиме —
файл в списке (`eatc run selfhost/Rt.eat lib/Parse.eat App.eat`).

Байтовый вывод: `write_byte(b: u8)` — один байт;
`write_span(a: [u8; N], off: u32, len: u32)` — диапазон
`a[off..off+len)` одним вызовом ОС (батч: на порядки дешевле цикла
`write_byte` для длинных диапазонов; `off + len > N` — trap).

## 8. Тесты — часть языка

`test`-блоки исполняются **при каждой сборке**; упавший `assert` или
нарушенный контракт вызываемой функции = сборка не удалась. В бинарник
тесты не попадают.

```text
func double(x: i32) -> i32
    requires x > -1073741824 and x < 1073741824
    ensures result == x * 2
{
    return x * 2
}

test double {
    assert double(21) == 42
    assert double(0) == 0
}
```

Связка «контракт + тест» ловит целый класс багов до запуска: тест
вызывает функцию → рантайм-проверка `ensures` замечает расхождение
между обещанием и реализацией.

## 9. Статическая верификация

При `eatc build` верификатор (интервальный + реляционный анализ)
пытается доказать каждую проверку. Доказанное **удаляется из
бинарника**; отчёт показывает счёт:

```text
OK build/All — stack depth: 3, tests passed: 1
  верификация: доказано 9 из 16 проверок (7 остаётся в рантайме)
    assert: 0/4, cast: 1/1, деление: 1/1, ensures: 4/4, ...
```

Анализ консервативен: ложных доказательств не бывает, недоказанное
честно остаётся trap'ом.

Как писать код, который доказывается:

- **Давайте верификатору факты через `requires`.** `requires b != 0`
  перед делением на `b` доказывает «деление на ноль невозможно» — и
  проверка исчезает из кода.
- **Ограничивайте диапазоны.** `requires x >= 0 and x <= 1000`
  делает доказуемыми переполнения всей арифметики над `x`.
- **Индексируйте под контрактом.** `requires j < 3` доказывает
  границы `data[j]` для `[i32; 3]`.
- **Простые `ensures` доказываются реляционно**: `ensures result >= a`
  у функции, возвращающей `a` в ветке `a > b`, доказан — анализ
  связывает `result` с возвращаемым выражением и знает факты ветки.
- **`ensures result == x * 2` доказывается структурно**, если тело
  возвращает то же выражение.
- **Линейные циклы ускоряются**: переменная, меняющаяся на константу
  за итерацию (`count = count - 1`, `sum = sum + 7`), получает точный
  интервал в теле и после цикла — `assert sum == 700` после ста
  итераций `+7` доказан. Нелинейные обновления (`total = total + x`)
  сбрасывают знание до диапазона типа — такие проверки остаются
  в рантайме. Это нормально.

## 10. Отчёт о памяти

`eatc build` печатает верхнюю границу памяти — точный расчёт возможен,
потому что рекурсия запрещена и все размеры статичны:

```text
  память (§8): стек худшей цепочки ≤ 1628 Б, статические данные 479 Б
    кадр main: 844 Б
    кадр describe: 784 Б
```

Стек считается по худшему пути в DAG вызовов из фактических размеров
кадров (ABI-размеры LLVM). Куч нет, `malloc` в сгенерированном коде
не встречается.

## 11. Каталог «выстрелов в ногу», которые не скомпилируются

| Попытка | Ответ компилятора |
| --- | --- |
| глобальная переменная | «на верхнем уровне допустимы только func, struct, enum, const, test (правило 6)» |
| рекурсия, в т.ч. косвенная | «рекурсия запрещена (правило 1): цикл вызовов f -> g -> f» |
| `while`-подобный цикл с переменной границей | «не константа времени компиляции (правило 2)» |
| выбросить результат вызова | «результат отброшен молча; используйте значение или discard (правило 7)» |
| `match` без варианта | «match не исчерпывающий, пропущено: Err (правило 7)» |
| функция с параметрами без `requires` | «обязана иметь requires; осознанный отказ — `requires true` (правило 5)» |
| присваивание в `let` | «объявлена как let — неизменяема» |
| неиспользуемая переменная | «переменная не используется (правило 10)» |
| код после `return` | «недостижимый код (правило 10)» |
| `i32 + u32` | «разные типы (неявных преобразований нет)» |
| shadowing | «имя уже объявлено: shadowing запрещён (правило 6)» |
| функция длиннее 60 statements | «длиннее 60 (правило 4)» |

А то, что нельзя поймать статически, ловится trap'ом в рантайме
с точными координатами: переполнение, деление на ноль, выход за
границы, превышение ёмкости строки, нарушенный контракт.

## 12. Пределы

Компилятор построен по модели TeX: фиксированные ёмкости вместо кучи,
превышение — штатная ошибка `capacity exceeded`. Основные пределы:
60 statements на функцию, 6 параметров, вложенность блоков 8, глубина
выражения 32, 131 072 токена в файле. Полный список — SPEC.md §6.

Эти же пределы — фундамент долгосрочной цели: компилятор EATLang,
написанный на EATLang, без единой динамической аллокации.
