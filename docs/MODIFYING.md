# Как модифицировать язык

Инструкция для любой правки EATLang: компилятора, синтаксиса,
встроенных функций, пределов. Контекст конвейера — в
[PROCESSES.md](PROCESSES.md).

## 0. Инварианты (нарушать нельзя)

1. **Python-бутстрап — эталон, self-hosted — целевой продукт.**
   Любая правка языка в `src/eatc/` обязана быть зеркально повторена
   в `selfhost/*.eat`, иначе `make verify_selfhost` краснеет.
2. **Паритет — байт-в-байт.** Дампы `lex/parse/sig/typed` и текст
   LLVM IR self-hosted компилятора diff'ятся с эталоном без допусков.
   Это относится и к текстам ошибок, и к trap-сообщениям.
3. **Интерпретатор — эталон семантики исполнения** и байт-точен:
   строки — байты, как в нативном рантайме. `make verify` требует
   вывод бинарника == вывод интерпретатора.
4. **Изменения языка (синтаксис, семантика, встроенные) — только с
   решением пользователя.** Примеры зафиксированных решений:
   `break` в любом цикле, но ≤ 1 на цикл (стиль MISRA); `exit(code)`
   только в `main` и единственный; битовые операторы только для
   беззнаковых (u64/u32/u16/u8) с приоритетами Rust. Если задача
   упирается в отсутствующую конструкцию — это находка про язык,
   а не повод её молча добавить.
5. **Power of 10 зашиты в язык.** Ни одна правка не должна открыть
   обходной путь: куча, рекурсия, цикл без границы, отброшенный
   результат, глобальное состояние. Как каждое правило воплощено и
   где проверяется — [POWER_OF_10.md](POWER_OF_10.md).

## 1. Порядок работ (для любой правки компилятора)

1. **Python-слои** в порядке конвейера:
   `lexer.py → parser.py → checks.py → typechecker.py →
   interpreter.py → verifier.py → codegen.py`
   (плюс дамперы `astdump.py`/`sigdump.py`/`typeddump.py`, если
   меняется форма дампа).
2. **Смоук на /tmp**: минимальный `.eat`-файл с новой конструкцией,
   `eatc check`, `eatc run`, `eatc build` — до того, как трогать
   selfhost.
3. **Selfhost-зеркало**: та же логика в `Tok.eat` / `Lexer.eat` /
   `Ast.eat` / `Parser.eat` / `Check.eat` / `Ir.eat`.
4. **Полная сверка**: `make check verify verify_suite
   verify_selfhost verify_bootstrap verify_trapcodes verify_sig
   verify_selfhost_verify verify_selfhost_verify_all`.
5. **Документация**: SPEC.md (грамматика, семантика, пределы),
   docs/GUIDE.md (как этим пользоваться), README.md (статус).

## 2. Кодировки, которые нельзя перенумеровать

Self-hosted компилятор кодирует сущности числами; дампы сверяются
по именам, но самоприменение ломается, если коды съезжают.

- **Токены (`selfhost/Tok.eat`)**: `54 NEWLINE`, `55 EOF`,
  `56..60` — битовые операторы, `61 TILDE`, `62 EXTERN`,
  `63 IMPORT`, `64 EXPORT`, `65 FROM`, `66 AS` (ключевые слова с
  кодами вне диапазона 4..29 — нумерацию не двигаем), `67 MODULE`
  (директива драйвера `#module "путь"`, значение — путь),
  `68 EXTEND`. Новые виды токенов добавлять **строго в конец**.
- **Коды операций AST**: `1..15` — исходные, `16..20` — битовые
  (`| ^ & << >>`), `21` — `~`. Новые — тоже в конец.
- **Виды узлов AST**: `0..49` — исходные, `50 extern`, `51 module`,
  `52 import`, `53 export`, `54 bind` (alias — спан в nx/ny),
  `55 extend` (методы вне блока struct; сливает `extend_pass` в
  Check до всех проходов). Виды типов Check.eat: `0..11` — исходные,
  `12 u16`, `13 u64`, `14 i64`.
- **Коды встроенных вызовов (`xa` в Check/Ir)**: `100..103` — касты
  i32/u32/u8/u16, `104` char, `105..110` — len/write/print и другие
  встроенные, `111 u64`, `112 i64`, `113 Ok`, `114 Err`, `115 Some`
  (конструкторы; None — путь имени, код не нужен). Новые — строго
  в конец.
- **Интринзики overflow (`Ir.eat`)**: id `1..3` i32, `4..6` u32,
  `7..9` u8, `10..12` u16 (i16), `13..15` u64 и `16..18` i64 (i64).
- **Виды ссылок-значений (`Ir.eat`)**: `0..4` — исходные, `5` —
  широкая константа (модуль > 2^32−1 в пуле `cwm`).
- **Trap-сообщения**: тексты в `selfhost/Ir.eat` (`trap_glob`)
  должны совпадать с `src/eatc/codegen.py` **до байта** —
  сообщения дедуплицируются в глобалы `@"str.N"`, и любое
  расхождение сдвигает нумерацию всех последующих строк IR.

## 3. Чеклисты по типу правки

### Новый оператор / синтаксис

- [ ] решение пользователя: типы операндов, приоритет, trap-семантика
- [ ] `tokens.py`, `lexer.py` (+ `Tok.eat`, `Lexer.eat`)
- [ ] `parser.py`, `ast_nodes.py`, `astdump.py` (+ `Ast.eat`, `Parser.eat`)
- [ ] `typechecker.py`, `typeddump.py` (+ `Check.eat`)
- [ ] `interpreter.py` — эталон семантики, включая trap'ы
- [ ] `verifier.py` — как оператор участвует в доказательствах
- [ ] `codegen.py` (+ `Ir.eat`) — идентичный IR
- [ ] SPEC.md §4 (EBNF) и §5; GUIDE.md; кейс в `tests/verify/` при
      новой trap-семантике; при необходимости — стресс в `tests/lex|parse|ir/`

### Новая встроенная функция

**Критерий ядра**: в ядро (компилятор/Rt.eat) попадает только
(а) аксиома ОС или (б) то, без чего не работает тайпчекер/кодоген.
Всё остальное — `.eat`-модуль в `lib/` (docs/plans/MODULES_PLAN.md §7),
подключаемый списком файлов после `selfhost/Rt.eat`.

Сначала решить, **аксиома это, библиотека или граница проекта**.
Если функция нужна конкретному проекту (драйвер периферии МК) —
это не встроенная, а `extern func` + C-файл проекта; компилятор
не трогается. Для языка:

- **Аксиома ОС** (нельзя выразить в языке): `runtime.c` +
  `codegen.py` + `interpreter.py` + `Ir.eat` + SPEC §7. Аксиом
  сейчас шесть — шим runtime.c из семи функций, у trap две формы
  (сообщение и код); каждая новая — решение пользователя.
- **Библиотечная**: реализуется на EATLang в `selfhost/Rt.eat`
  с контрактами и test-блоками; компилятор менять не нужно.

### Новый пример / программа на EATLang

Компилятор не трогать. Если языку чего-то не хватает — остановиться
и сообщить (см. инвариант 4). Идиомы без изменений языка:
wrap-арифметика — явные маски («маска — это доказательство»:
верификатор снимает такие проверки статически); маски и адреса —
hex-литералами (`0x…`, ≤ 8 цифр), инверсия — `~x`. Добавить цель
`run_<имя>` в Makefile и, если пример собирается в бинарник, — сверку
в `verify`.

Стиль исходников `*.eat` (всего репозитория): строка ≤ 100 кодпойнтов
Unicode — проверяет `tests/check_style.py` первым шагом `make check`
(REFACTOR_SELFHOST_PLAN, этап 1). Длинные конструкции переносить
внутри `()`/`[]` (лексер подавляет NEWLINE в скобках); elif-таблицы —
по ветке на строку в стиле `} elif … {`; длинное условие — обернуть
в `()`, продолжение с двойным отступом и `and`/`or` в начале строки.

## 4. Определение готовности

Правка закончена, когда:

```sh
make check verify verify_suite verify_selfhost verify_bootstrap verify_trapcodes verify_sig verify_selfhost_verify verify_selfhost_verify_all
```

(`verify_sig` — контроль дрейфа интерфейса lib/: при осознанном
изменении сигнатур/экспортов — `make regen_sig` и коммит diff'а
`tests/sig/lib.sig`.)

проходит целиком (bench — по ситуации, при правках пределов и
производительности), а SPEC.md/GUIDE.md/README.md отражают новое
поведение. Частично зелёное состояние не коммитится.
