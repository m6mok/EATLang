# Правила создания контейнеров

Действуют при **любой** работе с файлами проекта в контейнере (сборка
образа, прогон гейта, эксперимент с тулчейном). Архитектура и план —
[docs/plans/CONTAINERS_PLAN.md](plans/CONTAINERS_PLAN.md); образ и обвязка —
[containers/](../containers/). Процесс «контейнер-на-задачу» (цели
`task-*`, слияние после зелёного) — этап 1 плана, ещё **не** заведён; ниже
только правила **создания и гигиены** контейнеров, они стабильны и
действуют уже сейчас.

## 1. Чем и как

- **Только Podman.** Rootless, без демона, без привилегий. Docker не
  использовать. На macOS — через `podman machine` (Linux-VM: числа
  бенча в VM другие, перф-эталон остаётся нативным — см. CONTAINERS_PLAN
  §2).
- **Только пиннутый образ** из [containers/Containerfile](../containers/Containerfile)
  (`eatlang-dev:<этап>`), не `latest`. Версии тулчейна (база, clang/LLVM,
  go, rust, uv, llvmlite) — точные теги; менять пин = решение
  пользователя (CONTAINERS_PLAN §10).

  ```sh
  podman machine start                                    # на macOS
  podman build -f containers/Containerfile -t eatlang-dev:etap0 containers
  containers/run-gate.sh <путь-к-дереву>                  # полный гейт в образе
  ```

## 2. Как запускать (изоляция)

- **`podman run` — всегда `--rm`.** Контейнер одноразовый, после выхода
  ничего не оседает.
- **Монтировать ЧИСТЫЙ git worktree на HEAD**, а не рабочее дерево с
  незакоммиченным WIP: чужой (или свой) грязный WIP ломает полный гейт
  независимо от твоей правки (FAULTS 2026-07-16). Пример:
  `git worktree add ../EATLang-gate HEAD` → монтируем его.
- **venv — В КОНТЕЙНЕРЕ, не в смонтированном дереве.** У хостового
  `.venv` llvmlite под macOS, в linux-контейнере он не импортируется:
  `-e UV_PROJECT_ENVIRONMENT=/root/eat-venv` (так делает
  [run-gate.sh](../containers/run-gate.sh)).
- Rootless, без `--privileged`, без сети наружу в гейте (сетевые примеры
  сверяются транскриптом `EAT_NET`, не реальным сокетом).

## 3. Гигиена (чистить после каждого эксперимента)

Цель — не засорять диск VM Podman (образ EATLang ~2 ГБ; фальстарты
сборки оставляют dangling-слои).

- **После `podman run`** — контейнер уже удалён (`--rm`); проверить
  `podman ps -a`, что своих не осталось.
- **После `podman build`** — удалить **свои** промежуточные/dangling-
  слои (`<none>` от фальстартов), оставив только нужный финальный образ.
- **Перед новой сборкой/контейнером** — посмотреть возраст **своих**
  артефактов. Обычная работа агента ≤ 3 ч, значит всё **старше 12 ч** —
  почти наверняка устаревший leftover, можно удалять.

### Граница (обязательно)

Правило «> 12 ч → удалять» и любая чистка действуют **строго к
артефактам агента** (образы `eatlang-dev*`, свежие свои `<none>`-слои).

- **Никогда не делать** блэнкет-`podman system prune` / `image prune -a`.
- На машине живут **другие проекты пользователя** (напр. вышедшие
  контейнеры и тегированные образы fastapi/postgres/buildkit — месяцы) —
  их и dangling чужих сборок **не трогать** без спроса.
- **Не удалять** образ, намеренно оставленный для текущей работы (напр.
  под открытую эскалацию), даже если он подходит под чистку по имени.

Удалять по явному ID своих слоёв или по своему тегу:

```sh
podman ps -a                          # свои контейнеры? (должно быть пусто при --rm)
podman images --format \
  'table {{.Repository}}:{{.Tag}} {{.Size}} {{.CreatedSince}} {{.ID}}'
podman rmi <ID-своего-слоя> ...       # точечно; на общий предок Podman откажет — безопасно
```
