Статус на 2026-07-14, вечер. Эталон всех замеров —
`build/bench/baseline_3cebb58.txt` (двадцать первая итерация
[FINDINGS](../tests/bench/FINDINGS.md)); каждая работа ниже меряется
против него секциями `make bench` и `tests/verify_catalog.py`.
Выполнено к этому моменту (не переоткрывать): слой 0 — runtime.c с
`-O2` (Read +33 %, Io +26 %, 0be075c); слой 1а — llvm.assume из
доказанных requires (нейтрален, канал под верификатор, b88566a);
трек 3, ядро — инварианты пулов/полей + сентинел-дырки
(5270 → 5356 из 6945, 7e02d1e). Уроки, которые ограничивают выбор:
llvmlite не векторизует циклы с ветвлением и масочные редукции;
предсказанные trap-ветки на Apple Silicon почти бесплатны (~3–5 %) —
выигрыш снятых проверок ищи в размере кода и на in-order МК.

## 1. Верификатор: добить остаток (трек 3, продолжение)

### 1.1 Дизъюнктивный домен «интервал с дыркой» через циклы — главный рычаг

**Остаток**: 994 bounds (из них Check.eat 511), 167 requires,
294 overflow — большинство упирается в две формы, теряющие сентинел
на loop-join ([VERIFICATION_PLAN](VERIFICATION_PLAN.md), хвост):

- аккумулятор поиска: `var f: u32 = NONE … в цикле f = fi …
  return f` (fh_find и все find_*/lookup*);
- обход списка: `c = self.p.nns[c/65536][c%65536]` до `c == NONE`.

**Дизайн.** Значение вида `[lo, hi] ∪ {sent}` становится
первоклассным в `State.ivs` (третье поле или параллельная карта —
решить по числу точек касания; кандидаты: `_hull`/`_inter`/`_join`,
`_refine_neq`/`_refine_cmp`, `_flow_for`). Цикл: кандидат-инвариант
пути = дырочное объединение (вход ∪ все присваивания тела),
валидируется нисходящей итерацией тела в **probe-режиме** — флаг,
при котором `_mark` молчит: AND-слияние повторных оценок не должно
травиться пробными прогонами. Сходимость ≤ 2–3 итерации, иначе
консервативный kill как сейчас.

**Метрики**: каталог (цель — bounds ниже ~500), «доказано N из M» в
build, `__text`/trap-глобалы секции size (каждая снятая проверка —
минус ветка и минус строка), TrapBench-класс не регрессирует.
**Цена**: только verifier.py + кейсы (42+: аккумулятор, список,
негатив «дырка не выживает при записи произвольного значения»).
**Риск**: probe-режим обязан не оставлять следов (checks, req_sites,
summaries, сбор инвариантов) — снапшот/флаг на все копилки.

### 1.2 Реляционные инварианты полей (assert 0/103, ensures 27)

Класс `assert self.ln + o.ln <= 256` (Rt.eat append_str) — интервалы
не выражают сумму двух полей. Направление: разностные ограничения
между полями получателя как инварианты struct (`rels` уровня типа),
собранные тем же пассовым механизмом. Отдельно оценить, сколько из
103 реально этой формы (каталог по видам × файлам). Возможный итог —
«не окупается интервалами, ждёт SMT-класса» — тогда честно закрыть
направление в VERIFICATION_PLAN.

### 1.3 Мелочь: hive через границу модулей

`_contract_iv` (межмодульный вызов) не знает hive-summary —
find_*-функции, уехавшие в другие модули после этапа модульности,
теряют дырку на границе. Поправить симметрично _apply_summary.

## 2. Кодоген: факты верификатора → LLVM, копии

### 2.1 !range на чтениях пулов (build-only, дешёвый)

Верификатор теперь ЗНАЕТ инварианты пулов (`pool_iv`/`pool_sent`) —
кодоген может ставить `!range` метаданату на load'ы из пулов
(codegen.py, `indexed_ptr`-потребители). Механизм передачи —
как флаги узлов: верификатор аннотирует Index-узлы атрибутом
`elem_range=(lo, hi)`; build-only, канон `eatc ir` не меняется
(прецеденты: nsw/nuw, llvm.assume). После снятия проверок это
единственный канал, которым диапазон пула доходит до LLVM.
**Метрика**: TrapBench/BankBench/compiler-секция; ожидание скромное
(суперскаляр), возможен эффект на свёртку масок. Если нейтрально —
зафиксировать и оставить (нулевая цена в рантайме).

### 2.2 sret прямо в слот назначения (изменение канона — оба эмиттера)

`let x: T = f(...)` сегодня: свежая alloca вызывающего → вызов →
memcpy в слот x (`emit_call` codegen.py:1104, `copy_into` :288).
Передавать слот x как sret-указатель напрямую. **RetBench показал:
при инлайне LLVM уже съедает копию (404M вызовов/с)** — выигрыш
только у неинлайнящихся функций (большие тела). Selfhost: машины
компилятора крупные — проверить профилем, сколько memcpy-возвратов
живёт в SelfIr до начала работ. **Цена высокая**: меняется
канонический IR → зеркало selfhost/Ir.eat байт-в-байт, фикспойнт,
все дампы. Делать только если профиль покажет ≥ единиц процентов.

### 2.3 Хвосты трека 4 (selfhost-код, по остаточному профилю SelfIr 0.13 с)

- `lb_tp`/копии в банкованные пулы: аффинная по-банковая форма,
  которую LLVM распознаёт как memcpy (метрики: CopyBench 220M,
  BankBench 250M, compiler-секция); потенциал ~десятки мс;
- `g_hash` инкрементально при записи в пул вместо пересчёта на
  каждый `g_commit` (Ir.eat);
- `write_span_esc`/экранированные спаны дампов — батчить между
  escape-символами (профиль дампов lex/parse: 1.59–2.45 МБ вывода).

Перед каждым — свежий `sample`-профиль нативного SelfIr: после
модульного этапа и инвариантов расклад мог сместиться.

## 3. Рантайм-шим и сборка

### 3.1 Аксиома батч-чтения (решение уровня SPEC §7 — пользователь)

После слоя 0 ввод 466M байт/с против вывода 2091M — путь входа
побайтный принципиально (`read_byte` — Result на каждый байт).
Кандидат: аксиома `read_span(a: [u8; N], off, len) -> u32`
(прочитано байт; 0 — EOF), симметрично write_span, с теми же
bounds-обязательствами (кейс 28 верификатора). Выгодоприобретатель —
лексер selfhost (читает исходник побайтно). **Не начинать без
решения пользователя** (новая аксиома = SPEC §7, шимы host+mcu,
оба кодогена, интерпретатор, паритет). Оценку выигрыша дать заранее
через ReadBench-вариант на write_span-буфере.

### 3.2 LTO как release-режим (замерено, отложено)

−10 % времени самокомпиляции, −29 % размера, +2.75 с линковки
(шестнадцатая итерация). В build-путь не встраивать; кандидат —
флаг `--release`/Makefile-цель для финальных бинарников и МК
(в mcu-сборке уже живёт). Решение пользователя о форме флага.

### 3.3 Порог батчинга write_span (перепроверка)

Гибрид «< 32 байт побайтно, иначе fwrite» калибровался при -O0-шиме;
после -O2 порог мог сместиться. Быстрый замер IoBench с порогами
16/32/64 — одна строка runtime.c.

## 4. МК-профиль: где снятые проверки стоят по-настоящему

- **In-order метрика цены проверок**: прогнать TrapBench/BitsBench
  на QEMU Cortex-M (mps2-an385, инфраструктура `make mcu` готова) —
  честная цена trap-веток там, где нет предсказателя уровня M-чипов;
  завести в bench секцию или mcu/README таблицу;
- **размер после инвариантов**: секция size до/после 7e02d1e на
  SelfIr/Mos6502 (−41 bounds = минус ветки и trap-строки);
- прошивочные порты и периферийный шим — отдельный трек
  ([MCU_PLAN](MCU_PLAN.md)), не смешивать.

## 5. Comptime — вычисление тотальных функций ([COMPTIME_PLAN](COMPTIME_PLAN.md))

Самый крупный отложенный трек (решение «прорабатывать глубоко»
принято, открытые решения §7 плана — за пользователем: синтаксис
яруса A, бюджет шагов, форма const-таблиц, место в очереди).
Главный ценник — вычислитель Eval.eat в selfhost для паритета.
Профиль выгоды: crypto/таблицы/МК; selfhost — косвенно (точечные
интервалы → верификатор).

## 6. Порядок по умолчанию и готовность

1. **1.1 дизъюнктивный домен** (наибольший остаток, только Python) →
   попутно 1.3;
2. **2.1 !range из инвариантов** (дёшево, тот же контур замеров);
3. **4 (МК-метрика цены проверок + размер)** — закрывает вопрос
   «сколько стоят проверки там, где это важно»;
4. **2.3 хвосты по профилю** (lb_tp → g_hash → esc-спаны), каждый —
   отдельный коммит с числами до/после;
5. **3.1 read_span / 3.2 LTO-флаг / 2.2 sret** — по решениям
   пользователя и профилю;
6. **5 comptime** — после решений §7.

Каждый этап: `make check verify verify_suite verify_selfhost
verify_bootstrap verify_trapcodes` целиком зелёный + секции bench
до/после + запись в FINDINGS (стиль итераций). Правки только
verifier.py не требуют зеркала; правки канона IR — оба эмиттера и
фикспойнт; новые аксиомы — решение по SPEC §7 до кода.
