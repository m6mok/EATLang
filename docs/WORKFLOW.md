# WORKFLOW — процесс работы агентов над проектом

Правила ведения работ: планы, вопросы, итоги, параллельность.
Дополняет [PROCESSES.md](PROCESSES.md) (конвейер и команды) и
[MODIFYING.md](MODIFYING.md) (инварианты правок кода). Обязательны
для каждого агента наравне с правилами [AGENTS.md](../AGENTS.md).

## 1. Планы (docs/plans/)

1. **Каждая новая инициатива** — план в
   [docs/plans/](plans/README.md) по шаблону из README (шапка:
   статус · роль · связи).
2. Перед созданием плана — **проверить существующие на коллизии**
   (пересечение поверхностей, дубли, противоречия связям
   «опирается/блокирует»); коллизию решить до начала работ.
3. При каждом удобном случае **актуализировать существующие планы**:
   статусы в шапках и в индексе README, устаревшие «следующие шаги».
4. **Завершённые планы (✅) не дополнять** новыми работами — они
   живут как справка/контракт. Новая работа — новый план со ссылкой
   на завершённый.

## 2. Вопросы к пользователю

К каждому запросу пользователя подразумевается приписка:
**«Задавай, пожалуйста, вопросы, если нужно»**. Уточнять до начала
работ всё, где ответ меняет результат (объём, место, компромиссы);
не спрашивать то, что выводится из репозитория или конвенций.
Отдельно поощряются вопросы об изменениях самого процесса работы
агентов (этот документ).

## 3. Итог этапа / плана

После каждого исполненного этапа плана (и плана целиком) — вывести
пользователю итог. Итог пишется **понятным языком**: без жаргона и
внутренних кодовых имён; специальный термин упоминается только если
без него нельзя — и тогда с определением в одну строку (первое
упоминание). Проверка: итог должен читаться человеком, не смотревшим
в код проекта.

Содержание итога:

1. **Объём**: изменённые строки (`git diff --stat`) и число коммитов.
2. **Время**: за сколько выполнен этап/план (по таймстампам коммитов
   от начала этапа).
3. **Метрики до → после**, цифрами (паритетные файлы, обязательства,
   время прогонов, размеры пулов, число гейтов — что применимо).
4. **Ошибки** исполнения — вывести и записать в
   [docs/plans/FAULTS.md](plans/FAULTS.md) (журнал: дата · план/этап ·
   симптом · причина · как исправлено · урок).
5. **Предложения**: оптимизации и кандидаты в новые планы — с
   сохранением в docs/plans/ (черновиком или строкой backlog).
6. **Коммиты** — так, чтобы не задевать параллельные работы: перед
   правкой смотреть `git status`; файлы с чужими незакоммиченными
   изменениями не трогать и не коммитить, правку в занятый файл
   отложить и передать следующему шагу. Один этап — логичные коммиты
   в стиле истории (`git log --oneline`), каждый коммит проходит гейт.
7. **Следующий шаг**: если проект (трек) продолжается — готовый промт
   следующего этапа для другого агента: в чат (для копипасты) и
   коротко в тело плана («→ следующий шаг»). Если проект завершён —
   предложить пользователю на выбор 3 проекта (ссылки на файлы
   планов; промты для них не пишутся).

## 4. Параллельность, модели, контекст-бюджет

Полная проработка — план
[AGENTS_ORCHESTRATION_PLAN](plans/AGENTS_ORCHESTRATION_PLAN.md)
(параллельные агенты без столкновений, ранжирование задач под модели
и effort, дробление этапов под бюджет контекста). Действующий минимум
до его реализации:

- **Не сталкиваться по файлам**: единица владения — файл; перед
  правкой `git status`, чужие незакоммиченные файлы — читать можно,
  писать нельзя.
- **Дробить этапы** по границам файлов/фаз компилятора так, чтобы
  этап помещался в один сеанс без потери контекста; длинные прогоны
  (полные гейты) — в фоне, с работой в промежутке.
- **Дорогая модель — дорогим задачам**: механические сверки/прогоны
  не требуют максимального effort; дебаг паритета и архитектурные
  решения — требуют. Формальные критерии — в плане оркестрации.
