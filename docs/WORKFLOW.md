# WORKFLOW — процесс работы агентов над проектом

Правила ведения работ: планы, вопросы, итоги, параллельность.
Дополняет [PROCESSES.md](PROCESSES.md) (конвейер и команды) и
[MODIFYING.md](MODIFYING.md) (инварианты правок кода). Обязательны
для каждого агента наравне с правилами [AGENTS.md](../AGENTS.md).

## 1. Планы (docs/plans/)

1. **Каждая новая инициатива** — план в
   [docs/plans/](plans/README.md) по шаблону из README (шапка:
   статус · роль · связи).
2. Перед созданием плана — **проверить существующие на коллизии**
   (пересечение поверхностей, дубли, противоречия связям
   «опирается/блокирует»); коллизию решить до начала работ.
3. При каждом удобном случае **актуализировать существующие планы**:
   статусы в шапках и в индексе README, устаревшие «следующие шаги».
4. **Завершённые планы (✅) не дополнять** новыми работами — они
   живут как справка/контракт. Новая работа — новый план со ссылкой
   на завершённый.

## 2. Вопросы к пользователю

К каждому запросу пользователя подразумевается приписка:
**«Задавай, пожалуйста, вопросы, если нужно»**. Уточнять до начала
работ всё, где ответ меняет результат (объём, место, компромиссы);
не спрашивать то, что выводится из репозитория или конвенций.
Отдельно поощряются вопросы об изменениях самого процесса работы
агентов (этот документ).

## 3. Итог этапа / плана

После каждого исполненного этапа плана (и плана целиком) — вывести
пользователю итог. Итог пишется **понятным языком**: без жаргона и
внутренних кодовых имён; специальный термин упоминается только если
без него нельзя — и тогда с определением в одну строку (первое
упоминание). Проверка: итог должен читаться человеком, не смотревшим
в код проекта.

Содержание итога:

1. **Объём**: изменённые строки (`git diff --stat`) и число коммитов.
2. **Время**: за сколько выполнен этап/план — от первого действия
   этапа (или предыдущего коммита трека) до финального коммита;
   для одно-коммитных этапов таймстампы коммитов не дают длительности.
3. **Метрики до → после**, цифрами (паритетные файлы, обязательства,
   время прогонов, размеры пулов, число гейтов — что применимо).
   «До» — всегда замер на состоянии до правки (снятый заранее или
   откатом своего диффа во временной копии), не оценка по памяти;
   обе стороны — на одной машине и одном дереве.
4. **Ошибки** исполнения — вывести и записать в
   [docs/plans/FAULTS.md](plans/FAULTS.md) (журнал: дата · план/этап ·
   симптом · причина · как исправлено · урок).
5. **Предложения**: оптимизации и кандидаты в новые планы — с
   сохранением в docs/plans/ (черновиком или строкой backlog).
6. **Коммиты** — так, чтобы не задевать параллельные работы: перед
   правкой смотреть `git status`; файлы с чужими незакоммиченными
   изменениями не трогать и не коммитить, правку в занятый файл
   отложить и передать следующему шагу. Один этап — логичные коммиты
   в стиле истории (`git log --oneline`). Гейт и многокоммитный этап:
   полный гейт бежит один раз в конце этапа (§6), поэтому коммиты
   этапа складываются локально и публикуются пакетом после зелёного
   гейта; прервался этап — публикуется только префикс, прошедший гейт.
7. **Следующий шаг**: если проект (трек) продолжается — готовый промт
   следующего этапа для другого агента: в чат (для копипасты) и
   коротко в тело плана («→ следующий шаг»). Если проект завершён —
   предложить пользователю на выбор 3 проекта (ссылки на файлы
   планов; промты для них не пишутся).

Для **разовой работы вне плана-трека** (расследование, точечный фикс
одним коммитом) допустим сокращённый итог: объём, метрики до → после,
ошибки/урок (при наличии — в FAULTS); промт следующего этапа не
пишется, предложения-кандидаты сохраняются по п. 5.

## 4. Параллельность: владение файлами и гейты

Мотивация и история решений — план
[AGENTS_ORCHESTRATION_PLAN](plans/AGENTS_ORCHESTRATION_PLAN.md).

- **Единица владения — файл; владелец — трек.** Трек объявляет
  занятые файлы в шапке своего плана (поле «Рабочие поверхности»).
  Перед стартом этапа агент сверяется с поверхностями планов со
  статусом 🚧 и с `git status`; чужие незакоммиченные файлы — читать
  можно, писать нельзя (исключение — владелец/пользователь прямо
  просит правку в этом файле).
- **Общие файлы** (Makefile, SPEC.md, AGENTS.md, README планов и
  прочие, не принадлежащие одному треку) — очередь правок: если файл
  занят, правка откладывается и передаётся в промт следующего этапа
  (§3.7).
- **Точечный коммит в занятый общий файл** — когда очередь неуместна
  (правку прямо просит владелец/пользователь) и свой ханк не
  пересекается с чужими строками и не зависит от них по смыслу.
  В коммит кладётся только свой ханк, чужой WIP остаётся в рабочем
  дереве: (1) правка вносится в рабочий файл как обычно; (2) та же
  правка — в копию `git show HEAD:ФАЙЛ`, дифф копии против HEAD —
  в индекс через `git apply --cached`; (3) `git commit` без `-a` —
  коммит содержит ровно свой ханк. Гейт — worktree-стенд HEAD+дифф
  (рецепт ниже), в стенд копируется staged-версия файла, не рабочая.
  Пересекаются ханки или смысл — только очередь. Прецедент —
  backlog-строка в занятом README планов (коммит c98de60).
- **Worktree — по потребности.** По умолчанию работа на общем дереве
  по правилам владения; трек уходит в свой worktree (как постоянное
  рабочее место, рецепт ниже), когда параллельно активны ≥ 2 трека
  или их поверхности рискуют пересечься.
- **Один прогон на дерево** (build/ общий внутри каталога). Два
  одновременных make-прогона в одном каталоге затирают друг другу
  `build/` — ложные DIFF (FAULTS 2026-07-16, LEX DIFF на давнем
  файле). Гейт на общем дереве — только когда в нём не работает
  другой агент; при любом подозрении на соседа — worktree-стенд.
  Ограничение «один гейт на машину» СНЯТО 2026-07-17: сверочные
  дампы больше не делят фиксированные `/tmp/eat_*` — пофайловые
  циклы гейта идут воркерами `tests/gate/*.sh` (параллельно,
  `xargs -P $(JOBS)`) с уникальными mktemp-каталогами; гейты из
  разных worktree не сталкиваются. `pgrep -fl make` перед стартом
  остаётся полезной проверкой соседа в том же каталоге.
- **Worktree как рабочее место трека**: `git worktree add /tmp/ТРЕК
  -b трек/ИМЯ`, работа и коммиты — там (worktree разделяет один
  `.git`); перед этапом — `git rebase master`, по завершении —
  fast-forward в master с общего дерева. Гейт — по рецепту ниже
  (override `EATC`); `build/` в worktree свой, первый гейт
  пересобирает с нуля.
- **Красный гейт на общем дереве — сначала атрибуция, потом починка.**
  Три прогона упавшей цели: (а) общее дерево с откаченными во
  временную копию *своими* файлами — падает без тебя? (б) чистый HEAD
  в worktree — падало до всех? (в) HEAD + только свой дифф в
  worktree. Так различаются «сломал я» / «ломает чужой WIP» /
  «сломано в HEAD». Чужую причину не чинить в чужом незакоммиченном
  коде — зафиксировать в FAULTS и сообщить.
- **Полный гейт в worktree**: `git worktree add /tmp/ИМЯ HEAD`, свои
  файлы копируются поверх; `.venv` в worktree нет — цели гонять с
  override `make EATC="PYTHONPATH=src КОРЕНЬ/.venv/bin/python -m eatc"
  ЦЕЛИ`. Зелёный **полный** гейт на HEAD+дифф — достаточное
  определение готовности для коммита своих (дизъюнктных, §3.6)
  файлов, даже когда общее дерево красное чужим WIP.

## 5. Модели и effort

Цель — не жечь максимальную модель на механике. Ступени по
возрастанию цены: Haiku 4.5 → Sonnet 5 → Opus 4.8 → Fable 5;
у каждой — уровни усилия (effort).

| Класс задачи | Примеры из практики | Ступень |
| --- | --- | --- |
| Механическая правка по образцу | подъём пределов пулов, регенерация литералов, переименования | Haiku/Sonnet, низкий effort |
| Прогон и сверка гейтов | make-циклы, диффы дампов, сбор метрик | Haiku/Sonnet, низкий |
| Порт по эталону с известным механизмом | зеркалирование функции verifier.py в Verify.eat по готовому рецепту | Sonnet/Opus, средний |
| Дебаг паритета (поиск механизма) | бисекция расхождения дампов, минимальные репро | Opus/Fable, высокий |
| Архитектура, инварианты, правило 1 | разрыв циклов графа вызовов, дизайн пулов, пересмотр пределов | Fable, высокий |
| Ревью чужого этапа | проверка коммита против плана и гейтов | Sonnet/Opus, средний |

- **Назначение ступени — агент-диспетчер**: дешёвый агент в начале
  сессии классифицирует задачу по матрице и назначает ступень;
  пользователь в промте этапа может переопределить. Назначенная
  ступень фиксируется в промте этапа («→ следующий шаг») — иначе
  следующий агент её не увидит.
- **Задача пересекает классы** (расследование = дебаг высокой
  ступени плюс сбор метрик низкой): ступень этапа — по самому
  дорогому подэтапу;
  дешёвую механику внутри не выделять, если она меньше половины
  объёма, иначе дробить этап (§6).
- **Эскалация — стоп и промт-передача**: агент, упёршийся на своей
  ступени, фиксирует состояние в артефактах репозитория (план,
  FAULTS, минимальные репро) и выдаёт самодостаточный промт этапа
  (§3.7) для ступени выше. Понижение симметрично: оставшуюся
  механику — промтом для дешёвой ступени.

## 6. Дробление этапов под бюджет контекста

- **Правило размера этапа**: этап формулируется так, чтобы его промт,
  нужные файлы и итерации помещались в один сеанс без компрессии
  контекста; ориентир — один механизм / одна поверхность на этап.
- **Артефакты между этапами** — в репозитории (план, FAULTS,
  «→ следующий шаг»), не в памяти сессии; промт этапа
  самодостаточен (§3.7).
- **Длинные прогоны** — в фон, с полезной работой в промежутке;
  дорогие сверки (полный гейт) — один раз в конце этапа, не после
  каждой правки.
- **Если механизм не дробится** (размазан по одному файлу, как машина
  цикла в Verify.eat) — этап получает меньший объём других работ,
  а не режется поперёк механизма.
