## 1. Итог одной строкой

Сделать `selfhost/` читаемым: потолок 100 символов на строку с проверкой
в гейте, конструкторы вместо 7 копий строк-простыней инициализации в
`*Main.eat`, и конструкция `extend` в языке, чтобы распилить
`Verify.eat` (7742 строки после этапа 4 верификатора), `Check.eat`
(6922) и `Ir.eat` (5689) на файлы разумного размера.

## 2. Почему язык к этому приспособлен

Дух языка — явные границы на всё. Потолок длины строки — то же
bounded-мышление, применённое к исходнику: правило читаемости в стиле
Power of 10 («код должен проверяться глазами»). Гигантские файлы —
следствие грамматики `struct X { поля методы }`: все ~250 методов фазы
обязаны жить в одном блоке. `extend` снимает именно это ограничение
записи, не меняя семантики ни на йоту: набор методов тот же, проверки те
же, IR тот же.

## 3. Диагноз (зафиксирован 2026-07-16)

| Симптом | Причина | Лекарство |
| --- | --- | --- |
| строки до 8362 символов в `*Main.eat` | литералы `Lexer{…} → Parser{…} → Check{…} → Verify{…}` в одну строку, 7 дублей | многострочные литералы (грамматика уже допускает — проверено) + функции-конструкторы (этап 2) |
| elif-таблицы 479–1034 символа (`Ast.eat`, `Tok.eat`, `Lexer.eat`) | таблицы «код → имя» в одну строку | построчный формат (этап 1) |
| длинные строки в телах (`Check.eat:597` и т. п.) | глубокая индексация пулов `self.p.nns[a / 65536][a % 65536]` без хелперов | перенос внутри `()`/`[]` (лексер подавляет NEWLINE в скобках — проверено) и точечные хелперы (этап 1) |
| Verify/Check/Ir по 5,7–7,6 тыс. строк | один struct = один файл | `extend` (этап 3) + распил (этап 4) |

Масштаб: строк длиннее 100 **символов** (не байт — кириллица в UTF-8
двухбайтная) — selfhost 217, lib 3, examples 6, tests 1. Итого 227.

## 4. Числа (пределы)

- **Длина строки ≤ 100 символов** (кодпойнтов Unicode, не байт).
  Проверяется скриптом стиля по всем `*.eat` репозитория внутри
  `make check`. Это потолок *репозитория*, не языка; перенос в §6 SPEC
  как языковой предел — отдельное решение пользователя (см. §10).
- **Файл selfhost ≤ 2500 строк** после распила (ориентир, проверяется
  тем же скриптом стиля; сейчас худший — 7742).
- `extend` новых пределов §6 не вводит: действуют существующие потолки
  деклараций/методов; блок `extend` считается продолжением того же
  struct.

## 6. Архитектура / механизм

### 6.1. Скрипт стиля (этап 1)

`tests/check_style.py`: длина строк в кодпойнтах (`len(str)` в Python;
при будущем зеркале в языке — «байты минус продолжения 0x80–0xBF»),
плюс потолок строк на файл для `selfhost/`. Вшивается в `make check`
первым шагом — «определение готовности» в AGENTS.md не расширяется.

### 6.2. Конструкторы фаз (этап 2)

Функции возвращают структуры по значению уже сейчас (`fresh_st() -> St`,
`lib/Json.eat` возвращает `JDoc`). В файл каждой фазы добавляется
свободная функция-конструктор:

```eat
func lexer_new() -> Lexer          # selfhost/Lexer.eat
func parser_new(lex: Lexer) -> Parser
func check_new(p: Parser) -> Check
func ir_new(c: Check) -> Ir
func verify_new(c: Check) -> Verify
```

Семь `*Main.eat` сокращаются до цепочки вызовов; сегодняшние 7 копий
простыней (расползаются при каждом новом поле) исчезают. Контроль
перф-регрессии — `make bench` до/после (возврат больших структур по
значению; следить за defensive-копиями интерпретатора).

### 6.3. Конструкция `extend` (этап 3)

Рекомендуемый синтаксис (D1, решение пользователя):

```eat
extend Verify {
    func join(var self, ...) -> u32
        requires ...
        ensures ...
    { ... }
}
```

- EBNF: `extend_decl = "extend" IDENT "{" { func_decl } "}" ;`
  и `top_decl += extend_decl`. Только методы — **полей в extend нет**.
- Семантика: методы сливаются в именованный struct, как если бы были
  объявлены в его блоке. Struct обязан быть объявлен **раньше** по
  порядку склейки; дубликат имени метода — ошибка с координатами.
  Никаких новых сущностей в typed/sig/ir-дампах: после слияния фазы
  ниже парсера не отличают метод из `extend` от метода из struct.
- Поверхности зеркала: ключевое слово в лексерах (код токена — в конец
  таблицы, кодировки по docs/MODIFYING.md), узел AST (код — в конец),
  parse-дамп, слияние в Check — всё в обоих компиляторах байт-в-байт;
  SPEC (EBNF + §2), docs/GUIDE.md, editor/vscode (грамматика подсветки),
  стресс-кейсы tests/parse.

### 6.4. Распил файлов (этап 4)

Ориентировочные границы — по существующим секционным комментариям
(`# --- … ---`); финализируются при реализации (D2):

- `Check.eat` → `Check.eat` (структура, инфраструктура, пул типов,
  разрешение типов, сбор объявлений, модули, области видимости),
  `CheckConst.eat` (константные выражения, машина вычислителя §9.6,
  ярус A2), `CheckBody.eat` (машина выражений, машина инструкций, match,
  проверка тел, правило 1), `CheckDump.eat` (дампы typed/sig, рёбра).
- `Ir.eat` → `Ir.eat` (буферы, блоки, типы, глобалы, регистры),
  `IrEmit.eat` (эмиттеры, вызовы рантайма, копирование, области),
  `IrExpr.eat` (машина выражений, вызовы, встроенные, lvalue),
  `IrStmt.eat` (машина инструкций, генерация функции, печать модуля).
- `Verify.eat` → `Verify.eat` (структура, пулы путей, интервальное
  ядро), `VerifyExpr.eat` (машины выражений/булева, refine),
  `VerifyFlow.eat` (машина потока, кадры, accel, циклы),
  `VerifyRel.eat` (разностные ограничения, замыкание, дырочный домен),
  `VerifyDump.eat` (обязательства, сортировка, дамп).

Обновляются: списки `SELFHOST_*` в Makefile, карта в
[SELFHOST](SELFHOST.md), чеклисты docs/MODIFYING.md, карта репозитория
в AGENTS.md/README (если перечисляют файлы).

## 7. Что реализуем НЕ

- **Partial struct** (повторный `struct X {…}` со слиянием) — размывает
  «одно объявление — один источник полей», ухудшает диагностику
  дубликатов. Вместо этого явный `extend` без полей.
- **Поля в `extend`** — раскладка структуры обязана читаться в одном
  месте (и от неё зависит копирование по значению).
- **Автоформаттер `eatc fmt`** — вне рамок; потолок держит скрипт стиля,
  формат остаётся ручным.
- **Вынос фаз в DAG-модули import/export** — фазы остаются в склейке
  (этап 0 модулей); граница SPARK-модулей — отдельная тема
  [MODULES_PLAN](MODULES_PLAN.md) §12.
- **Продолжение выражения без скобок** (перенос после бинопа) — правка
  лексера ради стиля не нужна: скобки уже подавляют NEWLINE.

## 10. Открытые вопросы (решения пользователя)

- **D1 — синтаксис `extend`.** Рекомендация: `extend ИМЯ { методы }`
  (§6.3). Альтернативы: powtорный `struct ИМЯ` (отвергнут, §7),
  квалифицированные функции `func ИМЯ.метод(...)` без блока (больше
  повторов, нет группировки).
- **D2 — границы распила** §6.4: подтвердить/поправить состав файлов.
- **D3 — потолок 100 как языковой предел §6** (ошибка лексера для любых
  программ, зеркалится в selfhost-лексер) — сейчас потолок только
  репозиторный (скрипт стиля). Отдельное решение, не блокер.

## 11. Инварианты

- Power of 10 не трогаются: `extend` — чисто синтаксическая группировка,
  семантика тождественна методам внутри struct; ни кучи, ни рекурсии,
  ни новых обходов проверок этот план не открывает.
- Все дампы и IR: Python-эталон == selfhost байт-в-байт на каждом этапе;
  частично зелёное не коммитится.
- Производительность selfhost-компилятора не деградирует: `make bench`
  сверяется с эталоном на каждом этапе (особенно этап 2 — возврат
  структур по значению — и хелперы этапа 1 на горячих путях).
- Раскладка полей каждой структуры остаётся в одном блоке `struct`.
