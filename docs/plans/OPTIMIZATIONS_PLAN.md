### 1.5 NONE-гигиена пулов — открыто второй разведкой 2026-07-17

Вопрос пользователя «сделать NONE системной константой?» вскрыл
устройство класса 358 банк-bounds. NONE уже «системен» для
верификатора (`lib/Const.eat`, максимум u32 = сентинел дырочного
домена); эксперименты на /tmp-копиях входа показали двухчастную
причину недоказуемости:

1. **Инварианты пулов отравлены единичными записями без границ.**
   Минимальная репродукция идиомы «hoist в локаль + `assert x !=
   NONE` → банк доказан» зелёная, а в реальном коде guard не работал:
   `Check.unlink_top(e, prev)` — у `prev` нет `requires prev <
   131072`, и одна запись `nlc[0][0] = prev` растягивает плотную
   часть инварианта пула `(Parser, nlc)` на всю программу; пулы травят
   друг друга дальше копиями линков (`nns → nfc/nlc`).
2. **Чтения без guard'ов** (`nns[nlc[p]/65536][...]` напрямую) — код
   опирается на межполевые инварианты структуры данных («`nfc[p] !=
   NONE` ⇒ `nlc[p] != NONE`»), интервалам недоступные в принципе;
   «системность» константы это не изменит.

Замер (оба патча вместе, /tmp): чистка двух отравляющих записей
(+`assert m/prev < 131072`) + hoist-guard в `append_child` →
банк-bounds доказан; на сайт — −1 runtime-bounds, +1 runtime-assert
(нейтрально по счёту проверок), assert покрывает **все** последующие
чтения линка — чистый выигрыш на кластерах из нескольких чтений
(`insert_field` и т.п.).

**Направление (правки только `.eat`-кода, компилятор не трогается):**
(а) аудит отравляющих записей — `requires` на параметры-линки,
единицы мест; (б) идиома «guard — доказательство» (hoist + `assert !=
NONE`) на кластерах чтений. Потолок класса ~358 bounds. Системная
альтернатива — тип «индекс-или-NONE» с принуждением к narrowing
(правка языка, решение пользователя, большой ценник) — превращает
класс из «недоказуемо» в «непредставимо», рассмотреть отдельным
планом при желании.

**СДЕЛАНО 2026-07-17** (тридцать шестая итерация FINDINGS): на
самоприменении фронтенда runtime-bounds 762 → 539 (−223, банковый
класс 358 → 151), runtime-проверок всего 1260 → 1031 (−18 %), цена —
+14 runtime-assert идиомы; доказанность 79 → 82.6 %. Рецепты (все —
правки `.eat`, зеркал не требуют): чистка отравляющих записей
(assert на неограниченный параметр-линк перед записью, ветвление
`== NONE` вместо копии «дырочного» значения — одна запись hull'а
растягивает плотную часть инварианта пула на весь u32); хелперы
`first_child`/`second_child`/`ty_a`/`ty_b`/`frame_node`/`qp_next`
(assert внутри, наружу плотный результат через ensures); bind-return
вместо `return self.f(...)` в обёртках find-функций (прямой return
вызова теряет hive-дырку summary). Попутно Parser.eat упёрся в
потолок 2500 строк — распил через extend (ParserExpr.eat, машина
выражений), по образцу REFACTOR §6.4. Остаток 539: кадровые/стековые
индексы 233 (класс «nf ≥ 1», data-flow), банковые 151 (цепочка
`tres`/`xa` — поля без сентинела в домене верификатора, research),
прочее 155 (payload-стеки vs/vm, `i±j`). Дальше дешёвых ходов нет —
хвост либо research-домены верификатора (сентинел для полей), либо
системный тип «индекс-или-NONE».

## 2. Кодоген: факты верификатора → LLVM, копии

### 2.1 !range на чтениях пулов — ОТВЕРГНУТО замером 2026-07-14

Реализовано и откачено (двадцать шестая итерация FINDINGS).
Верификатор аннотировал Index-узлы `val_range=(lo,hi,sent)`
(build-only), кодоген эмитил `!range`, дырка → оборачивающийся
интервал; 562 сайта, канон чист, все ворота зелёные, семантически
корректно. Но замер: TrapBench x512 0.77 → 0.83 с (воспроизводимый
регресс ~8 %, не шум), выигрыша нигде. Политика §2.1 была «оставить
только если нейтрально» — не выполнена, откат. Канал «инварианты →
LLVM» уже закрыт `nsw`/`nuw` + `llvm.assume`; `!range` из пуловых
инвариантов на llvmlite=LLVM 22 / Apple Silicon вредит горячему
циклу. Не переоткрывать без нового стека LLVM.

### 2.2 sret прямо в слот назначения (изменение канона — оба эмиттера)

`let x: T = f(...)` сегодня: свежая alloca вызывающего → вызов →
memcpy в слот x (`emit_call` codegen.py:1104, `copy_into` :288).
Передавать слот x как sret-указатель напрямую. **RetBench показал:
при инлайне LLVM уже съедает копию (404M вызовов/с)** — выигрыш
только у неинлайнящихся функций (большие тела). Selfhost: машины
компилятора крупные — проверить профилем, сколько memcpy-возвратов
живёт в SelfIr до начала работ. **Цена высокая**: меняется
канонический IR → зеркало selfhost/Ir.eat байт-в-байт, фикспойнт,
все дампы. Делать только если профиль покажет ≥ единиц процентов.
**Профиль снят 2026-07-17 (§7.3): в горячем пути эмулятора sret-путь
и копии стоят 0 % стены — НЕ делать.**

### 2.3 Хвосты трека 4 (selfhost-код, по остаточному профилю SelfIr 0.13 с)

- `lb_tp`/копии в банкованные пулы: аффинная по-банковая форма,
  которую LLVM распознаёт как memcpy (метрики: CopyBench 220M,
  BankBench 250M, compiler-секция); потенциал ~десятки мс;
- `g_hash` инкрементально при записи в пул вместо пересчёта на
  каждый `g_commit` (Ir.eat);
- `write_span_esc`/экранированные спаны дампов — батчить между
  escape-символами (профиль дампов lex/parse: 1.59–2.45 МБ вывода).

Перед каждым — свежий `sample`-профиль нативного SelfIr: после
модульного этапа и инвариантов расклад мог сместиться.

**Оценка 2026-07-14: ниже пола измерения.** Фазы SelfIr: lex 0.01,
parse 0.02, sig 0.01, typed 0.09, ir 0.14 с — эмиссия IR (ir−typed)
≈ 0.05 с. Хвосты §2.3 — под-части этих 0.05 с, абсолютный выигрыш
единицы мс, а разброс bench ±10–20 мс. SelfIr (0.14 с, ×9 к Python)
слишком быстр для чистого `sample` (пул узлов ≤131072 не даёт
раздуть вход). Оптимизировать вслепую = риск нерегистрируемого
регресса. Трек 4 подтверждённо «низкий приоритет»: браться, только
если появится дешёвый способ надёжного профиля (TCG-плагин под МК
или инструментированный прогон).

## 3. Рантайм-шим и сборка

### 3.1 Аксиома read_span — Python-сторона ПРОВЕРЕНА, selfhost-зеркало отложено (2026-07-14)

Решение пользователя «давай всё» получено. **Python-сторона
реализована и доказана, но откачена** (паритет: «частично зелёное не
коммитится»): `read_span(a: [u8; N], off, len) -> u32` симметрична
write_span, но ПИШЕТ в массив, поэтому `a` обязан быть изменяемым
lvalue (`var`) — новая проверка мутабельности (переиспользует логику
`_check_mutable_receiver`). Сделано и проверено на ветке (не в
HEAD): runtime.c host+mcu (курсор общий с read_byte), codegen.py
(declare + gen_read_span + dispatch), typechecker (_read_span +
_check_mutable_array + сигнатура), interpreter (mutate массива —
`obj[off+i]=b`), verifier (bounds как write_span + **убийство пула
`a`**: read_span пишет произвольные u8, иначе чтение из
заполненного пула получило бы заниженный интервал — несостоятельно),
SPEC §7. **Тест `interp == binary` зелёный**, проверка мутабельности
отвергает `let`-массив.

**Блокер — selfhost-зеркало** (Check.eat + Ir.eat), интрикатное как
u64/extern: ДВЕ раздельные системы кодов (Check `xa` write_span=110,
Ir emit-код write_span=8) + typed-дамп, всё байт-в-байт, плюс
`xrspan` (bounds+gep как `xwspan` :3143, но результат i32 через
`rpre`/`e_rspan`, не void `vpre`/`e_wspan`), новый код в обеих
системах, ert/result-конвенция значащего вызова. Оценка — отдельный
фокус-этап (как каждый axiom-коммит в истории); риск — bootstrap
фикспойнт. Python-дизайн де-рискован, зеркало механическое.

**По выигрышу оговорка:** после слоя 0 read_byte уже 466M байт/с;
read_span экономит только цену вызова-аксиомы на байт, а обработку
(классификацию байта лексером) — нет. Реальный выигрыш на лексере
skromный; замерить ReadBench-вариантом до адаптации лексера.

### 3.3 Порог батчинга write_span — СДЕЛАНО 2026-07-15 (32 → 16)

Гибрид «< 32 байт побайтно, иначе fwrite» калибровался при -O0-шиме.
Спан-микробенч после -O2 (тридцатая итерация FINDINGS): кроссовер ~12
байт (fixed-cost fwrite ~22 нс/вызов против ~2 нс/байт), к 16 байтам
fwrite ×1.4. Порог опущен к измеренному оптимуму (32 → 16, риск-free —
C-шим, паритет не трогается). **На продакшн-workload'ах ниже пола**:
SelfIr-эмиссия/parse-дамп/typed-дамп compute-bound, спаны окна 12–31
тонут в вычислении; IoBench нечувствителен. Корректно для
write-доминированных путей, нейтрально на текущих. Закрыто.

недетерминирован внутри цели.

Уроки: (1) `VAR=x cmd` из переменной POSIX-shell не является
присваиванием — воркеры зовут `env $EATC …`; (2) негативные кейсы
эталона (`eatc verify` с ненулевым кодом) в исходных циклах
маскировались `;` — в воркерах с `set -e` это явные `|| true`.

### 3.4 Стоимость верификатора в build — O(n²) устранён 2026-07-15

cProfile verify() на SelfIr: `_func_by_key` — линейный скан по
`program.decls` на каждый межпроцедурный вызов (0.206 с self /
24642 вызова). Заменён индексом (dict, первое совпадение), verify()
351 → 314 мс (−10 %), доказательства идентичны. Побочный вывод:
2-проход тел циклов (§1.1) **не** бутылочное горло — гейт не нужен.
Остаток self-time — `_iv` (интервал-эвалюатор, ядро) и `_path_of`
(у пола) — не дешёвые сканы. Тайпчекер профилирован отдельно: плоский,
супер-линейности нет. Дальше по верификатору дёшево не выжать.

## 7. Кросс-языковой замер 2026-07-17: внешние рычаги

Источник — контур `make bench_crosslang` (CROSSLANG_BENCH_PLAN ✅,
отчёт tests/bench/CROSSLANG.md): EATLang против C/Rust/Go/Python на
шести микробенчах и эмуляторе mos6502, сверки stdout байт-в-байт,
счётчики instructions/cycles. Итог фона: геосреднее EAT/C = 0,79,
паритет 1,02 без выбросов в пользу EAT. Рычаги, извлечённые из цифр:

### 7.1 llvm.assume в оси -O — СДЕЛАНО 2026-07-17: снят как инертный, гипотеза вреда ОПРОВЕРГНУТА

Вариант A доставлен: `llvm.assume` из доказанного `requires` (слой 1а,
23-я итерация FINDINGS) не эмитится вовсе — элизия и `nsw`/`nuw`
остаются; правка зеркальная (codegen.py + IrStmt/IrEmit, минус
`assume_safe`/`assume_leaf`/`e_assume`, −130 строк зеркал), паритет
`eatc ir -O` == SelfIrOpt байт-в-байт держится, полный гейт зелёный.

**Гипотеза «assume съедает выигрыш элизии» опровергнута числами**:

- branch: в оси `-O` assume **0 вызовов** (нечему вредить — IR оси
  совпадает с build с точностью до trap-строк); разрыв 1,46–1,52
  против 1,09–1,28 у build — это **конвейер сборки** (llvmlite у
  build против clang -O2 по .ll), передан в §7.2;
- mos6502: снятие всех 15 assume не сдвинуло инстр/шаг
  (56,32 → 56,29–56,30), стена 4,00 → 3,89–3,90 нс/шаг при дрейфе
  канона −2…−2,7 % (шум фона); разрыв оси `-O` против канона
  (56,3 против 54,6 инстр/шаг на одном clang) — цена самой элизии
  (кроссовер аутлайн/инлайн, уже разложено в §7.3, п. 1);
- sort/u64: без изменений (0,404/4,73–4,83) в коридоре шума.

Ожидания −5…−10 % не было и быть не могло — согласуется с §7.3
(«assume ≈ 0, проверено вырезанием» на llvmlite-пути; теперь
подтверждено и на clang-пути). Assume оставлен снятым: инертный
механизм в parity-критичных зеркалах — чистый пассив; канал «факты,
недоступные LLVM» при нужде вернётся отдельным решением.

Попутные находки (детали — FAULTS, FINDINGS 38-я итерация):
посылка «build = элизия без assume» из отчёта CROSSLANG была
**неверна** (build эмитил те же assume — общий кодоген); гейт
`verify_selfhost_opt` не покрывает бенч-конкатенацию mos6502 — под
ней вскрыт предсуществующий дрейф одного решения верификаторов
(`hex_digit` requires: Python proven, selfhost runtime; 330 против
331 из 491) — кандидат-этап правки верификатора.

### 7.2 Конвейер сборки — СДЕЛАНО 2026-07-17: llvmlite ≥ clang, маршрут build остаётся

Чистый эксперимент проведён (`tests/bench/crosslang/backends.py`):
один и тот же байт-идентичный .ll — неоптимизированный, с элизией,
без assume, который `eatc build` пишет рядом с бинарником — прогнан
через оба бэкенда; методика run.py (медиана 5 прогонов), два полных
прохода подряд, дрейф между проходами ≤ 4 %. Сверки stdout всех
вариантов байт-в-байт зелёные.

**Вердикт: clang-путь НЕ стабильно ≥ — наоборот, штатный
llvmlite-путь быстрее или равен на всех 5 нагрузках.** Смена
маршрута build не предлагается, вопрос снят.

| нагрузка | build (llvmlite), нс/оп | .ll → clang -O2 | clang к build |
| --- | --- | --- | --- |
| arith | 1,154–1,156 | 1,150–1,183 | −0,3…+2,3 % (паритет) |
| sort | 0,381–0,384 | 0,396–0,398 | +3,1…+4,5 % |
| branch | 1,096–1,132 | 1,440–1,447 | **+28…+31 %** |
| u64 | 4,426–4,509 | 4,428–4,601 | −1,8…+4,0 % (паритет) |
| mos6502 (нс/шаг) | 3,632–3,665 | 3,846–3,878 | +4,9…+6,8 % |

Разложение (гибриды мид-энд × кодоген на том же .ll): на **branch
виноват кодоген clang** (clang-мид → llvmlite-кодоген быстр как
build: 1,08–1,13; llvmlite-мид → clang медлен как clang: 1,43–1,46;
инстр/оп 14,47 против 15,48, IPC 3,84 против 3,09 — llvmlite-бэкенд
строит csel-цепь плотнее); на **mos6502 виноват мид-энд clang**
(llvmlite-мид → clang быстр: 3,69; clang-мид → llvmlite медлен:
3,80–3,92). Причина обоих направлений одна: llvmlite 0.48 несёт
LLVM 22.1.0 — **новее** Apple clang 21.0.0; «родной clang» на деле
старее штатного конвейера. `-flto` (путь `--release`) по скорости
== clang -O2 везде (на mos6502 +5,8…+7,2 % к build), выигрыш только
в размере: 33К против 50К на микробенчах — release остаётся осью
размера, не скорости.

Иллюзия исходного замера «clang не хуже на 5/7» разобрана: там
сравнивались РАЗНЫЕ IR (clang на каноне против llvmlite на элизии),
а канон эмулятору сам по себе быстрее элизии (§7.3, п. 1) — на
байт-идентичном .ll преимущество clang исчезает. Урок и версионный
перекос текстового .ll (LLVM 22 ↔ clang 21) — FAULTS 7.2;
числа и методика — FINDINGS, 39-я итерация.

### 7.3 Эмулятор: инструкций меньше C, стена хуже — ВЕРДИКТ 2026-07-17

Расследование проведено (тридцать седьмая итерация FINDINGS; xctrace
недоступен — на машине только CommandLineTools, без root; метод —
попарные изоляции: один вход 936,9 M шагов, счётчики
`/usr/bin/time -l`, варианты IR в /tmp, сверки stdout байт-в-байт).

**Подозреваемые §2.2 оправданы полностью — доля стены 0 %.** В
горячем пути эмулятора sret/memcpy и «копий var self» нет вообще:
`var self` в бинарнике — указатель; memcpy в модуле 8 штук, все —
холодные строковые хелперы Rt и разовый `cpu_new` на старте.
**Рекомендация: §2.2 НЕ делать** — канон IR не менять, выигрыша нет
ни на эмуляторе, ни (по RetBench) на инлайнящихся вызовах.

Реальное разложение разрыва build → C (12,35 → 10,97 такта/шаг,
+12,6 % стены), цепочкой изоляций на llvmlite-конвейере build:

1. **Элизия оси -O вредит эмулятору**: канон (все-trap) через тот же
   llvmlite — 11,59 такта/шаг, т.е. элизия стоит **0,76 такта/шаг**
   (на clang-пути 1,17). Механизм — кроссовер 2×2 (аутлайн/инлайн ×
   канон/элизия): при аутлайновых хелперах канон быстрее (trap-icmp
   в вызывающем восстанавливает диапазон результата через границу
   вызова — `ensures result <= 65535` LLVM не видит), при полном
   инлайне быстрее уже элизия (маски видны насквозь). assume ≈ 0
   (16 вызовов на модуль), nsw/nuw = 0 — проверено вырезанием.
2. **Аутлайн 4 горячих хелперов** (`ea_mem`/`ea_alu`/`shift_val`/
   `flags_byte` — llvmlite и clang их не инлайнят): **0,36 такта/шаг**
   (11,59 → 11,23 канон+alwaysinline через llvmlite).
3. **Остаток формы кода** против C: 0,26 такта/шаг (плотность цепочки
   зависимостей шага: load pc → load mem[pc] → диспетчер → операнд →
   флаги; C несёт больше независимых инструкций — IPC 5,31).

**Находка-рычаг (лучший вариант EAT, быстрее C)**: штатный build-IR
(элизия) + `alwaysinline` четырём хелперам post-parse через
llvmlite-API (`ValueRef.add_function_attribute`) — **10,61 такта/шаг,
49,2 инстр/шаг, 2,92 с — быстрее C -O2 на 3,3 % тактов (−7 % стены)**;
канонический текст .ll не меняется (пишется до оптимизации), зеркало
не требуется. Слепое поднятие `inlining_threshold` вредит (13,87 на
build-IR, 17,38 на каноне — раздувает всё). Кандидат-этап: эвристика
инлайн-хинтов малых листовых функций в build-конвейере (Python-only,
решение пользователя).

### 7.4 Строки: лидерство ×1,8–2,6, но 492 инстр/оп — подтверждён §лib

strcmp: EAT 25,7 нс/оп против C snprintf 67, Rust format! 47,
Go strconv 43 — специализированный Rt.append_u32 бьёт идиомы
соперников. Внутри всё ещё O(cap)-идиомы str<256> (Rt.eq скан,
append) — рычаги lib/NumParse·StrCmp из шапки реестра получили
внешнюю планку.

### 7.5 Отрицательные результаты (фикс)

Маска-как-доказательство работает: bank/u64/arith — паритет с C
(Rust unchecked ≈ safe: LLVM сам снимает маскированные границы).
Элизия окупается на ветвистом коде: branch build быстрее канона на
41 % — вес research-веток NONE-домена (§1.4/1.5) подрос: на
dependency-chain коде недоказанный trap стоит дороже всего.
Скорость компиляции: чистая эмиссия SelfIrOpt — 10 мс микробенч /
20–30 мс эмулятор (~25× Python-эталона); цепочка ограничена clang.
