паритет `бинарник == интерпретатор`».

## 1. Итог одной строкой

Вытесняющая многопоточность — **нет** (гонки данных, недетерминизм,
неограниченный стек несовместимы с §8 и правилами 1/9). Кооперативная
асинхронность — **да**, и ярус 0 почти не требует правок языка.
Каноничный ответ мира Power of 10 — не `async/await`, а **циклический
исполнитель** (cyclic executive): именно так работает лётный софт JPL,
и именно под него в языке уже зарезервирован единственный `loop` в
`main`.

## 2. Почему EATLang к этому приспособлен

- **Слот исполнителя уже есть.** Единственный `loop` в `main`
  ([SPEC.md §209](../../SPEC.md)) введён как «модель реактивной
  системы». Планировщик — это и есть содержимое loop; ничего нового
  в грамматику вводить не нужно.
- **Тотальность решает главную боль кооперативной модели.** В обычных
  языках задача может зажать планировщик навсегда. Здесь каждый шаг
  задачи — обычная функция, а значит доказуемо завершается. Язык может
  дать то, чего почти никто не даёт: **доказанную границу времени
  отклика** одного витка loop (конечная композиция тотальных функций +
  оценка в духе §8).
- **Кучи не нужно.** Кадр задачи (что живёт через точку ожидания) —
  обычный `struct` статического размера. Это модель Rust Embassy,
  только получается по построению, без аллокатора.
- **Идиома обкатана.** Selfhost-парсер — автомат с явным стеком кадров
  ([POWER_OF_10.md §54-57](../POWER_OF_10.md)). Задача-автомат с методом
  `step()` — тот же стиль, тот же скилл.
- **Верификатор уже умеет инварианты полей struct** (трек 3,
  [TRACKS.md](../TRACKS.md)). Это ровно то, что нужно, чтобы факты
  переживали точку ожидания: локальные факты умирают на yield,
  инварианты кадра — нет.

## 3. Что мешает и требует решения пользователя

1. **Блокирующие аксиомы.** `read_byte` блокирует — с ним асинхронность
   бессмысленна. Нужны неблокирующие аксиомы (решение по SPEC §7, по
   образцу argv, трек 6): poll-семантика ввода + монотонное время.
   Эскиз — §6.
2. **Детерминизм и `make verify`.** Паритет «бинарник == интерпретатор
   байт-в-байт» — краеугольный камень. Следствия жёсткие:
   - планировщик строго детерминирован (фиксированный round-robin,
     никакой зависимости от реального времени);
   - вся недетерминированность входит только через аксиомы — как
     сегодня stdin;
   - для `ticks()` в режиме сверки нужны **виртуальные часы**
     (детерминированный тик на виток), по образцу `INPUT=`-прошивки
     для МК (`embed_input.py`).
3. **Формулировка тотальности.** Теорема «каждая программа завершается,
   кроме loop в main» переходит в стандартную для реактивных систем
   форму «каждая *активация* (виток loop, шаг задачи) завершается».
   По сути менять нечего — только явно зафиксировать в SPEC §2.
4. **Нет указателей на функции (правило 9).** Таблица задач не может
   быть динамической: диспетчеризация — только `match` по статически
   известным задачам. Это не ограничение, а гарантия: граф вызовов
   остаётся статическим DAG, §8 продолжает считать стек точно.

## 4. Три яруса реализации (по нарастанию цены)

### Ярус 0 — циклический исполнитель (правок языка нет)

Superloop в `main` по кругу вызывает `step()` каждой задачи; состояние
задачи — `struct` с `var self`; «ожидание» — ранний `return` из `step`.
Это канонический ответ мира Power of 10.

- **Цена:** только неблокирующие аксиомы (§6): шим `runtime.c` +
  `mcu/common/runtime.c`, `declare` в обоих кодогенах, тайпчекер,
  интерпретатор, SPEC §7, полный паритет и фикспойнт. По объёму — как
  трек argv.
- **Синергия с треком 2.** Идеально стыкуется с уже запланированным
  «периферийным шимом (gpio/systick/uart-кольцо)»: прерывания живут в
  C-шиме и пишут в кольцевой буфер, язык видит их через аксиомы. Так в
  язык **не** протекают volatile/атомики/гонки — самая дорогая часть
  асинхронности отсекается на границе доверия, ровно как у `extern`.

### Ярус 1 — библиотечное оформление (`lib/`, без правок компилятора)

Зафиксировать паттерн словарём и кирпичами:

- enum `Poll { Ready(T), Pending }` (или `Ready`/`Pending` без данных
  для fire-and-forget задач);
- конвенция `step(var self, ...) -> Poll`;
- хелперы: таймеры/дедлайны поверх `ticks()`, кольцевые буферы,
  дебаунс — в `lib/` (кандидат: `lib/Async.eat` или `lib/Sched.eat`).

**Честная оговорка:** сам планировщик обобщённой библиотекой быть не
может (нет указателей на функции) — `match` по конкретным задачам
остаётся в пользовательском `main`. Библиотека даёт словарь и кирпичи,
не executor. Это следствие правила 9, не недоработка.

### Ярус 2 — сахар `task`/`yield` в языке

Компилятор сам режет тело задачи по точкам ожидания: локали, живущие
через `yield`, становятся полями кадра-`struct`, тело — `match` по
номеру состояния. Технически это чистая source-to-source трансформация
в уже разрешённое подмножество — ничего из Power of 10 не открывает.

- **Цена — самая инвазивная категория** (как отвергнутый `str<N>`,
  [TRACKS.md трек 1](../TRACKS.md)): оба кодогена, тайпчекер, семантика
  контрактов на срезах (что значит `ensures` для одного `step`?),
  верификатор, SPEC, зеркало байт-в-байт, фикспойнт.
- **Открытый вопрос контрактов:** у резаной на состояния функции
  требуется определить, где живут `requires`/`ensures` — на входе в
  задачу, на каждом `step`, или инвариант кадра между шагами. Скорее
  всего последнее (инвариант кадра = связь полей struct), что смыкается
  с реляционным доменом верификатора (трек 3, OPTIMIZATIONS_PLAN §1.2).

## 5. Что реализуем НЕ (явные отказы)

- **Потоки / вытеснение / SMP** — гонки данных и недетерминизм
  несовместимы с паритетом и правилом 9. Параллелизм железа (второе
  ядро, DMA) живёт за границей доверия в C-шиме, язык видит результат
  через аксиомы.
- **Динамический пул задач** — нет указателей на функции; число и типы
  задач известны статически (это и держит §8 точным).
- **Реальное время внутри языка** — время входит только аксиомой
  `ticks()`; в режиме сверки — виртуальное.

## 6. Аксиомы (формы решены пользователем 2026-07-16, §9)

По образцу трека 6 (argv). Встроенные enum не меняются.

| Сигнатура | Смысл | Детерминизм в сверке |
| --- | --- | --- |
| `in_avail() -> u32` | сколько байт stdin можно прочитать `read_byte` **без блокировки**; `0` — ждать нечего | stdin-файл: `размер − логическая позиция` (ftell поверх stdio-буфера) — одинаково в шиме и интерпретаторе; пайп/tty — живой режим (FIONREAD), в сверке не участвует |
| `ticks() -> u64` | монотонное время, мс от старта процесса | режим виртуальных часов: счётчик `+1 на вызов ticks()` — обе стороны считают вызовы, знание о loop не нужно |
| ~~`poll_byte()`~~ | **отвергнуто** — тянуло новый встроенный enum либо `WouldBlock` во все match по `IoError` | — |
| ~~`yield_hint()`~~ | **отложено** до первого МК-примера с потребителем WFI | — |

Тонкости:

- у `in_avail`/`ticks` нет trap-границ (аргументов нет, результат —
  полный диапазон типа); верификатор считает результат неизвестным
  `u32`/`u64`;
- переключатель виртуальных часов — переменная окружения (одна и та
  же для бинарника и интерпретатора), по образцу «недетерминизм
  входит только через аксиомы»: сверочные цели Makefile выставляют
  её обеим сторонам;
- TOCTOU-гонки `in_avail`+`read_byte` нет: программы EATLang
  однопоточны по построению, между вызовами stdin никто не крадёт.
