## Итог одной строкой

Честно замерить, сколько стоит EATLang рядом с C, Rust, Go и Python:
скорость исполнения, эффективность кода для процессора (инструкции,
IPC), скорость компиляции, размер бинарника, память — и отдельно
«налог на безопасность» (проверенные trap'ы против сырого кода).

## Почему язык к этому приспособлен

Это не фича языка, а испытание его центрального тезиса: bounded-модель
плюс верификатор дают безопасность **бесплатно** там, где границы
доказаны (`-O` элидирует trap'ы). Сравнение с C (потолок скорости без
проверок), Rust (проверки границ в safe-коде), Go (проверки + GC) и
Python (интерпретация) впервые ставит этот тезис в внешний контекст.

## Соперники и оси проверок (решение пользователя 2026-07-17)

| Язык | Компилятор | Базовый режим | Ось «цена проверок» |
| --- | --- | --- | --- |
| EATLang | `eatc build` (LLVM O2) | канон (все trap'ы) | `-O` (fold + элизия доказанных) |
| C | Apple clang 21, `-O2` | без проверок | `-fsanitize=undefined,bounds` (UBSan) |
| Rust | rustc stable, `--release` | safe (проверки границ) | `unsafe get_unchecked` / `wrapping_*` |
| Go | go 1.24 | проверки границ + GC | `-gcflags=-B` (снятие bounds-check) |
| Python | CPython 3.11 | интерпретация | — (одна точка) |

Все бинарники и замеры — на одной машине (Apple Silicon, darwin),
в одной сессии.

## Метрики

1. **Скорость исполнения** — время утяжелённого прогона (REPEAT-
   масштабирование по методике `tests/bench/bench.py`), нормировка
   в нс/оп; медиана из ≥ 5 прогонов, шум фиксируется.
2. **Эффективность кода для процессора** — `/usr/bin/time -l`:
   instructions retired, cycles elapsed → **IPC** и **инструкций на
   операцию**. Это прямой ответ «какой код плотнее»: меньше инструкций
   на ту же работу = эффективнее кодоген, независимо от частоты.
3. **Скорость компиляции** — время сборки нагрузки от исходника до
   бинарника (eatc и selfhost-цепочка против clang/rustc/go build);
   у Python — нет.
4. **Размер бинарника** — после `strip`.
5. **Пиковая память (RSS)** — peak memory footprint исполнения.
6. **Налог на безопасность** — дельта база ↔ ось проверок внутри
   каждого языка (таблица выше).

## Нагрузки

**Микро** (порты `tests/bench/programs/*.eat`, шесть штук — по классу
конструкций на каждый):

| Бенч | Что меряет |
| --- | --- |
| ArithBench | целочисленная арифметика в горячем цикле |
| SortBench | data-dependent ветвления и свопы (сортировка вставками) |
| BranchBench | 4-плечевой диспетчер по битам (профиль эмулятора) |
| BankBench | банкованная память `p[a/4096][a%4096]`, зависимый адрес |
| StrCmpBench | сборка строки + сравнение с общим префиксом |
| U64Bench | 64-битная арифметика, деление, широкие сдвиги |

**Макро** — эмулятор mos6502 (`examples/mos6502/`, ~900 строк): порт
на все четыре языка, вход — тяжёлый ROM (миллионы шагов), чтобы
исполнение доминировало над запуском процесса.

## Методика честности

- **Дифференциальная сверка между языками**: каждый порт обязан выдать
  тот же checksum-вывод байт-в-байт, что EATLang-оригинал (и native, и
  интерпретатор уже сверены между собой). Расхождение = ошибка порта,
  замер не засчитывается.
- **Одинаковая структура кода**: порт повторяет алгоритм и границы
  оригинала 1:1 (те же типы, та же банковка, те же свёртки), без
  идиоматических ускорений — меряем компиляторы, не программистов.
- **Контроль вырождения**: контрольные суммы уже спроектированы против
  выкидывания работы (несимметричная свёртка, 127 проходов — см.
  `tests/bench/README.md`); после сборки — спот-проверка, что горячий
  цикл не свёрнут в константу (сравнение времени с ожидаемым объёмом
  работы; при подозрении — дизассемблирование).
- **REPEAT одинаков внутри языка-класса**: у компилируемых — один
  множитель (цель 1–5 с), у Python — меньший с нормировкой на оп.
- **Замер запуска отдельно**: цена старта процесса вычитается
  (прогон с REPEAT=0 или минимальным).

## Что реализуем НЕ

- **Не** идиоматические переписывания (векторизация вручную, SIMD,
  numpy у Python) — сравнение потеряло бы предмет.
- **Не** многопоточность — EATLang однопоточен по построению.
- **Не** бенчи с кучей/рекурсией — их нельзя выразить на EATLang;
  сравниваем только выразимое (это ограничение честно фиксируется
  в отчёте).
- **Не** микро-оптимизация EATLang по итогам — это отдельные планы
  (реестр [OPTIMIZATIONS_PLAN](OPTIMIZATIONS_PLAN.md)).
- **Не** CI-гейт: `bench_crosslang` — ручная цель, в полный гейт
  не входит (зависит от чужих тулчейнов на машине).

## Паритет

Зеркало selfhost не требуется: компилятор и язык не меняются. EATLang-
бинарники собираются штатным `eatc build` (+ `-O` для оси проверок);
порты на другие языки живут в `tests/bench/crosslang/` и на паритет
компиляторов не влияют.

## Открытые вопросы

- Тяжёлый ROM для mos6502: сгенерировать программой (цикл 16-битных
  счётчиков с записью в $0200) — решается на этапе 3 без правок языка.

## Инварианты

План не открывает обходов Power of 10: код EATLang не меняется вовсе;
порты на другие языки — вне языка и вне его гарантий (это и есть
предмет сравнения). Никаких правок `src/eatc/`, `selfhost/`, `lib/`.

## Затронутые поверхности

- `Makefile` — цель `bench_crosslang` (комментарий: ручная, не гейт);
- `tests/bench/README.md` — раздел о кросс-языковой секции;
- `tests/bench/CROSSLANG.md` — отчёт с результатами;
