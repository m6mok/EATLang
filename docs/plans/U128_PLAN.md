# План: 128-битная арифметика (u128/i128)

- **Статус:** ✅ ВЫПОЛНЕН (2026-07-17, этапы 0–3 целиком; этап 4
  intrinsics — только по замеру, отдельное решение пользователя)
- **Роль:** справка (дизайн-контракт `lib/U128.eat`)
- **Рабочие поверхности:** —
- **Правки языка/грамматики:** вариант A — **нет**; варианты B/C — да
  (решение пользователя обязательно)
- **Новые аксиомы ОС (SPEC §7):** нет
- **Зеркало в selfhost:** вариант A — не требуется (чистый
  `lib/U128.eat` на EATLang, паритет через `verify_selfhost` даром);
  B/C — требуется байт-в-байт
- **Опирается на:** [MODULES_PLAN](MODULES_PLAN.md) ✅ (модуль в
  `lib/`), [FIXED_POINT_PLAN](FIXED_POINT_PLAN.md) ✅ (прецедент
  «модуль без правок компилятора» и первый клиент — Q32.32)
- **Блокеры:** нет (решения §9 приняты 2026-07-17, см. таблицу)
- **Блокирует:** — (Q32.32 доставлен в `lib/Fixed.eat`; будущие
  крипто/PRNG-выносы — poly1305, pcg64 — теперь могут опираться)

## 1. Итог одной строкой

128 бит достижимы **без единой правки языка**: `lib/U128.eat` на
32-битных лимбах в `u64` («маска — это доказательство», идиома
mos6502); нативные типы `u128`/`i128` (вариант C) откладываются до
доказанной боли — тем более что self-hosted верификатору для их
интервалов самому нужен программный 128-бит, то есть вариант A —
пререквизит C в любом случае.

## 2. Зачем (клиенты)

- **Q32.32** — [FIXED_POINT_PLAN §4](FIXED_POINT_PLAN.md) отклонил
  формат честно: `mul` 1.0 × 1.0 — произведение raw = 2^64 > i64max,
  «нужен 128-бит промежуток, его в языке нет». `U128` снимает отказ,
  не трогая язык.
- **Deci6** (тоже отложен в Fixed): произведение raw до ~2^126 —
  влезает в 128 бит, граница mul/div формата исчезает.
- **Крипто/PRNG**: лимбовые произведения 64×64→128 (poly1305,
  curve25519, pcg64/xoshiro) — сегодняшний
  [CRYPTO_SUITE_PLAN](CRYPTO_SUITE_PLAN.md) нарочно живёт в mod-2^32,
  U128 открывает следующий ярус.
- **Checked-помощники**: точное умножение `u64` с проверяемой
  гарантией «без потерь» (`hi == 0`).

## 3. Три варианта и их цены

| Вариант | Что это | Цена | Вердикт |
| --- | --- | --- | --- |
| **A: `lib/U128.eat`** | программный 128-бит на лимбах | правок компилятора **ноль**, паритет даром; скорость ~5–15× медленнее нативного | **v1 — начать здесь** |
| B: intrinsics | `mul_wide`/`umulh` и т. п. как встроенные | правка языка + зеркало (lexer/typechecker/interp/codegen/verifier); согласуется с [GENERICS_PLAN §7.2](GENERICS_PLAN.md) «числовые intrinsics» | только по **замеру** горячей точки, отдельное решение |
| C: типы `u128`/`i128` | полноценные ширины в языке | литералы до 39 цифр, comptime, Rt-печать, runtime, МК (`__udivti3` из compiler-rt), SPEC §6, editor; **интервалы верификатора**: selfhost считает их в языке, где шире `u64` ничего нет ⇒ ему самому нужен программный U128 (= вариант A) | **отклонить сейчас**; вернуться после A и только по доказанной боли |

Ключевой аргумент против C сейчас — замкнутый круг self-hosting'а:
интервальный анализ 128-битных типов в `selfhost/Verify*.eat` требует
128-битной арифметики в компиляторе, написанном на языке с потолком
`u64`. Вариант A разрывает круг и в одиночку закрывает всех известных
клиентов §2.

## 4. Представление и инварианты (вариант A)

```text
struct U128 { w0: u64, w1: u64, w2: u64, w3: u64 }
# лимбы по 32 бита (w0 — младший), инвариант: w_i <= 0xFFFFFFFF
```

- Лимб — 32 бита **в `u64`-слоте**: headroom для переносов и
  произведений; wrap всегда записан явно маской (`& 0xFFFFFFFF`,
  `>> 32`) — идиома
  [mos6502](../../examples/mos6502/Cpu6502.eat): «язык трапит
  переполнение, поэтому wrap-around записан явно».
- **Доказательство MAC** (несущая лемма модуля): шаг умножения
  `a*b + acc + carry` при a, b, acc, carry ≤ 2^32−1 даёт максимум
  (2^32−1)² + 2·(2^32−1) = **2^64−1** — в `u64` влезает впритык,
  trap невозможен. Зафиксировать в шапке модуля и тестом на краях.
- Инвариант нормализованности лимбов — в `requires`/`ensures` каждой
  операции (`w_i <= 4294967295`): верификатор снимает проверки масок,
  как в пулах §1.5 OPTIMIZATIONS.
- Массив `[u64; 4]` вместо четырёх полей — вопрос §9.5 (поля читаются
  в контрактах прямее, массив короче в циклах).

## 5. API v1 (черновик, уточняется решениями §9)

- **Конструкторы**: `u128(lo: u64)`, `u128_hi_lo(hi: u64, lo: u64)`.
- **Арифметика**: `add`/`sub` (перенос/заём явные; поведение на
  переполнении 128 бит — §9.2), `mul_64(a: u64, b: u64) -> U128`
  (точное, главный кирпич), `mul` (128×128, §9.2), `shl`/`shr`
  (`requires k <= 127`).
- **Сравнения**: `eq`/`lt`/`le`, `is_zero`.
- **Деление**: состав — §9.3 (полное 128/128 — сдвиговое, ровно 128
  итераций с константной границей; либо только `divmod` на малый
  делитель — его хватает для печати).
- **Печать**: `repr_hex` (по лимбам, тривиально) и `repr_dec`
  (через divmod 10).
- **Знаковый `I128`** — знак + магнитуда поверх `U128` (нужен
  Q32.32) — §9.4.

## 6. Что реализуем НЕ

- **Нативные `u128`/`i128` (вариант C)** — см. §3, отдельное решение
  пользователя после A.
- **256-бит и произвольная точность** — клиента нет; DAG `lib/`
  позволит später построить U256 поверх U128 тем же приёмом.
- **Float** — как и всюду в языке.

## 8. Power of 10 — как соблюдается

- нет кучи: `U128` — struct фиксированного размера на стеке;
- нет рекурсии: методы не зовут себя, граф — DAG;
- все циклы с константной границей: 4 лимба, деление — ровно 128
  итераций, `break` ≤ 1;
- результат не отбрасывается, контракты у каждой функции явные;
- вариант A не трогает компилятор вовсе — обходов не открывается
  по построению.
