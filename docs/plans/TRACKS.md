гибрид с порогом 32 байта.

## Трек 1. Строковая модель `str<256>` (слой 4, решение по SPEC)

**Сделано 2026-07-13**: направление «read-only по ссылке» реализовано
по плану [STR_MODEL_PLAN.md](STR_MODEL_PLAN.md) (решения пользователя
в §6 там же): defensive copy у вызываемого снята, перекрытие с
`let self`-получателем закрывает caller-копия, атрибуты
`noalias nocapture nofree nonnull [readonly]`; SPEC §5.4, пример
`examples/alias/`, уроки — десятая итерация FINDINGS.

**Итог `str<N>` — ОТВЕРГНУТО исследованием 2026-07-15** (тридцать первая
итерация FINDINGS). Разбор текущего кода показал: все три выигрыша,
которыми обосновывалась параметризация ёмкости, уже сняты прошлыми
оптимизациями, и параметризация не даёт измеримого эффекта:

- **«копии трогают 260 байт»** — неверно: копия str сайзится по
  фактической длине `4 + ln` ([codegen.py:344-351](../src/eatc/codegen.py#L344-L351));
- **«кадры раздуты str-полями»** — неверно: в struct'ах компилятора
  ноль полей `str<N>`, строковые поля — `[u8; N]`-массивы
  ([Lexer.eat:100](../selfhost/Lexer.eat#L100), Ir.tp `[u8; 65536]`);
- **str-параметры** биндятся по указателю без alloca/копии в callee
  ([codegen.py:538-539](../src/eatc/codegen.py#L538-L539)), а почти
  все `str<N>` в коде — параметры (`kw: str<16>`, `s: str<64>`, …);
  в отчёте кадров §8 str-тяжёлые функции не появляются.

Остаётся ровно 3 слот-локали с мелким str (`io`/`pe`/`pre`) — суб-
килобайт кадра при стеке 256 МБ, ниже любого пола. Против —
самая инвазивная правка репо: новая семантика ёмкости (сейчас
`compatible()` молча разрешает str<256>→str<8>, рантайм-проверки нет),
ABI RtStr (struct `{ln,[256×i8]}`, все методы хардкодят 256), зеркало
Check.eat+Ir.eat байт-в-байт, верификатор, интерпретатор, SPEC,
фикспойнт. Как `§2.1 !range` — не окупается, не реализуем. Трек 1
закрыт целиком (все три направления: by-reference сделано, len-копия
в коде, `str<N>` отвергнуто).

**Проблема (исходная).** Строка — фиксированные 260 байт (`{i32, [256 x i8]}`),
передаётся по значению: каждое связывание — memcpy 260 байт, даже
для трёхбайтового имени. Это доминирующая цена строкового кода
(StrBench ~88 M оп/с против 820 M у арифметики; получатель без
`let self` копирует агрегат при каждом вызове — `Rt.write(self)`
копирует 260 байт, чтобы вывести 5).

**Направления дизайна** (обсудить до кода, у каждого свои trade-off):

- копия по фактической длине (`memcpy ln + 4` вместо 260) — семантика
  не меняется, меняются оба кодогена; выигрыш ограничен: длина
  известна только в рантайме, LLVM теряет константный размер;
- read-only параметры по ссылке (передача `str` в функцию без
  `let` — без копии; язык уже передаёт указатель, копию делает
  вызываемый) — самый большой выигрыш, но надо доказать отсутствие
  алиасинг-ловушек (`s.append_str(s)`) и зафиксировать семантику
  в SPEC §5;
- `str<N>` с настоящей параметризацией ёмкости (сейчас рантайм
  фиксирован на 256) — отдельная большая тема, тянет за собой
  layout рантайма.

**Объём.** SPEC §5 + оба кодогена + Rt.eat + интерпретатор +
verifier; полный паритет и фикспойнт. Самый инвазивный трек.

## Трек 2. Микроконтроллеры: размер, trap-строки, cross-compile

Язык уже идеален для МК по построению (статическая память, точная
верхняя граница стека §8, нет кучи). Не хватает:

- **политика trap-строк — сделано 2026-07-13**: режим `--trap-codes`
  (оба компилятора: флаг eatc / `SelfIrCodes` из `IrCodesMain.eat`,
  паритет — `make verify_trapcodes`); `eat_trap_code(u32)` в шиме,
  таблица `; trap <код>: <сообщение>` в хвосте .ll. Числа: Mos6502
  `__const` 4.8 → 0.1 КБ (−98 %), флеш-метрика −30 %; SelfIr −64 КБ
  (тринадцатая итерация FINDINGS);
- **замер размеров секций — сделано 2026-07-13**: секция `size` в
  bench (бинарник, `__text`/`__const`, строковые/trap-глобалы,
  канон .ll; одиннадцатая итерация FINDINGS);
- **cross-compile — сделано 2026-07-13**: mos6502 собран под
  Cortex-M3 (thumbv7m) и проходит `make verify_arm` — вывод в QEMU
  mps2-an385 байт-в-байт равен интерпретатору. Шим аксиом поверх
  CMSDK UART (теперь [mcu/common/runtime.c](../mcu/common/runtime.c)), вход
  прошивается в образ (у UART нет EOF), exit/trap — полухостинг;
  сборка clang'ом из канонического `.ll` + `ld.lld`, LTO на коде
  программы (кандидат из трека 4 встроен: −32 % флеша, линковка
  мгновенная). Флеш 7.1 КБ (хост-бинарник — 69.7 КБ); детали и
  числа — [mcu/README.md](../../mcu/README.md), семнадцатая итерация
  FINDINGS;
- стек по §8: mos6502 ≤ 262 594 Б (Cpu в кадре main) — в 4 МБ RAM
  платы влезает с запасом ×15; отчёт §8 — готовая нижняя граница
  выбора чипа;
- **этап 2 — «уровень реального проекта»: язык и порты сделаны
  2026-07-14** ([MCU_PLAN.md](MCU_PLAN.md)): в языке — hex-литералы,
  `~x`, u16, `extern func` (полный паритет, 51a8a0e/06d45ee/4e47a81);
  структура портов mcu/common + mcu/boards (mps2-an385, micro:bit
  nRF51, STM32VLDISCOVERY, Netduino Plus 2 — `make verify_mcu`
  сверяет все байт-в-байт), `make mcu BOARD= SRC= EXTERN=`,
  автосверка §8 против RAM платы валит сборку при переполнении;
  пример границы — examples/extern (один .eat, два драйвера,
  идентичный вывод хоста и QEMU);
- **u64 + i64 — сделано 2026-07-14** ([MCU_PLAN.md](MCU_PLAN.md)
  §3.4): виды 13/14, полный пакет; 64-битные `__aeabi_*` для ARM32 —
  mcu/common/eabi64.c, Blinky гоняет u64-деление/сдвиги на всех
  4 платах;
- **§5–6 — сделано 2026-07-16, MCU_PLAN закрыт**: периферийный шим
  `mcu/Mcu.eat` + `mcu/common/shim.c` (gpio/systick_ms/uart_poll,
  ISR → кольцо SPSC → extern, полная таблица векторов с диспетчером
  `board_irq`), флагман `examples/blinky_cli` (суперцикл `loop`,
  UART-CLI; 4 QEMU-платы байт-в-байт с хостовым эталоном) и
  4 прошивочных порта — `pico` (свой boot2.S + elf2uf2.py → UF2),
  `bluepill`, `f4discovery`, `nrf52840dk` (`make mcu_flash`);
  флеш 7.5–8.4 КБ, RAM ~1.5 КБ, тридцать четвёртая итерация
  FINDINGS. Остались исследовательские хвосты §7 (MMIO-встроенные,
  ISR на EATLang) — отдельные решения пользователя.

## Трек 4. Хвосты производительности компилятора (низкий приоритет)

По остаточному профилю SelfIr (0.13 с) и FINDINGS; детальный план и
порядок — [OPTIMIZATIONS_PLAN.md](OPTIMIZATIONS_PLAN.md) §2–3:

- `lb_tp`/копии в банкованные пулы: скалярный масками-доказанный
  цикл; попытка аффинной по-банковой формы, которую LLVM распознаёт
  как memcpy, не делалась — потенциал ~десятки мс;
- `g_hash` считается по содержимому при каждом commit — можно
  инкрементально при записи в пул;
- экранированные спаны дампов (`write_span_esc`, значения токенов) —
  посимвольные; батчить между escape-символами;

  −10 % времени самокомпиляции, −29 % размера, цена — 2.75 с

- F2 — **закрыт целиком 2026-07-13**: `eprint` в Rt.eat, все шесть

## Трек 6. Аргументы бинарника (решение пользователя 2026-07-14)

Программа должна уметь получать аргументы командной строки, а не
только stdin. Отдельный проект (не часть модулей/stdlib —
[MODULES_PLAN.md](MODULES_PLAN.md)); форма аксиом — решение по
SPEC §7 до кода. Эскиз:

- новые аксиомы шима (кандидат): `arg_count() -> u32`,
  `arg_len(i: u32) -> u32`, `arg_byte(i: u32, j: u32) -> u8` —
  побайтный доступ без кучи и буферов; циклы клиента ограничены
  `arg_len` (Power of 10), выход за границы — trap;
- интерпретатор: `eatc run App.eat -- a b c` — аргументы после
  `--`, паритет с бинарником;
- МК: у прошивки нет argv — аргументы прошиваются в образ по
  образцу `embed_input.py` (INPUT=), сверки остаются
  детерминированными;
- объём: runtime.c + mcu/common/runtime.c (шим растёт с 7 функций),
  declare в обоих кодогенах, тайпчекер/интерпретатор, SPEC §7,
  полный паритет и фикспойнт.
