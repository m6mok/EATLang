# План: HTTP-сервер на EATLang

- **Статус:** 📋 ЗАПЛАНИРОВАН (черновик)
- **Роль:** активный план
- **Правки языка/грамматики:** нет
- **Новые аксиомы ОС (SPEC §7):** да — сокеты (§5, решение пользователя)
- **Зеркало в selfhost:** `lib/Http.eat` и пример — на EATLang, зеркала не требуют; сокет-аксиомы шима зеркалятся в рантайм
- **Опирается на:** [MODULES_PLAN](MODULES_PLAN.md) ✅ (`lib/`)
- **Блокеры:** сокет-аксиомы (решение пользователя)
- **Блокирует:** —

Требует новых аксиом
ОС (сокеты) — а значит **решения пользователя** по SPEC §7
([MODIFYING.md](../MODIFYING.md), инвариант 4). Библиотечная часть
(парсер, роутер, шаблоны) правок компилятора не требует и выражается
наличными средствами. Правила Power of 10 не ослабляются: любое
превышение потолка — штатная HTTP-ошибка (413/431/400), а не trap.

Документ рамочный: он про **сервер HTTP** (нативный бинарник). WASM-клиент,
DOM-FFI и изоморфные server functions — **вне этого файла** (отдельный
трек уровня МК, начинать после шагов §8). Сначала снимаем самый крупный
архитектурный риск — совместимость accept-loop с «единственным `loop` в
`main`».

## 1. Итог одной строкой

HTTP/1.0 и **HTTP/1.1 — реализуемы** ценой объявления потолков (как
JSON: каждое свойство протокола бьётся об одно правило и лечится одним
числом). **HTTP/2 — нет** (четыре одновременных конфликта с языком +
обязательный TLS, §7). Сервер — это **циклический исполнитель**
([ASYNC_PLAN.md](ASYNC_PLAN.md)): accept-loop ложится на зарезервированный
единственный `loop` в `main`, обработка соединения — на тотальные
функции, парсинг — на автомат с явным стеком по образцу
[selfhost/Parser.eat](../../selfhost/Parser.eat) и [JSON_PLAN.md](JSON_PLAN.md).

## 2. Почему EATLang к этому приспособлен

- **Слот accept-loop уже есть.** Единственный `loop` в `main` введён как
  «модель реактивной системы» (SPEC §209). Слушающий сервер — это
  `loop { poll; принять готовое соединение; шагнуть }`. Ничего нового в
  грамматику не нужно.
- **Транзакция запрос→ответ тотальна по построению.** «Прочитал запрос
  (ограниченный) → посчитал → написал ответ» — конечная композиция
  тотальных функций. `eatc build` печатает верхнюю границу стека (§8) —
  готовая нижняя граница выбора платформы.
- **Идиома парсинга обкатана.** Request-line и заголовки — тот же автомат
  с явным стеком/курсором, что лексер/парсер selfhost и SAX-режим JSON.
- **bounded-depth — не ограничение, а security-фича.** Фиксированные
  потолки на заголовки/тело/вложенность бесплатно закрывают классический
  DoS (slowloris по таймауту, header-flood по `MAX_HEADERS`,
  тело-бомбу по `MAX_BODY`).
- **Ноль кучи проверяется отчётом.** `eatc build` показывает нулевой heap;
  пулы соединений и буфера живут в статике/кадре `main` — модель reqwless
  `TcpClientState`, но по построению, без аллокатора.

## 3. Рамка: свойство HTTP → правило → потолок

| Свойство HTTP | Правило | Потолок |
| --- | --- | --- |
| accept бесконечен | 2 / `loop` только в `main` | **единственный `loop` в `main`** |
| соединений заранее неизвестно сколько | 3 (нет кучи) | `MAX_CONN` — пул `[Conn; N]` + курсор |
| запрос произвольной длины | 3 | `REQ_CAP` — буфер `[u8; N]` → **413** |
| заголовков произвольно много | 3 | `MAX_HEADERS` — `[Header; N]` → **431** |
| keep-alive: N запросов на соединении | 2 (нет `while`) | `MAX_REQ_PER_CONN` → `Connection: close` |
| chunked: «читать, пока не 0-чанк» | 2 | `MAX_BODY` — граница цикла → **413** |
| `read_byte` блокирует исполнитель | (блокирующая аксиома) | неблокирующие сокет-аксиомы (§5) |
| ответ произвольной длины | 3 | `RESP_CAP` — буфер `[u8; N]`, курсор |

Ни одно превращение не открывает обход: превышение любого потолка —
именованная HTTP-ошибка-значение, программа продолжает обслуживание.

## 4. Числа (пределы сервиса)

Стартовые значения; выносятся в `const`, подбираются под задачу (edge-API,
IoT-шлюз, панель управления МК). Все — в границах §6 (массив ≤ 65 536).

| Константа | Значение | Смысл |
| --- | --- | --- |
| `MAX_CONN` | 16 | размер пула соединений (единица конкурентности) |
| `REQ_CAP` | 8 192 | буфер запроса на соединение → 413 при переполнении |
| `RESP_CAP` | 16 384 | буфер ответа на соединение |
| `MAX_HEADERS` | 64 | число заголовков (edge-http/httparse) → 431 |
| `MAX_REQ_PER_CONN` | 100 | запросов на keep-alive до принудительного close |
| `MAX_BODY` | 65 536 | суммарное тело (в т.ч. сумма chunked) → 413 |
| `MAX_ROUTES` | 64 | размер const-таблицы маршрутов |

Стоимость пула — `MAX_CONN × (REQ_CAP + RESP_CAP)` ≈ 384 КБ в кадре
`main`; для хоста ничтожно (стек self-host — 256 МБ), для МК число режется
под RAM платы, автосверка §8 валит сборку при переполнении (трек 2).

## 5. Сокет-шим: аксиомы ОС (решение пользователя, SPEC §7)

Единственная граница с сетью — расширение шима аксиом (`runtime.c`;
на МК аналога сети нет — сервер хостовый) по образцу трека 6 (argv) и
неблокирующих аксиом [ASYNC_PLAN.md §6](ASYNC_PLAN.md). Формы —
предмет построчного обсуждения:

| Кандидат-сигнатура | Смысл | Заметки |
| --- | --- | --- |
| `socket_listen(port: u16) -> Result<u32, IoError>` | bind+listen, вернуть дескриптор слушателя | дескриптор — `u32`, не указатель (правило 9) |
| `socket_accept(fd: u32) -> Result<u32, IoError>` | принять соединение; `Err(WouldBlock)` если нет | **неблокирующий** — иначе исполнитель встанет |
| `socket_read_byte(fd: u32) -> Result<u8, IoError>` | побайтный ввод; `Err(WouldBlock)` при пустом | границы/ошибки → значение, не trap |
| `socket_write_span(fd, buf, off, len)` | батч-вывод (по образцу `write_span`) | единственный путь ответа, без посимвольного |
| `socket_close(fd: u32)` | закрыть соединение, вернуть слот пулу | idempotent |

Тонкости:
- **`WouldBlock` меняет встроенный `IoError`** — это общая с
  [ASYNC_PLAN.md](ASYNC_PLAN.md) правка (`IoError { Eof, Fail, WouldBlock }`);
  решать один раз для обоих треков.
- Дескриптор `u32` — «адрес» в таблице ядра, язык его только передаёт;
  указателей и алиасинга не возникает.
- Нарушение контракта аксиомы (пишем в закрытый fd, читаем чужой) —
  trap с координатами, как у argv.
- Объём: `runtime.c` (шим растёт с 7 аксиом), `declare` в обоих
  кодогенах, тайпчекер, интерпретатор, SPEC §7, полный паритет и
  фикспойнт.

### Детерминизм и `make verify` — главный вопрос

Паритет «бинарник == интерпретатор байт-в-байт» — краеугольный. Сеть
недетерминирована, поэтому сверять «живой» сервер нельзя. Решение — по
образцу `INPUT=`-прошивки МК и виртуальных часов ASYNC: **записанный
транскрипт соединения** как детерминированный вход.

- В режиме сверки сокет-аксиомы читают не из ядра, а из прошитого в образ
  сценария (последовательность `accept`/байты запроса/close); вывод
  `socket_write_span` идёт в тот же поток, что обычный `write`.
- Интерпретатор и бинарник видят один сценарий → один вывод → `make
  verify` остаётся байт-в-байт.
- «Живой» режим (реальные сокеты) — только вне гейта сверки, отдельной
  целью (`make serve`), не в `make verify*`.

Это открытый вопрос §9: форма транскрипта и как переключается режим.

## 6. Архитектура сервера

### 6.1 Пул соединений без аллокации

```text
struct Header {
    name_off: u32   # срез в буфере запроса
    name_len: u32
    val_off: u32
    val_len: u32
}

struct Conn {
    fd: u32                  # дескриптор ядра, SENT если слот свободен
    req: [u8; 8192]          # REQ_CAP
    req_len: u32
    resp: [u8; 16384]        # RESP_CAP
    resp_len: u32
    headers: [Header; 64]    # MAX_HEADERS
    nh: u32
    state: u32               # автомат: 0 accept · 1 read · 2 handle · 3 write · 4 close
    served: u32              # для keep-alive: счётчик до MAX_REQ_PER_CONN
}

struct Server {
    listen_fd: u32
    conns: [Conn; 16]        # MAX_CONN — весь «heap» сервера
}
```

Слот берётся курсором/линейным поиском свободного `fd == SENT` — паттерн
«пул + курсор», который верификатор уже доказывает (трек 3). Указателей
нет: «текущее соединение» — `u32`-индекс в `conns`.

### 6.2 Accept-loop = единственный `loop` в `main`

Один `loop`; каждый виток — неблокирующий `poll` пула (циклический
исполнитель ASYNC yarus 0). Внутренние циклы — `for` со статической
границей, **не** второй бесконечный цикл:

```text
func main() {
    var srv: Server = server_init(8080)
    loop {
        # 1) принять готовые соединения в свободные слоты
        discard accept_ready(srv)
        # 2) один шаг автомата у каждого активного соединения
        for i in 0..16 {                 # MAX_CONN — не while
            if srv.conns[i].fd != SENT {
                discard step_conn(srv, i)   # тотальная, доказуемо завершается
            }
        }
        # завершение сервера — по сценарию/сигналу через аксиому
        if should_stop(srv) { break }    # единственный break на loop
    }
}
```

`step_conn` двигает автомат `state` соединения ровно на один шаг и
возвращается (модель Embassy/cyclic executive). Ни рекурсии, ни второго
`loop`, ни блокировки — throughput ограничен (однопоточный event-loop,
как nginx worker/Redis), масштабирование — процессами по `SO_REUSEPORT`,
не потоками (§7 «отказы»).

### 6.3 Чтение запроса — `for` с ранним `break`

```text
func read_request(var c: Conn) -> Result<bool, HttpErr>
    requires true
    ensures true
{
    for _ in 0..8192 {                   # REQ_CAP — граница, не while
        match socket_read_byte(c.fd) {
            Ok(b)  { if push_byte(c, b)? { return Ok(true) } }  # true = увидели CRLFCRLF
            Err(e) {
                match e {
                    WouldBlock { return Ok(false) }   # ещё не всё пришло — шагнём позже
                    Eof        { return Ok(true) }
                    Fail       { return Err(HttpErr.IoFail) }
                }
            }
        }
    }
    return Err(HttpErr.TooLarge)         # переполнили REQ_CAP → 413
}
```

## 7. Версии HTTP: что реализуемо, а что НЕ

### HTTP/1.0 — тривиально

Одна транзакция на соединение, `Content-Length` или закрытие. Прямое
попадание в тотальную модель.

### HTTP/1.1 — реализуемо ценой потолков

Всё, что 1.1 добавляет, **ограничиваемо**, а значит language-friendly:

- **keep-alive** — соединение живёт через N запросов: не «неограниченное
  время», а `served < MAX_REQ_PER_CONN`, затем `Connection: close`.
  Крышка цикла — идиома языка, не костыль.
- **chunked** — «читать до чанка нулевой длины» = цикл с зависящей от
  данных границей (нарушение правила 2); лечится потолком `MAX_BODY`
  (`for` до границы, превышение → 413).
- **Host обязателен, 100-continue, pipelining** — доп. состояния в том же
  автомате парсинга; новых механизмов не требуют.

Вывод: **HTTP/1.1 — честная цель.** Транзакция тотальна по природе, язык
на неё ложится идеально.

### HTTP/2 — НЕ реализуемо (четыре конфликта + TLS)

Не «отдельная фича невозможна» (почти всё поодиночке ограничивается) —
ломается **форма**: долгоживущее мультиплексируемое соединение упирается в
язык сразу в четырёх местах:

1. **Модель соединения против «одного `loop`».** Единица работы в 2 —
   долгоживущее соединение, мультиплексирующее параллельные стримы = по
   сути event-loop на соединение. Всё придётся свернуть в один плоский
   автомат над пулом `соединений × стримов` внутри главного `loop` —
   чудовищный state machine, вся «естественная конкурентность» протокола
   уходит в ручную диспетчеризацию.
2. **HPACK.** Сжатие заголовков держит **динамическую таблицу** —
   мутабельный словарь на всё время соединения со вставками/вытеснением
   (LRU по байтовому бюджету) плюс **декодирование Хаффмана** (побитовый
   проход кодами переменной длины). Представимо через `[Entry; N]` +
   курсор, но это уже безопасно-чувствительная стейтфул-подсистема
   (HPACK-бомба — известный класс CVE), которой в 1.x нет.
3. **Тотальность + trap работают против сервера.** Flow control 2 — учёт
   кредитов окна (`WINDOW_UPDATE`), знаковая арифметика, окна временно
   уходят в минус. В EATLang переполнение — **trap, не wrap**: кривой пир
   подбирает значение и роняет сервер = DoS. Каждую операцию над данными
   пира надо доказывать верификатором или оборачивать saturating-логикой.
4. **TLS решает вопрос за нас.** Браузеры говорят по 2 практически только
   поверх TLS с ALPN `h2`; `h2c` клиенты не поддерживают. TLS в языке нет
   и по-хорошему быть не может (проект уровня embedded-tls). Значит «2 на
   чистом EATLang для браузера» отпадает ещё до HPACK.

### HTTP/3 — вне рамок полностью

QUIC поверх UDP: собственный транспорт, congestion control, встроенный
TLS 1.3, connection migration. Это целый сетевой стек, не «версия HTTP».

### TLS

HTTPS — либо терминация на прокси (nginx/haproxy впереди, как рекомендует
picoserve), либо отдельный C-шим (OpenSSL/mbedTLS). Чистый EATLang-TLS —
огромный самостоятельный проект, вне этого плана.

## 8. Этапы

Порядок — «сначала самое ограничивающее»: сокеты и accept-loop → парсинг
без кучи/рекурсии → роутер. Границы этапов и связь с `lib/` (import/export,
снапшот `verify_sig`) — по [MODULES_PLAN.md](MODULES_PLAN.md).

0. **Сокет-шим + эхо-сервер.** Аксиомы §5 (неблокирующие), пул из одного
   слота, `main` с единственным `loop`, чтение в `[u8; N]` и `socket_write_span`
   обратно. Режим сверки через транскрипт (§5). **Порог:** стабильный эхо,
   нулевой heap в отчёте `eatc build`, `make verify` зелёный на сценарии.
1. **Минимальный HTTP/1.1, один запрос за раз.** `lib/Http.eat`: автомат
   request-line + заголовки в фиксированные буфера; статический ответ.
   **Порог:** «Hello, world» по HTTP/1.1, ноль malloc, 413 при переполнении
   `REQ_CAP`.
2. **Пул + keep-alive.** `[Conn; MAX_CONN]`, автомат `state`,
   `MAX_REQ_PER_CONN`, циклический исполнитель §6.2. **Порог:** несколько
   одновременных соединений на одном потоке, корректный `Connection: close`.
3. **Роутер + обработчики + шаблоны.** `const ROUTES: [Route; N]` (метод,
   шаблон пути, id); матчинг — `for` по таблице с `break`, диспетч —
   исчерпывающий `match` по id (указателей на функции нет — граф статичен);
   параметры пути в поля фиксированного размера; тело ответа — bounded
   string building поверх `append_*` из [Rt.eat](../../selfhost/Rt.eat) →
   `socket_write_span`. **Порог:** несколько маршрутов с параметрами,
   генерируемый HTML. Роут-таблица — кандидат на comptime
   ([COMPTIME_PLAN.md](COMPTIME_PLAN.md): perfect-hash на компиляции).
4. **Тело/сериализация.** JSON-тело — через [JSON_PLAN.md](JSON_PLAN.md)
   (bounded-depth, глубже лимита → 400). **Порог:** round-trip `struct` ↔
   JSON в `test`-блоках.

WASM-клиент, DOM-FFI, изоморфные server functions — **вне этого файла**;
открывать отдельным треком после этапа 3, самостоятельным решением.

## 9. Открытые вопросы (нужно решение пользователя)

- Формы и имена сокет-аксиом §5; расширять `IoError` вариантом
  `WouldBlock` (общее с [ASYNC_PLAN.md](ASYNC_PLAN.md)) или отдельный тип.
- **Модель детерминизма для `make verify`**: форма транскрипта соединения,
  как переключается «сверка/живой» режим, отдельная цель `make serve`.
- Стартовые числа §4 (особенно `MAX_CONN`/`REQ_CAP` под целевую платформу).
- Нужен ли `lib/Http.eat` уже на этапе 1 или сначала «сырой» пример в
  `examples/`.
- Порт как аргумент бинарника (трек 6, argv) или `const`.

## 10. Что реализуем НЕ (явные отказы)

- **HTTP/2 и /3** — §7: конфликт формы с языком + обязательный TLS;
  терминировать на прокси, на EATLang оставить 1.1 за прокси.
- **Потоки / вытеснение** — гонки и недетерминизм несовместимы с паритетом
  и правилом 9; масштабирование — процессами (`SO_REUSEPORT`), не потоками.
- **Динамические сессии/структуры** — только пул фиксированного размера с
  вытеснением (LRU по курсору, как heapless); внешнее хранилище (БД) —
  свой сокет/IPC-шим отдельным проектом.
- **WebSocket/дуплекс** — долгоживущее накопление состояния плохо ложится
  на тотальную модель; максимум — фиксированное число WS-слотов с
  обработкой кадров в главном `loop`, полноценный дуплекс требует
  event-loop-примитивов в C. Пока — отказ.
- **TLS в языке** — только прокси или C-шим (§7).
- **Float** — цены/аналитика через целые/fixed-point
  ([FIXED_POINT_PLAN.md](FIXED_POINT_PLAN.md)); в языке float нет осознанно.

## 11. Инварианты (не открывать обходов)

- Ни одной функции, вызывающей себя прямо или косвенно (граф — DAG);
  парсинг — только явный стек кадров.
- Единственный бесконечный `loop` — accept-loop в `main`; внутри только
  `for` со статической границей, один `break` на цикл.
- Превышение любого потолка §4 — HTTP-ошибка-значение (413/431/400/431),
  не trap и не молчаливое усечение; результаты аксиом не отбрасываются
  молча (правило 7).
