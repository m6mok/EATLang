## 1. Итог одной строкой

Language server для EATLang, **написанный на самом EATLang** и линкующий
фазы компилятора `lex+parse+check+verify` как модули: один бинарник
говорит по JSON-RPC поверх stdio и делает видимыми в VSCode ровно те
доказательства, которыми уникален язык, — диагностики, доказанные/оставшиеся
trap-проверки, границы памяти §8, типы и контракты.

## 2. Почему язык к этому приспособлен

LSP-сервер — это событийный цикл: читать сообщение, ответить, повторить.
Это **ровно реактивная модель EATLang** (SPEC §5.1): один корневой `loop`
в `main`, каждая итерация тотальна, состояние — bounded-пулы в кадре
`main`. Сервер строится как `Server.tick` из `lib/http/Server.eat`, только
источник байтов — stdin (`read_byte`), а не сокет.

Больше того — язык приспособлен к этому **лучше**, чем к обычному
приложению, потому что три вещи уже готовы и переиспользуются как есть:

- **Анализатор.** Фазы `selfhost/` — не монолит, а структуры-по-значению
  с публичными конструкторами (`lexer_new → parser_new → check_new →
  verify_new`) и результатами в SoA-пулах (токены в `Lexer`, узлы AST в
  `Parser`, обязательства `ck_k/ck_n/ck_o` в `Verify`). Дампы — отдельные
  методы; ось `-O` уже читает `v.nck` напрямую, минуя текст. LSP-хэндлер
  делает то же: гоняет фазы, читает пулы, **не зовёт** `dump()`/`print`
  (stdout занят каналом протокола).
- **Транспорт.** `Content-Length` + JSON-тело — та же рамка, что у HTTP:
  `Header`/`find_header`/`span_u32`/`content_length` и `Body.begin_cl`/`push`
  из `lib/http/Http.eat`; разбор и сборка JSON — `JDoc.parse` + `JOut`
  из `lib/json/Json.eat` — переносятся почти 1:1.
- **Реестр документов** — прямой шаблон `Store` из
  `examples/http/todo/`: массив фикс. размера + счётчик + флаг слота,
  живёт в кадре `main`, переживает сообщения.

Диагностики сервера — это диагностики **настоящего** компилятора (те же
фазы), поэтому они не могут разойтись с поведением сборки.

## 3. Рамка: свойство → правило → потолок

| Свойство сервера | Правило Power of 10 | Потолок |
| --- | --- | --- |
| Цикл обслуживания | правило 1 (один `loop` в `main`) | 1 корневой `loop`, `break` по `exit`-уведомлению |
| Реестр открытых документов | правило 3 (пул — часть типа, не куча) | `LSP_MAX_DOCS` слотов, каждый — буфер `LSP_DOC_CAP` байт |
| Кадр входящего сообщения | правило 3 | `LSP_MSG_CAP` байт (по образцу `HTTP_MAX_BODY`) |
| Разбор JSON-RPC и обход AST | правило 1 (нет рекурсии) | явные стеки кадров (как в `Json.eat`), пулы фикс. размера |
| Линковка всех фаз в один бинарник | правило 4 (≤ 2048 функций) | ~1017 (фронтенд) + транспорт/диспетч/хэндлеры ≈ **1250/2048** |
| Прогон анализа на сообщение | правило 2 (`for` с границей) | один общий анализ-кадр, переиспользуется (`reset`) |

## 4. Числа (пределы)

Все — bounded, кандидатные значения (уточняются замером на этапе 0–1):

- `LSP_MAX_DOCS = 32` — открытых документов одновременно. Полный реестр →
  сервер отвечает ошибкой, **не растёт** (аналог `507` в `Store`).
- `LSP_DOC_CAP = 65536` байт на документ (как `HTTP_MAX_BODY`; крупнее —
  диагностика «файл превышает предел LSP», редактирование не блокируется).
- `LSP_MSG_CAP = 262144` байт на входящий JSON-RPC кадр (тела `didChange`
  крупнее одного документа с запасом; превышение → ошибка протокола).
- **Функции:** фронтенд `lex+parse+check+verify` = 744 (уже существует как
  `VerifyMain`); + все пять фаз с `ir` = 1017 (не нужен — ir серверу не
  требуется). Бюджет LSP-слоя (транспорт + диспетчер + ~12 хэндлеров) —
  оценка 200–350 функций. Итог ~950–1100 из 2048, запас ~2×.
- **Память:** большие пулы фаз (токены `[u32;65536]×4`, узлы AST
  `[u32;65536]×2`, обязательства `[u32;16384]×3`) существуют в кадре
  `main` в **одном** экземпляре — общий анализ-кадр. На документ хранится
  только его текст (`LSP_DOC_CAP`) и **компактный** кэш диагностик
  (позиция + вид + вердикт), не полное состояние анализа. Пересчёт — по
  запросу (`didChange`), в переиспользуемый кадр.

## 5. Аксиомы ОС

Новых нет. Всё покрыто наличными аксиомами (SPEC §7):
- вход — `read_byte() -> Result<u8, IoError>` (stdin, `Err(Eof)` в конце);
- выход — `write_span(a, off, len)` (тело ответа одним вызовом ОС),
  при нужде `write_byte`;
- журнал сервера (не мешая каналу протокола) — `write_err_byte` в stderr;
- завершение — `exit(code)` в `main` по уведомлению `exit`.

## 6. Архитектура / механизм

Два артефакта:

**(A) Сервер `editor/lsp/` — программа на EATLang.** Склейка модулей на
входе `eatc build`: `Rt.eat` → `lib/json` → новый `editor/lsp/Transport.eat`
(header-парсер и stdio-драйвер) → `editor/lsp/Rpc.eat` (диспетчер по
`method`) → фазы `selfhost/lex,parse,check,verify` → `editor/lsp/Handlers.eat`
→ `editor/lsp/LspMain.eat` (`main` с реестром документов и корневым `loop`).

Виток цикла (по образцу `Server.tick`, источник — stdin):
1. дренаж `read_byte()` в header-парсер до `PARSE_DONE` (пустая строка);
2. `content_length()` → `Body.begin_cl(n)` → добор `n` байт тела;
3. тело → `JDoc.parse` → `method`/`id`/`params`;
4. диспетчер → хэндлер; хэндлер строит ответ `JOut`;
5. `Content-Length: {JOut.n}\r\n\r\n` + `write_span(JOut.buf, 0, n)`;
6. `break` только по уведомлению `exit`.

**Реестр документов** — `struct DocStore { docs: [Doc; 32]; count }` в
кадре `main` до `loop`; `Doc { uri, version, buf[65536], used }`. `didOpen`
→ занять слот; `didChange` (полная синхронизация текста на старте) →
заменить текст + пересчёт; `didClose` → освободить слот. Поиск по URI —
линейный, как `Store.find`.

**Анализ** — общий кадр `let an = analyzer_new()` (или пересоздание на
запрос, если так чище по памяти): `parser_new(lexer над текстом)` →
`check_new` → `run_typed()` → `verify_new` → `analyze()`. Читаем позиции
ошибок (`EatError` → line:col) и обязательства (`v.ck_*`); **не** зовём
`dump()`.

**Конверсия позиций.** Компилятор даёт 1-based `line:col` в байтах; LSP
хочет 0-based `line`/`character` в кодовых единицах UTF-16. Нужен слой
пересчёта байтовой колонки → UTF-16 (bounded проход по строке документа).
Это единственная нетривиальная новая машинерия транспорта; её граница —
длина строки документа.

**(B) Клиент `editor/vscode/` — тонкое расширение.** `LanguageClient`
(vscode-languageclient), который запускает бинарник сервера как процесс с
транспортом stdio. `package.json`: `activationEvents` на язык `eat`, `main`
= точка входа, зависимость `vscode-languageclient`. TextMate-грамматика
остаётся — LSP её дополняет (семантика поверх лексической подсветки), не
заменяет.

## 7. Что реализуем НЕ

- **Не порождаем процессы.** В EATLang нет process-аксиом — сервер не
  «зовёт `eatc`», а линкует фазы в себя. Это осознанный выбор, а не
  ограничение: диагностики тем самым тождественны компиляторным.
- **Не инкрементальная синхронизация текста на старте.** Полный текст на
  `didChange` (`TextDocumentSyncKind.Full`). Инкрементальные правки —
  возможный поздний этап, только по замеру задержки.
- **Не инкрементальный анализ.** Пересчёт всего документа на изменение
  (документ bounded ≤ 64 КБ — быстро). Кэш фаз между правками — не делаем.
- **Не форматирование, не переименование, не rename-refactor** на первых
  этапах — это отдельные capability, вводятся по спросу.
- **Не multi-root / не кросс-файловый анализ** сверх модели модулей
  компилятора на первых этапах.
- **Не сервер на Python (pygls)** и не shell-out — отвергнуто выбором
  пользователя в пользу self-host.
- **Не трогаем `selfhost/*` на запись** — только читаем как модули.

## 8. Этапы

Каждый — со своим определением готовности.

**Этап 0 — Транспорт и рукопожатие (proof of concept). ✅ 2026-07-18.**
Сделано: `editor/lsp/Transport.eat` — самодостаточный мини-парсер
заголовков (`scan_clen`, старт сразу с заголовков, без request-line
Http; регистронезависимо через `to_lower` из `lib/fmt/Ascii.eat`),
stdio-драйвер на аксиоме `read_byte` (`next_line`/`read_headers`/`read_body`
→ тело в `JDoc.put`), диспетч (`handle`/`dispatch`) и ответы билдером
`JOut` (`reply_initialize`/`reply_shutdown`/`reply_error`, `send` с
CRLF-обрамлением). `editor/lsp/LspMain.eat` — корневой `loop` в `main`,
зовёт `server.step()`, выход по EOF/`exit`, единственный `exit(0)`.
Клиент `editor/vscode/extension.js` (чистый JS, `vscode-languageclient`)
запускает сервер по stdio; подсветка декларативна и переживает падение
сервера. Хэндлеры: `initialize` (capabilities: `textDocumentSync: 1`,
`serverInfo`), `initialized`, `shutdown` (`result: null`), `exit`;
неизвестный запрос → `-32601`. *Готово:* `make verify_lsp` зелёный
(транскрипт рукопожатия == `tests/lsp/handshake.golden`, вход строит
`tests/lsp/verify.sh` с CRLF-обрамлением); `make check` включает LSP
(27 тестов, `scan_clen` чистый — без аксиом); `verify_sig` пересобран
(`to_lower` в `lib.sig`). *Отклонение от §6:* диспетч этапа 0 живёт в
`Transport.eat` (не отдельные `Rpc.eat`/`Handlers.eat`) — он ~40 строк;
распил на `Rpc`/`Handlers` — при росте на этапе 1. Клиент автоматически
не тестируется (нет npm/VSCode в гейте) — проверяется вручную; сервер
и рукопожатие в гейте.

**Этап 1 — Диагностики (`publishDiagnostics`).** `didOpen`/`didChange`
(full sync) → реестр документов → прогон `lex+parse+check` в анализ-кадр
→ `EatError` (line:col) → конверсия позиций → `publishDiagnostics`.
*Готово:* синтаксическая/типовая ошибка в `.eat` подчёркивается красным
вживую; транскрипт-тест на примере с ошибкой.

**Этап 2 — Verifier-inlays (флагман).** Линковка `verify`; чтение
`v.nck`/`ck_k`/`ck_n`/`ck_o`; публикация как `textDocument/inlayHint`
(или диагностики уровня Hint): ✓ «доказано на компиляции» / ⚠ «останется
trap в рантайме» на каждой проверке `bounds/overflow/div/cast/shift/
requires/ensures/assert`. *Готово:* на примере с недоказанной границей
видно ⚠, с доказанной — ✓; переключение `-O` отражается.

**Этап 3 — Hover.** `textDocument/hover`: тип выражения из структур
`Check`/typed; сигнатура функции; эффективный контракт (омиссия ≡ `true`).
Плюс **учительский hover «почему trap»**: на ⚠-обязательстве — какого
инварианта/`requires` не хватило верификатору, подсказка в рамках языка.
*Готово:* наведение на выражение даёт тип; на ⚠-проверку — объяснение.

**Этап 4 — Автодополнение + snippets.** `textDocument/completion`:
встроенные (`print`/`write`/`socket_*`/…), ключевые слова, поля структур
и методы по типу приёмника, конструкторы `Ok/Err/Some/None`. Snippets:
`func` с `requires/ensures`, `for … in`, `match`, `test`. *Готово:*
дополнение полей/методов и вставка каркаса функции с контрактами.

**Этап 5 — CodeLens бюджета §8.** Над `main`/функциями — отчёт памяти
(стек худшей цепочки, кадры, глобалы) и `функций N/2048` из отчёта
`build`. Power of 10 правило 3–4 в глаза. *Готово:* CodeLens показывает
стек и счётчик функций, обновляется на правку.

**Этап 6 (опц.) — Test Explorer и CodeLens действий.** `test`-блоки → в
Test Explorer VSCode; CodeLens `run ▸ / verify ✓ / build ⚙` над `main`.
*Готово:* тест-блоки запускаются из панели тестов.

## 9. Паритет

Зеркала `src/eatc/ ↔ selfhost/` нет — план новый код компилятора не
пишет (см. поле «Зеркало в selfhost»). Сверка вместо паритета — **транскрипт
JSON-RPC-сессии** (`tests/lsp/*.jsonl` → `make verify_lsp`), по образцу
`EAT_NET` для HTTP: детерминированная последовательность запрос→ответ,
байт-в-байт. Если позже сервер захочет собственный структурный вывод
(например, эталон для дифф-теста) — оформить как стабильный построчный
формат, как у прочих дампов.

## 11. Инварианты

План **не открывает** ни одного обхода Power of 10:

- **Куча — нет.** Реестр документов, кадр сообщения, анализ-кадр — пулы
  фикс. размера в кадре `main`; переполнение → ошибка-значение (как `507`
  у `Store`), не рост.
- **Рекурсия — нет.** Разбор JSON и обход AST — явные стеки кадров
  (наличная машинерия `Json.eat`/фаз); граф вызовов остаётся DAG.
- **Цикл без границы — нет.** Единственный `loop` — корневой в `main`
  (правило 1), как у HTTP-сервера; все внутренние циклы — `for` с
  bounded-границей.
- **Глобалов — нет.** Состояние сервера живёт в кадре `main` значением,
  передаётся по `let self`.
- **Молча отброшенного результата — нет.** Ответы протокола и коды фаз —
  явные; неиспользуемое — через `discard`.
- **Warning вместо ошибки — нет.** Сервер лишь **показывает** вердикты
  компилятора; правил не ослабляет.
