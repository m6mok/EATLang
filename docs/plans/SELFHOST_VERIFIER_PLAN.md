# План: статический верификатор в self-hosted компиляторе

- **Статус:** 🚧 В ПРОЦЕССЕ (этап 0 из 6 готов, коммит e76a025)
- **Роль:** активный план
- **Правки языка/грамматики:** нет
- **Новые аксиомы ОС (SPEC §7):** нет
- **Зеркало в selfhost:** да — план и есть перенос фазы `verifier.py` в `selfhost/Verify.eat`, дампы `eatc verify` байт-в-байт
- **Опирается на:** [VERIFICATION_PLAN](VERIFICATION_PLAN.md) ✅ (эталон верификатора), [SELFHOST](SELFHOST.md) ✅ (архитектура фаз)
- **Блокеры:** нет — этап 1 (`selfhost/Verify.eat`) можно начинать
- **Блокирует:** — (собственный этап 5 «элизия» подчинён [SELFHOST_OPT_PLAN](SELFHOST_OPT_PLAN.md))

Верификатор — **единственная фаза, не зеркалированная в selfhost**
(таблица фаз в [SELFHOST.md](SELFHOST.md), строка «Верификатор |
`verifier.py` | — (только Python)»). Этот документ — план его порта:
`selfhost/Verify.eat` как зеркало [verifier.py](../src/eatc/verifier.py)
(2248 строк — самый алгоритмически плотный модуль эталона), с
паритетом байт-в-байт по дампу `eatc verify` и — на этапе 5 — элизией
проверок в оптимизационной оси `-O`.

Решение пользователя (2026-07-15): идём **полным планом до
фикспойнта** (этапы 0–5); наблюдаемый артефакт сверки — **новая
команда `eatc verify`** (детерминированный дамп обязательств, diff
пофайлово как `lex/parse/sig/typed/ir`).

**Связь с [SELFHOST_OPT_PLAN.md](SELFHOST_OPT_PLAN.md).** Этапы 0–4
(движок анализа + дамп `eatc verify`) ортогональны оптимизационной
оси и являются той самой разблокировкой, которую OPT_PLAN ждёт
(его §2, §5: «когда selfhost получит верификатор — снятие проверок
войдёт в `-O` одним семейством»). **Этап 5 (элизия в IR) целиком
подчинён OPT_PLAN**: доставка — не отдельный бинарник, а проход(ы)
единой оси `-O`/`SelfIrOpt` (см. §3 этапа 5). Верификатор — триггер,
делающий ось `-O` осмысленной (сегодня в ней лишь fold, OPT_PLAN D0).

## 1. Поверхность паритета (суть задачи)

У всех зеркалированных фаз есть эталонный дамп, который diff'ается
байт-в-байт. У верификатора наблюдаемых артефактов **два**, оба нужно
сделать сверяемыми:

1. **Результат анализа** — множество `(вид проверки, узел) → доказано?`
   и агрегат отчёта. Сейчас детерминированно нигде не печатается:
   `build` выводит только сводку `доказано N из M`, порядок обязательств
   — по `id(node)` (невоспроизводим). → **этап 0** вводит канон дампа.
2. **Элизия в IR** — доказанная проверка исчезает из бинарника
   ([codegen.py](../src/eatc/codegen.py): `if getattr(stmt, "proven",
   False): return`; атрибуты `no_overflow/in_bounds/div_safe/cast_ok/
   shift_ok`, `requires_proven/ensures_proven`). Но **канон selfhost —
   это `eatc ir`, где проверок нет — они все в рантайме**. Верификатор
   в selfhost создаёт **новый канон IR** — `eatc build`-IR (проверки
   сняты), — который SelfIr-с-верификатором обязан воспроизвести
   байт-в-байт, включая **новый фикспойнт bootstrap**. → **этап 5**.

## 2. Почему это самый инвазивный self-host трек

`verifier.py` держит структуры, дорогие в EATLang (нет куч, хеш-мап,
рекурсии; всё — фиксированные пулы):

- `State` (verifier.py:211) = **словари по строковым путям** (`ivs`,
  `rels`, `nz`, `holes`; ключи `имя` / `self.поле` / `var.поле`).
  Хеш-мап нет → **интернированный пул путей** + параллельные массивы,
  `copy`/`join`/`kill` над индексами.
- **Замыкание разностных ограничений** (`State.closure`, verifier.py:272,
  Флойд–Уоршелл) — O(n³) по пулу путей.
- **Итеративные пассы-фикспойнт** инвариантов пулов/полей (`run`,
  verifier.py:381, до 3 проходов) + «пробный» проход без следов
  (`_probe`, `_begin_pass`/`_finish_pass`).
- `Iv` (диапазоны), hull/inter, символический `result`, summaries
  функций (`hive`, дырки-сентинелы) — всё поверх пулов, без рекурсии.

Образцы recursion-free машин уже есть в [Check.eat](../selfhost/Check.eat)
(машины выражений/инструкций, resolve/const_eval — явные стеки кадров);
порт идёт по этим образцам. Объём — сопоставим с `Check.eat` (237 КБ).

## 3. Этапы

### Этап 0. Канон дампа `eatc verify` (только Python) — СДЕЛАН 2026-07-15

Реализация: `verify_dump` в [verifier.py](../src/eatc/verifier.py)
(канонический порядок видов — `_KIND_ORDER`), `cmd_verify` в CLI;
`verify_suite` кросс-сверяет агрегат дампа со stats. На полном
bootstrap-входе (SELFHOST_IR): 8381 обязательство, дамп
детерминирован байт-в-байт, ~0.9 с.

- Новая команда `eatc verify <файл>` → детерминированный отчёт
  обязательств в **стабильном порядке** (файл → строка → колонка →
  вид), строки вида `bounds 12:7 proven` / `overflow 30:5 runtime`,
  футер `verify proven=N total=M bounds=a/b overflow=c/d div=… cast=…
  shift=… requires=… ensures=…`.
- Стабилизировать порядок (сейчас `id(node)` в `Verifier.checks`,
  verifier.py:351) сортировкой по позиции узла.
- Согласовать с `verify_suite` (он проверяет значения `#! expect:`,
  этот дамп — источник тех же чисел).
- Инвазивность нулевая: переиспользует существующую карту `checks`.

### Этап 1. Инфраструктура состояния в EATLang

- `selfhost/Verify.eat`: пул интернированных путей; `Iv` = пара
  (lo, hi) + флаги; `State` на параллельных массивах (ivs/rels/nz/holes
  по индексу пути); `copy`/`kill`/`join`.
- Машина потока (`_flow_stmt/_flow_block/_flow_if/_flow_for/_flow_match`,
  verifier.py:752–1289) — явный стек кадров, по образцу `Check.eat`.
- `_iv` (verifier.py:1666) — интервальная оценка выражений; `_refine`
  (уточнение по условиям/requires/assert).
- Паритет `eatc verify` интервального ядра на подмножестве
  `tests/verify/` (кейсы 01–06, 09–10, 16 — bounds/overflow/div).
  Гейт — на **курируемом списке файлов**, растёт по мере покрытия
  (как `verify_suite`).

### Этап 2. Отношения и контракты

- Разностные ограничения + замыкание (`closure`, `_decide_offset`,
  `_decide_rel`, verifier.py:1587–1665); структурное равенство
  (`_expr_equal`, verifier.py:1519, коммутативность `+`/`*`); ускорение
  циклов (`_accelerate/_loop_body_env/_loop_exit`, verifier.py:982–1150);
  lockstep-счётчики (`_lockstep`, verifier.py:1151).
- Модульные контракты и summaries (`_assume_ensures/_contract_iv/
  _summary/_map_side/_combine_sides`, verifier.py:1953–2149).
- Паритет на кейсах 07–15, 18–38.

### Этап 3. Пассы-фикспойнт и дырочный домен

- Инварианты пулов/полей (`_begin_pass/_finish_pass/_note_pool/
  _note_field/_store_array`, до 3 проходов, verifier.py:403–619).
- Сентинел-дырки (`holes`, `_hive_summary`, verifier.py:708–751);
  дизъюнктив через циклы (`_loop_exit`, `_exit_rec`, verifier.py:1117).
- Пробный проход без следов (`_probe`) — как флаг подавления отметок.
- Паритет на кейсах 39–50 и полном `tests/verify/`.

### Этап 4. Полнорепный паритет дампа

- `SelfVerify` (`selfhost/VerifyMain.eat`) против `eatc verify` на
  **каждом** `.eat` репозитория (как `verify_selfhost`).
- Главный тест — **верификатор верифицирует сам себя** (конкатенация
  всех модулей фронтенда + `Verify.eat`).
- Новый гейт `make verify_selfhost_verify`.

### Этап 5. Элизия в IR — проход(ы) оси `-O` (подчинён OPT_PLAN)

**Механизм задан [SELFHOST_OPT_PLAN.md](SELFHOST_OPT_PLAN.md), не этим
документом.** OPT_PLAN §2 запрещает бинарник-на-оптимизацию (`SelfBuild`
/ `SelfIrFold` / … — комбинаторный взрыв); доставка — **семейство
проходов единой оси** `eatc ir -O` ↔ `SelfIrOpt`.

- Прокинуть аннотации доказанности из `Verify.eat` в
  [Ir.eat](../selfhost/Ir.eat) по контракту OPT_PLAN §4 (массивы
  аннотаций по id узла, как `nty`; кодоген читает доп. `if`):
  `no_overflow/in_bounds/div_safe/cast_ok/shift_ok`,
  `requires_proven/ensures_proven` (+ `llvm.assume`, `nsw/nuw` —
  build-only каналы, OPTIMIZATIONS_PLAN §1а/2.1).
- **Канон-эталон** — `eatc ir -O`, гоняющий **ровно selfhost-подмножество
  проходов** (OPT_PLAN §5, D2), а НЕ весь `eatc build`. Порядок проходов
  фиксирован: `fold` до `verify` (как в Python
  [cmd_build](../src/eatc/__main__.py) — точки `[v,v]` снимают проверки
  ниже); порядок — часть паритет-контракта.
- Синергия fold↔верификатор (comptime-точки → снятые границы,
  OPTIMIZATIONS_PLAN §5) воспроизводится обеими сторонами идентично,
  потому что обе гоняют один набор проходов в одном порядке.
- Паритет: `eatc ir -O` == `SelfIrOpt` байт-в-байт (гейт
  `verify_selfhost_opt`, расширяется на верификаторное семейство).
  **Bootstrap-фикспойнт на opt-режиме — необязателен** (OPT_PLAN D6):
  канон `eatc ir` ↔ `SelfIr` (без `-O`) остаётся жёстким фикспойнт-
  гейтом; `-O` сверяется своим diff-гейтом.
- Проверить взаимодействие с `--trap-codes`.

**Зависимость от решения D0 (OPT_PLAN §7):** запуск оси `-O` —
решение пользователя. Пока ось не заведена (или заведена только под
fold), этап 5 ждёт; этапы 0–4 самодостаточны и дают полноценный
self-hosted верификатор со сверкой дампа.

## 4. Определение готовности трека

К существующему гейту добавляются:

```sh
make ... verify_selfhost_verify        # этапы 0–4
make ... verify_selfhost_opt           # этап 5 (расширение оси -O)
```

`verify_selfhost_verify` — `SelfVerify == eatc verify` на каждом `.eat`
плюс самоприменение; `verify_selfhost_opt` — `eatc ir -O == SelfIrOpt`
байт-в-байт с верификаторным семейством проходов (OPT_PLAN §10.3).
Канонный фикспойнт `verify_bootstrap` (без `-O`) не трогается;
фикспойнт на `-O` необязателен (OPT_PLAN D6). Частично зелёное не
коммитится.

## 5. Риски и открытые вопросы

- **Пулы путей**: оценить максимум различимых путей в одной функции
  фронтенда — заложить предел пула (SPEC §6) и trap на превышение,
  как у прочих пулов. Замер — на `Check.eat`/`Ir.eat`.
- **Кадр `main`**: `Verify.eat` добавит пулы состояния — проверить
  `STACK_FLAGS` (сейчас 128 МБ) на самоприменении.
- **Стабильный порядок обязательств**: узел может породить несколько
  проверок (напр. `overflow` и `bounds`) — фиксировать вторичный
  ключ (вид) детерминированно.
- **Стоимость замыкания** O(n³): если горячо на больших функциях —
  ограничить пул путей / кэшировать, сверяя, что паритет цел.
