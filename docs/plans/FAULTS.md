## 2026-07-19 · OPTIMIZATIONS_PLAN · энейблер §2.3/§4 (счётчик инструкций)

1. **Debian собирает `--enable-plugins` только для `qemu-system-*`;
   `qemu-user` (`qemu-aarch64`) — без плагинов.** Симптом: `qemu-aarch64
   -plugin libinsn.so hello` → `qemu: unknown option 'plugin'`, тогда как
   `qemu-system-arm -help | grep plugin` показывает поддержку. Причина:
   пакетная политика Debian (`qemu-system-arm` PLUGINS ДА, `qemu-user`
   нет). Обойдено: §2.3 (user-mode профиль self-hosted бинарников) взял
   **valgrind callgrind** — «инструментированный прогон», который сам
   §2.3 называл допустимым энейблером (и точнее: пофункциональная
   атрибуция). §4 (system-mode) остался на qemu-system-arm+плагин. Урок:
   выбор «Debian qemu + плагин» покрывает только system-mode; user-mode
   insn-плагин требует сборки qemu из исходника — если он реально нужен,
   это отдельное решение (30-мин сборка), а не побочный эффект.

2. **Вывод insn-плагина в system-mode молчит без `-d plugin`.** Симптом:
   прошивка отработала (serial корректен, rc=0), но `total insns:` нет ни
   в stdout, ни в stderr. Причина: `qemu_plugin_outs()` пишет в qemu-лог
   категории `plugin`, которая по умолчанию выключена. Исправлено: `-d
   plugin -D <файл>` (лог в файл). Урок: system-mode плагин требует и
   `-d plugin`, и `-D`; в user-mode этого нет — ещё один аргумент против
   молчаливой эквивалентности режимов.

3. **valgrind даёт клиенту 16 МБ стека и игнорирует `ulimit -s`.**
   Симптом: `SelfLex` под callgrind прошёл, `SelfParse` — SIGSEGV
   «Access not within mapped region … main; main thread stack size used
   … 16777216». Причина: пулы фаз живут ОДНИМ кадром `main` (~85 МБ,
   фаза 5), а valgrind по умолчанию клиенту 16 МБ (не наследует `ulimit
   -s 262144`). Исправлено: `--main-stacksize=268435456` (и
   `--max-stackframe=…` от эвристики). Урок: EATLang-бинарники с большими
   пуловыми кадрами под valgrind требуют явного `--main-stacksize`,
   иначе ложный «stack overflow».

4. **`contrib/plugins/insn.c` переехал в `tests/tcg/plugins/insn.c`.**
   Симптом: сборка образа падала `curl 404` на исходнике плагина.
   Причина: в qemu v10.0 счётчик инструкций лежит в `tests/tcg/plugins/`,
   а не `contrib/plugins/` (там `execlog`/`hotblocks`, но не `insn`).
   Исправлено: пиннутый URL под верный путь тега `v10.0.0`. Урок:
   проверять путь исходника плагина под конкретную серию qemu.

5. **Профиль засеял общий `build/` Linux-ELF (та же грабля, что у LSP
   §4).** Симптом: `make measure_selfir` над живым деревом положил в
   `build/Self*` aarch64-Linux ELF — на хосте они бы дали ложный красный
   гейт. Исправлено: (а) удалил артефакты; (б) измерение теперь пишет в
   **отдельный `build/measure/`** (цель `measure_selfir`), `build/Self*`
   гейта не трогает. Урок: контейнерное измерение изолирует свой вывод от
   каталога гейта — либо чистый worktree, либо отдельный подкаталог
   `build/`.

6. **Параллельный воркер `git add -a` смёл чужой WIP Makefile в свой
   коммит.** Симптом: `git diff Makefile` показал лишь крошечный хвост
   правки, хотя measure-секцию я добавлял целиком; она уже в HEAD под
   LSP-коммитом соседа (52827b3), а скрипты `containers/measure_*.sh`
   (untracked) — нет → HEAD ссылается на несуществующие файлы (для гейта
   безвредно: `measure` не в гейте). Причина: сосед закоммитил
   `git add -a`/`-A`, а не свои файлы точечно (WORKFLOW §6 «коммиты мимо
   параллельных работ»). Обойдено: **историю соседа не переписываю**; мой
   коммит достраивает фичу (скрипты + фикс `build/measure/`). Урок:
   параллельные задачи должны стейджить свои пути (`git add <files>` /
   apply --cached), не `-a`; читателю чужого коммита — сверять `--stat` с
   ожидаемым составом, случайно попавший файл достраивать, а не откатывать.

## 2026-07-19 · LSP_PLAN · этап 4 (completion)

1. **Linux-бинарники в общем `build/` от контейнерного прогона — ложный
   красный гейт на хосте.** Симптом: `make verify_selfhost` падает
   `./build/SelfLex: Permission denied` на каждом файле. Причина: в
   `build/` лежали `SelfIr/SelfLex/SelfParse/SelfSig/SelfTyped` — ELF
   aarch64 **Linux** (mode `rw-------`), артефакты параллельной
   контейнерной задачи, смонтировавшей это дерево; их mtime свежее
   исходников, хостовый make не пересобирал, а исполнить Linux-ELF на
   macOS нельзя. Не мой дифф (атрибуция по `file build/SelfLex`).
   Исправлено: пять ELF-артефактов удалены (производные, make пересобрал
   нативные), гейт перегнан. Урок: контейнер обязан монтировать **чистый
   worktree**, не живое дерево (CONTAINERS.md); на хосте «Permission
   denied» на артефакте `build/` — сперва `file`: чужая архитектура =
   загрязнение каталога, а не поломка кода.

2. **«Прогал» восстанавливает ровно одну строку — вторая сломанная строка
   файла оставляет completion без разбора.** Симптом (черновой
   транскрипт): запрос за `self.` вернул `null`, хотя владелец очевиден.
   Причина: в тестовом файле было ДВЕ строки с висящей точкой (`self.` и
   `p.`); прогал (замена строки курсора пробелами перед повторным лексом)
   чинит только строку запроса — вторая ломала разбор целиком. Исправлено:
   кейсы транскрипта разнесены по файлам (по одной сломанной строке);
   поведение «прогал не спас → честный `null`» закреплено кейсом id 42.
   Урок: восстановление после ошибок у fail-fast-парсера здесь ровно
   однострочное — это предел механизма, не баг; несколько незавершённых
   правок разом дадут null-completion до исправления текста.

## 2026-07-19 · LSP_PLAN · этап 3 (hover)

1. **Строки языка ограничены ёмкостью 256 — тело hover-markdown длиннее не
   собрать одной строкой.** Симптом: `editor/lsp/Hover.eat` не
   компилировался — `ёмкость строки 512 вне (0, 256]` на поле
   `body: str<512>` (учительский разбор ⚠ + тип + сигнатура в одном
   markdown-значении легко переваливают 256 байт, тем более на кириллице
   ×2 байта/символ). Причина: `str<N>` в EATLang — тип фиксированной
   ёмкости с потолком **256** (SPEC §6), а не растущий буфер. Исправлено:
   значение hover собирается из СЕГМЕНТОВ ≤ 256 (`body/body2/body3`,
   разделитель `\n\n`), которые эмитятся ПОДРЯД прямо в `JOut`
   (`emit_hover_value` + помощник `lsp_qesc` — экранированная дописка
   str<256> без кавычек); ни одна отдельная строка потолок не превышает,
   а склеенное JSON-значение произвольно длинное. Учительский текст тоже
   разбит: заглавие обязательства (`lsp_obl_head`) и разбор (`lsp_teach`)
   — разные сегменты. Урок: markdown/текст произвольной длины на EATLang
   строится ПОТОКОВОЙ эмиссией в билдер (как HTTP `Body`/дампы), а не
   аккумуляцией в один `str<N>`; поле-строку под потолок 256 закладывать
   сразу, длинное — резать на сегменты.

2. **Предел 6 параметров функции включает `self`.** Симптом:
   `emit_hover(self, di, id, h, lz, cs, ce)` — `capacity exceeded:
   параметров функции (предел 6)`. Причина: `self` считается параметром,
   7 суммарно > 6. Исправлено: перенёс вычисление позиций диапазона
   (`lz/cs/ce`) ВНУТРЬ `emit_hover` из `h.line/col/endcol` — сигнатура
   `emit_hover(self, di, id, h)` = 4. Урок: у метода бюджет параметров —
   пять сверх `self`; лишнее упаковывать в передаваемую структуру или
   вычислять на месте.

3. **`extend Check`/`extend Verify` в файле сервера — рендер по `self` без
   копии огромной структуры, паритет не задет.** Не сбой, а решение:
   `Check`/`Verify` весят мегабайты (пулы токенов/узлов/типов), передать
   их в свободный помощник ПО ЗНАЧЕНИЮ = взрыв стека. Методы-рендеры
   (`lsp_type_str`, `lsp_sig_str`, `lsp_hover`) объявлены `extend Check`/
   `extend Verify` в `editor/lsp/Hover.eat` и работают по `self` (без
   копии). Файл в списках `SELFHOST_*` отсутствует → эти методы
   компилируются ТОЛЬКО в сборке LSP, а не в эталонной — `eatc sig`/дампы
   фаз их не видят, паритет `src/eatc ↔ selfhost` цел. Урок: чтобы читать
   пулы фазы из приложения-надстройки без копии и без правки фаз — расширь
   тип фазы блоком `extend` в файле СВОЕЙ склейки (не в `selfhost/`);
   паритетные гейты компилируют только свой фиксированный список.

## 2026-07-19 · LSP_PLAN · живой запуск в VS Code

1. **stdout нативного сервера блочно буферизуется на пайпе — ответ
   `initialize` не доходил, VS Code рестартовал сервер по кругу.**
   Симптом: в `build/lsp.log` — только повторные ``make: `build/eat-lsp'
   is up to date.`` (по строке на каждый рестарт), в Output — маркер
   serve.sh; сервер «жив», но клиент не работает. Причина: аксиома
   `read_byte` (runtime.c) сбрасывает stdout перед блокирующим чтением
   только при tty-эвристике (`isatty(stdin) || isatty(stdout)`); под
   VS Code оба конца — пайпы → флаш не срабатывал, ответ рукопожатия
   навсегда оставался в буфере libc. Батч-сверки (`verify_lsp`) зелёные,
   потому что при выходе процесса буфер сбрасывается — маскировка.
   Исправлено: явный флаг окружения `EAT_INTERACTIVE=1` расширяет
   эвристику интерактивности (runtime.c), ставит его `serve.sh`;
   `build/eat-lsp` зависит и от `runtime.c` (Makefile). Урок: батч-тест
   с EOF не доказывает интерактивность — проверяй ответ при ЖИВОМ
   (незакрытом) stdin; диалог по пайпам объявляет себя явно, tty-детект
   его не видит.

## 2026-07-18 · LSP_PLAN · этапы 1a–1c (go-to-definition + резолв импортов)

1. **`#module` СБРАСЫВАЕТ счётчик строк — offset-ремап был вреден.**
   Симптом (этап 1c): реальная типовая ошибка в модульном файле не
   подчёркивалась. Причина: синтетический поток `[Rt + deps + файл]`
   подавался лексеру, и я вычитал `offset` (строк до файла) из позиции
   ошибки — но директива `#module` в лексере сбрасывает нумерацию строк
   на каждый модуль, так что позиция УЖЕ локальна файлу; вычитание
   уводило `el` в отрицательную зону → `el <= off` → диагностика
   подавлялась как «ошибка в зависимости». Исправлено: удалил ремап,
   `el/ec` из фаз берутся напрямую. Урок: перед ремапом позиций проверь,
   как источник нумерует строки в мультимодульном потоке — здесь
   per-module, не сквозная.

2. **Библиотечный файл без `main` → ложная «нет функции main» на КАЖДОМ
   lib-модуле.** Симптом: валидный `lib/http/Server.eat` под модульным
   анализом краснел «нет функции main — точки входа». Причина: `Check`
   требует `main` (точку входа программы), но библиотечный модуль его
   законно не имеет. Исправлено: обобщил подавление
   (`lsp_suppress_err`) на префиксы «import:» И «нет функции main». Урок:
   LSP анализирует файл как программу, а большинство `.eat` — библиотеки;
   program-only ошибки (нет main) — шум, подавлять.

3. **Пустой путь `#module ""` лексер отвергает.** Симптом: «пустой путь
   в директиве #module» на маркере открытого файла. Причина: открытый
   файл получал `#module ""`. Исправлено: заглушка-путь
   `#module "__open__.eat"` (имя не важно — файл никто не импортирует).
   Побочно подтвердилось: escape `\"` в строковом литерале EATLang
   работает (`"#module \"__open__.eat\""`).

4. **Библиотека (без main) не тайпчекалась — Check бросает «нет функции
   main» ДО проверки тел.** Симптом: типовая ошибка в теле функции
   `lib/http/Server.eat` не подчёркивалась (подавлялась как «нет main»).
   Причина: точка входа проверяется раньше тел → fail-fast, тела не
   доходят. Исправлено: если в открытом файле нет `func main` (подстрока,
   `has_main`), сервер дописывает в поток синтетическую `func main() {{}}`
   → «нет main» проходит, тела проверяются. Урок: чтобы получить
   диагностику ТЕЛ библиотечного модуля через program-only чекер, добавь
   фиктивную точку входа.

5. **`{}` в строковом литерале — пустая интерполяция.** Симптом: `"func
   main() {}"` → «пустая интерполяция {} в строке». Причина: `{` в строке
   начинает интерполяцию. Исправлено: литерал скобок — `{{` и `}}`
   (`"func main() {{}}"`). Урок: литеральные фигурные скобки в строках
   EATLang удваиваются.

6. **eprint компилятора засорял Output панель VS Code.** Симптом:
   пользователь видел `err: import:` в Output и читал как провал (хотя
   подчёркивания корректны — одиночный проход подавляется, модульный даёт
   верный ответ). Причина: stderr сервера (eprint фаз) шёл в LSP Output.
   Исправлено: `serve.sh` уводит stderr сервера в `build/lsp.log`
   (`exec … 2> build/lsp.log`), маркер serve.sh печатается ДО exec и
   остаётся в панели. Урок: диагностический stderr языкового сервера —
   не в Output клиента; подчёркивания идут по stdout (JSON-RPC).

## 2026-07-18 · LSP_PLAN · этап 1 (живые диагностики)

1. **Модульный `lib/json` и конкатенационные фазы `selfhost/` не
   сочетаются в одной программе.** Симптом: склейка `build … Rt LIB_FRONT
   lib/json/Json.eat <фазы> Main` (как задумано §6 плана) падает
   `import из "lib/fmt/Hex.eat": модуль не встретился раньше по потоку`.
   Причина: два несовместимых режима линковки. (а) Модульный драйвер —
   один главный файл с import-блоками, драйвер строит DAG, вставляет
   `#module`-директивы, символы ходят через export/import. (б) Плоская
   конкатенация — список файлов, всё в модуле 0, имена глобальны, но
   import-блок запрещён (проверка требует предшествующей `#module`-метки,
   которой в склейке нет). Фазы `selfhost/` — режим (б) без экспортов;
   `lib/json` — режим (а) с импортами `hex_val/is_digit/digit_value`.
   Модуляризовать фазы нельзя — сломает конкатенационные гейты
   `SELFHOST_*`. Исправлено: плоская склейка (модуль 0) + производный
   `build/JsonFlat.eat` — `Json.eat` со снятыми import-блоками
   (`tools/json_flat.py`); их символы уже глобальны из `LIB_FRONT`
   (Hex/Ascii). Все `editor/lsp/*` — без import/export. Урок: чтобы
   переиспользовать модульный `lib/*` в плоской склейке, снять его
   import-блоки на сборке (зависимости обязаны быть глобальны в потоке);
   два режима линковки исключают друг друга — файл с import'ами не
   конкатенируется, файл без экспортов не импортируется.

2. **Колонки компилятора — в КОДПОЙНТАХ, не байтах; конверсия в UTF-16
   промахнулась.** Симптом: на строке с `абв` перед ошибкой LSP выдал
   `character: 14` вместо `16`. Причина: предположил, что `line:col`
   компилятора — байтовые; на деле лексер двигает колонку по кодпойнтам
   (continuation-байты 0x80..0xBF колонку не двигают — `Lexer.eat`
   §колонки). `u16_upto` шёл (col-1) БАЙТ. Исправлено: идти (col-1)
   КОДПОЙНТОВ, единиц UTF-16 = 1 на кодпойнт BMP (ведущий < 0xF0), 2 на
   supplementary (ведущий ≥ 0xF0, суррогатная пара). Урок: конверсия
   позиций обязана знать единицу колонки источника; здесь кодпойнты →
   UTF-16 совпадает всюду, кроме 4-байтовых UTF-8 (эмодзи и т. п.).

3. **Свободная `send` затенила верхнеуровневую `send` парсера (правило
   6).** Симптом: склейка падает `имя send затеняет объявление верхнего
   уровня` (ParserExpr.eat). Причина: в плоском модуле 0 свободные
   функции LSP делят глобальное пространство со ВСЕМИ фазами. Исправлено:
   префикс `lsp_send`. Урок: в плоской склейке именовать свободные
   функции приложения с префиксом (`lsp_*`) — иначе коллизия с фазами.

## 2026-07-18 · OPTIMIZATIONS_PLAN · §7.6 кандидат 2, этап 3 (сквозной замер, закрытие)

1. **Глобальный вырез `alwaysinline` из `.ll` рушит бинарь компилятора —
   строка-литерал в эмиттере.** Симптом: A/B «атрибут / без» для
   компилятора-на-себе строил «без хинтов» вариант через `sed
   's/ alwaysinline//g'` по `.ll`; clang падал «constant expression type
   mismatch: got type `[0 x i8]` but expected `[13 x i8]`». Причина:
   исходник эмиттера (`Ir/IrStmt.eat`) **печатает** литерал `"
   alwaysinline"` — при компиляции SelfIr этот литерал живёт в `.ll` как
   константа `{i32 13, [13 x i8] c" alwaysinline"}`; глобальная замена
   срезала и её → `[0 x i8]`. Микробенчи корпуса (arith/u128/…) IR не
   эмитят, литерала не содержат — там глобальный replace безвреден, потому
   и всплыло только на компиляторе. Исправлено: вырез **только** на
