Три связанные инициативы пользователя (2026-07-16): параллельные
агенты без столкновений, ранжирование задач под модели/effort,
дробление работ под бюджет контекста. Прорабатываются вместе —
у них общий предмет: **кто, чем и каким инструментом работает
одновременно**.

## 1. Параллельные агенты без столкновений

Наблюдение из практики: за одну сессию трека верификатора параллельно
легли 3 чужих коммита (JSON, МК, интерпретатор) — столкновений не было
только потому, что поверхности не пересеклись (у верификатора заняты
`Verify.eat`/`Check.eat`, у соседей — `lib/Json.eat`, `mcu/`,
`interpreter.py`).

Проработать:

- **Единица владения** — файл или трек? Предложение: трек объявляет
  занятые файлы в шапке своего плана («рабочие поверхности»), агент
  перед стартом сверяется с планами со статусом 🚧.
- **Общие файлы** (Makefile, SPEC.md, AGENTS.md, README планов) —
  очередь правок: правка откладывается и передаётся в промт следующего
  этапа, если файл занят (так уже сделано с README планов).
- **Изоляция**: git worktree на трек — стоит ли; как тогда гонять
  полный гейт (он требует целого дерева).
- **Гейты и параллельность**: полный гейт готовности проверяет ВСЁ
  дерево — чужие незакоммиченные правки могут уронить чужим кодом.
  Конвенция: гейт перед коммитом гоняется на чистом дереве либо
  красный чужой файл фиксируется в FAULTS и обходится.

## 2. Ранжирование задач под модели и effort

Цель — экономия токенов: не жечь максимальную модель на механике.
Доступные ступени (по возрастанию цены): Haiku 4.5 → Sonnet 5 →
Opus 4.8 → Fable 5; у каждой — уровни усилия (effort).

Проработать матрицу «класс задачи → модель/effort». Черновик:

| Класс задачи | Примеры из практики | Ступень |
| --- | --- | --- |
| Механическая правка по образцу | подъём пределов пулов, регенерация литералов, переименования | Haiku/Sonnet, низкий effort |
| Прогон и сверка гейтов | make-циклы, диффы дампов, сбор метрик | Haiku/Sonnet, низкий |
| Порт по эталону с известным механизмом | зеркалирование функции verifier.py в Verify.eat по готовому рецепту | Sonnet/Opus, средний |
| Дебаг паритета (поиск механизма) | бисекция расхождения дампов, минимальные репро | Opus/Fable, высокий |
| Архитектура, инварианты, правило 1 | разрыв циклов графа вызовов, дизайн пулов, пересмотр пределов | Fable, высокий |
| Ревью чужого этапа | проверка коммита против плана и гейтов | Sonnet/Opus, средний |

Открытые вопросы: кто назначает ступень (пользователь в промте этапа /
агент-диспетчер); как эскалировать (дешёвая модель упёрлась → передать
выше с накопленным контекстом — формат передачи).

## 3. Дробление этапов под бюджет контекста

Наблюдение: этап 4 верификатора съел длинную сессию с многократной
компрессией контекста; работоспособность сохранилась благодаря
файлам-артефактам (план, дампы в /tmp, минимальные репро).

Проработать:

- **Правило размера этапа**: этап формулируется так, чтобы его промт +
  нужные файлы + итерации влезали в один сеанс без компрессии;
  ориентир — один механизм/одна поверхность на этап.
- **Артефакты между этапами**: всё, что нужно следующему агенту, —
  в репо (план, FAULTS, «→ следующий шаг»), не в памяти сессии;
  промт этапа самодостаточен (правило WORKFLOW §3.7).
- **Длинные прогоны** — в фон, с полезной работой в промежутке;
  дорогие сверки (полный гейт) — один раз в конце этапа, не после
  каждой правки.
- **Когда дробить нельзя** (механизм размазан по файлу, как машина
  цикла в Verify.eat) — этап получает меньший объём других работ.

## 4. Определение готовности

План процесса «готов», когда: (а) матрица §2 согласована с
пользователем и вписана в WORKFLOW.md; (б) конвенция владения
файлами §1 вписана в WORKFLOW.md и шаблон плана (поле «рабочие
поверхности»); (в) правило размера этапа §3 вписано в WORKFLOW.md;
(г) по ним отработано ≥ 2 реальных этапа без столкновений — итоги
зафиксированы в FAULTS/итогах этапов.
