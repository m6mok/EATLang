## 1. Итог одной строкой

Сделать «ссылку в пуле» первоклассным типом `idx<N>?` — опциональным
индексом с ниша-кодированным сентинелем NONE, который **нельзя
использовать как индекс/аргумент, пока он не сужён** guard'ом
`if x != NONE`/`match`, — превратив класс банковых ссылок из
«недоказуемо интервалами» в «непредставимо без разбора» (усиление
правила 7, не обход).

## 2. Почему язык к этому приспособлен

Тип не добавляет мощности — он **отнимает** её у `u32`: значение
«ссылка на узел» перестаёт быть произвольным числом и становится
«индекс из `[0, N)` **или** пусто». Это ровно дух Power of 10 —
сузить представимое так, чтобы ошибочная программа не компилировалась,
а не ловилась в рантайме. Три опоры уже в языке:

- **Сентинель как приём — прецедент §7.** `socket_accept(fd) -> u32`
  возвращает `0xFFFFFFFF` (`NO_CONN`) — «никто не ждёт»
  ([SPEC.md:375](../../SPEC.md)); `lib/Const.eat:16` фиксирует
  `NONE = 4294967295` с обоснованием «валидные индексы малы (≤ 65536),
  максимум u32 не совпадает ни с одним». Домен индексов и сентинель
  **не пересекаются** (§6 SPEC: массив ≤ 65 536, узлы AST ≤ 131 072 —
  оба заведомо меньше `2³²−1`).
- **Дырочный домен верификатора — готовая модель.** `State.holes`
  (`src/eatc/verifier.py:222`) хранит «путь → (плотная часть, сентинель)»
  = ровно `[0, N) ∪ {NONE}`; guard `!= NONE` схлопывает дырку к плотной
  части и разблокирует снятие bounds (`_refine_neq`, `verifier.py:1388`;
  зеркало `collapse_hole`, `selfhost/VerifyExpr.eat:2217`). Верификатор
  **не ведёт своей решётки типов** — тип он проецирует в интервал одной
  функцией `_ty_iv` (`verifier.py:84`). Значит новый тип не заводит
  параллельный механизм, а **проецируется в уже существующий**
  (плотный диапазон + флаг сентинеля).
- **Разбор перед использованием — уже правило 7.** `match` обязан быть
  исчерпывающим, enum с нагрузкой нельзя `==`/`!=` (только `match`) —
  [SPEC.md:246-247](../../SPEC.md). «Нельзя использовать оптиндекс без
  narrowing» — та же дисциплина, продлённая на индексы.

Где язык к этому **не** приспособлен: флоу-типинга (сужение типа
переменной по `if`-условию) в тайпчекере сегодня **нет ни в каком
виде** — единственное сужение это связывание payload в arm `match`
(`typechecker.py:1096`). Это и есть ядро новой машинерии и главный
ценник (§11).

## 3. Рамка: свойство → правило → потолок

| Свойство | Правило Power of 10 | Потолок (§6 SPEC) |
| --- | --- | --- |
| Ссылка — индекс `[0,N)` или пусто, не произвольный u32 | П.7 (разбор перед использованием; усиление) | `N` из типа (`idx<131072>`, `idx<65536>`, `idx<512>`…) |
| NONE — валидное значение типа, а не «мусор в памяти» | П.3 (нет кучи), инициализатор обязателен (грамматика) | сентинель = `2³²−1`, вне домена индексов |
| Оптиндекс нельзя индексировать без narrowing | П.7, П.10 (ошибка, не warning) | — |
| Запись в слот пула — только `idx<N>` или NONE | П.7 (нельзя «отравить» инвариант) | `N` слота |

## 4. Числа (пределы)

- **`idx<N>`** — плотный индекс, носитель `u32`, инвариант `[0, N)`.
  `N` — константа-потолок пула (те же числа §6: 131072 для AST-узлов,
  65536 для массива, 512 для тип-пула, 64 для кадров). Новых чисел не
  вводит — переиспользует существующие пределы.
- **`idx<N>?`** — ниша-опциональный индекс: `[0, N) ∪ {NONE}`, носитель
  тот же `u32`, `NONE = 2³²−1` в нише. Ровно одно «пустое» значение.
- Представление **однословное** — байт-в-байт как сегодняшний `u32`-слот
  пула. Миграция лейаута пулов = **ноль** (в отличие от тегового
  `Option<u32>` — см. §7).

## 5. Аксиомы ОС

Не нужны. Тип чисто языковой; ввода/вывода не добавляет.

## 6. Архитектура / механизм

### 6.1 Нотация (рекомендация; синтаксис — решение пользователя)

Рекомендуется по образцу существующего `str<N>`:

- `idx<N>` — плотный индекс (kind 15 в тип-пуле `Check.eat`, «коды не
  перенумеровываются» — следующий свободный 15);
- `idx<N>?` — суффикс `?` делает индекс опциональным (либо отдельный
  kind 16, либо флаг на kind 15 + поле `N`; предпочтительно флаг, чтобы
  `idx<N>` и `idx<N>?` делили `N`);
- `NONE` — единственный литерал «пусто», уже существует (`lib/Const.eat`),
  теперь типизируется как `idx<N>?` в контексте ожидания (контекстный
  литерал по образцу `None`, `typechecker.py:1167`).

Семантически **`idx<N>?` ≡ ниша-кодированный `Option<idx<N>>`**: то же
значение «есть/нет», но представление — сентинель в нише носителя, а не
тег-агрегат. Это не второй параллельный sum-мир: тип определяется через
уже имеющуюся семантику Option, разрешённую здесь потому, что носитель
(индекс < N ≤ 65536/131072) оставляет `2³²−1` свободной нишей.

**Отвергнутые нотации:**
- `?u32` — слишком грубо: без `N` верификатор не получает плотный
  диапазон, всё равно нужен `requires`. `N` в типе — то, что делает
  bounds доказуемым «из типа», а не из контракта.
- `Option<idx<N>>` напрямую — Option сегодня **контекстный** (только
  return / `const`-аннотация, `SPEC.md:251`), не может быть типом поля/
  элемента пула; лейаут теговый (двойной размер, `codegen.py:196`);
  narrowing только `match` (теряется идиома in-place `if`). `idx<N>?` —
  это Option, разрешённый как поле и ниша-кодированный.

### 6.2 Правила присваивания / сравнения / каста (звучность §9)

- `idx<N>?  ← NONE` — ок (пустой литерал).
- `idx<N>?  ← idx<N>` — ок (расширение: плотное → опциональное).
- `idx<N>?  ← u32` — **запрещено неявно**; нужен явный сужающий каст
  `idx<N>?(x)`, который либо доказан (`x < N` или `x == NONE`), либо
  эмитит один runtime-trap. **Это структурно гасит «отравление» §1.5**:
  запись неограниченного `u32` в слот пула становится ошибкой типа.
- `idx<N>  ← idx<N>?` — **запрещено** вне narrowing-ветки (нельзя молча
  «снять» опциональность).
- `idx<N> → u32` — **свободное расширение** (плотный индекс читается как
  `u32 ∈ [0,N)`); нужно для банковой арифметики `p/65536`, `p%65536` —
  она остаётся интервально-доказуемой, как сегодня.
- `x == NONE` / `x != NONE` на `idx<N>?` — **разрешено** (ниша-тест) и
  есть **единственный примитив narrowing**. Это не противоречит запрету
  `==`/`!=` на sum-типах с нагрузкой (`SPEC.md:247`): там запрет потому,
  что тег-payload нельзя сравнить осмысленно; здесь NONE — нульарная
  ниша конкретного значения `2³²−1`, сравнение с ней определено (как с
  `NO_CONN`). Различие «теговый Option (только match) ↔ ниша idx?
  (== NONE + match)» надо зафиксировать в SPEC §5.3.
- Арифметика на `idx<N>` напрямую **не определена** (индекс — не число):
  считать → расширить до `u32`, посчитать (trap-контроль как сейчас),
  вернуть сужающим кастом с доказательством. Держит инвариант «идентити
  индекса не портится арифметикой молча».

### 6.3 Narrowing (новая машинерия — ядро фичи)

Две формы, обе сужают **тип переменной** в ветке (не только интервал):

1. **Флоу-narrowing по `if` (рекомендуется как основной).**
   ```
   const c: idx<131072>? = self.nfc[n / 65536][n % 65536]
   if c != NONE {
       # здесь c : idx<131072>  — можно индексировать
       const d: idx<131072>? = self.nns[c / 65536][c % 65536]
   }
   # вне ветки c снова idx<131072>?
   ```
   Тайпчекер, входя в then-ветку `if c != NONE`, переписывает тип `c` в
   `idx<N>` для тела (и симметрично else-ветку `== NONE`), затем
   восстанавливает после блока. Зеркально тому, что `_refine_neq` уже
   делает на интервалах — но теперь на уровне типов, в обоих чекерах.
2. **`match` (переиспользует существующее).** `match c { Some(i) {…}
   None {…} }` — `i : idx<N>` внутри arm. Дороже эргономически (новое
   имя `i`, теряется in-place-идиома), но машинерия готова
   (`check_match`, `typechecker.py:1070`).

Рекомендация: основной — флоу-narrowing по `!= NONE` (совпадает и с
текущей руко-идиомой `if nxt == NONE {} else {}`, и с семантикой
верификатора), `match` — как альтернатива. Обе формы — **решение
пользователя** (семантика).

**Разделение с интервалами (звучность).** Narrowing **типа** ключуется
строго на `== NONE`/`!= NONE`/`match` — и больше ни на чём. Интервальное
сужение по `x < N` верификатор делает как и раньше, но **тип** от `< N`
не сужается. Разведение двух механизмов исключает рассинхрон
«тайпчекер думает dense, верификатор — нет».

## 7. Переход «недоказуемо → непредставимо» на живых сайтах

Сегодняшняя идиома (§1.5) — руко-`assert`, который **можно забыть**;
забыл → молчаливый runtime-trap, который верификатор снять не может.
С типом забыть нельзя — код без narrowing не компилируется.

### 7.1 `Check.unlink_top` ([selfhost/Check.eat:1215](../../selfhost/Check.eat))

Сейчас:
```
func unlink_top(let self, e: u32, prev: u32)
    requires e < 131072
{
    assert prev < 131072                     # load-bearing, можно забыть
    const nxt: u32 = self.p.nns[e/65536][e%65536]
    if nxt == NONE { … } else { assert nxt < 131072; … }
    …
    self.p.nlc[0/65536][0%65536] = prev      # плотность держит assert выше
}
```
С типом (`nns/nlc` — элемент `idx<131072>?`, `prev : idx<131072>`):
```
func unlink_top(let self, e: idx<131072>, prev: idx<131072>)
{
    const nxt: idx<131072>? = self.p.nns[e/65536][e%65536]
    if nxt != NONE { … nxt здесь idx<131072> … }   # else-ветка = NONE
    …
    self.p.nlc[0][0] = prev                   # prev уже idx<131072> — плотно по типу
}
```
`assert prev < 131072` **исчезает** — `prev : idx<131072>` гарантирует
это на границе вызова (вызывающий обязан передать плотный индекс или
сузить). `nns[e]` возвращает `idx<131072>?` — обращение к `nxt` без
`if nxt != NONE` — **ошибка компиляции**. Отравить `nlc` неограниченным
`prev` нельзя: слот принимает только `idx<131072>?`.

### 7.2 `Parser.insert_field` / `append_child`
([selfhost/Parser.eat:221,240](../../selfhost/Parser.eat))

Три вложенных стража `if …== NONE / else { assert < 131072 }` на одну
вставку (`last=nx[s]`, `head=nfc[s]`, `nxt=nns[last]`). С типом каждый
`nx/nfc/nns[...]` — `idx<131072>?`; ветки `if head != NONE { … }`
narrow'ят `head` к `idx<131072>` **и это единственный способ его
использовать**. Записи `nns[f] = head` (в then-ветке `head` уже
плотный) типизируются без assert'а. Кластер из трёх руко-assert'ов →
три обязательных `if`, которые и так уже написаны — но теперь их
**нельзя пропустить**, и «копия дырочного значения одной строкой»
(§1.5, отравление `nns`) непредставима.

### 7.3 `Check.first_child` / `second_child`
([selfhost/Check.eat:615](../../selfhost/Check.eat))

Хелпер-страж §1.5: `const c = nfc[n]; assert c != NONE; return c` с
`ensures result < 131072`. С типом: `nfc[n] : idx<131072>?`, `return`
требует `idx<131072>` → narrowing обязателен; `ensures result < 131072`
исчезает (тип несёт границу). Хелпер остаётся (одна точка «я обещаю
не-NONE»), но `ensures`/`assert` заменяются структурой типа. Стоимость
рантайм-проверки та же (один trap, если верификатор не докажет), но
**структурно принуждена**, а не оставлена на дисциплину.

**Итог перехода:** класс банковых ссылок (151 сайт, §1.5) — из
«недоказуемо, держится руко-assert'ами, регрессирует при отравлении»
в «непредставимо без narrowing, отравление — ошибка типа».

## 8. Что реализуем НЕ

- **НЕ теговый `Option<u32>` в пулах.** Двойной размер (`{i32,i32}`),
  ломает 8-КБ структуры, не сходится с дырочным доменом. Ниша —
  обязательна.
- **НЕ общий флоу-типинг на все типы.** Narrowing вводится **только**
  для `idx<N>?` по `== NONE`/`!= NONE`. `bool`, `Result`, enum — как
  сегодня (match). Держит машинерию узкой.
- **НЕ снятие кадрового класса «nf ≥ 1» (233 сайта).** Это data-flow
  (`f = nf−1` при `nf ≥ 1`), не «индекс-или-NONE»; тип его не берёт
  (§10). Не переоткрывать как задачу этого типа.
- **НЕ пользовательские дженерики.** `idx<N>` — встроенный, как `str<N>`;
  не открывает `Vec<T>`/generic-инфраструктуру (SPEC.md:475 в силе).
- **НЕ арифметика на `idx`.** Считать — через расширение до `u32`.

## 9. Звучность против Power of 10 (риск незвучности)

- **Правило 3 (нет кучи):** тип — скаляр `u32`, размер в типе. ✓ не
  ослабляет.
- **Инициализация:** NONE — валидное значение типа, не «дырка в памяти»;
  инициализатор по-прежнему обязателен грамматикой. ✓ усиливает (сегодня
  сентинель — магическая библиотечная константа, станет типизированным
  литералом).
- **Правило 7 (разбор перед использованием):** narrowing-принуждение —
  **усиление**, не новое правило (по образцу исчерпывающего match). ✓
- **Правило 10 (ошибка, не warning):** использование оптиндекса без
  narrowing — ошибка компиляции. ✓
- **Тотальность:** bounded, без рекурсии/цикла-без-границы. ✓

**Точки риска незвучности (что доказать при вводе):**
1. **Каст `u32 → idx<N>`** не должен впустить значение `∈ [N, 2³²−1)`
   (между доменом и сентинелем): либо доказано `< N`, либо `== NONE`,
   иначе trap. Дырка «протёк大 индекс» = баг верификатора.
2. **Флоу-narrowing тайпчекера обязан совпасть с интервальным narrowing
   верификатора** байт-в-байт по вердикту (оба схлопывают на `!= NONE`).
   Рассинхрон = либо ложное снятие проверки (незвучно!), либо лишний
   trap. Держать: тип-narrowing только на NONE-тест, ничего сверх.
3. **`idx<N>?` в поле struct-литерала / возврате / hive-сводке** — тип
   должен нести дырку **с объявления** (параметр `prev: idx<131072>?`
   входит в функцию уже с `holes[prev] = ((0,131071), NONE)`), а не
   только при чтении из пула. Это новая точка в `_ty_iv`/`nty→interval`
   обоих верификаторов.
4. **Bootstrap-фикспойнт:** компилятор пересобирает сам себя, его
   собственные пулы мигрируют в `idx<N>?` — сходимость обязана
   держаться.

Обхода Power of 10 дизайн **не открывает**: ни кучи, ни рекурсии, ни
цикла без границы, ни глобала, ни молча отброшенного результата. Это
**ужесточение** (сужение представимого), а не ослабление. Легитимный
путь ввода — явная правка SPEC §2/§5.3/§6 + POWER_OF_10.md (как
прецеденты `loop`/`break`/`exit`), а не побочный эффект.

## 10. Замер потолка класса (сколько из 539 берёт)

База — самоприменение фронтенда, `eatc verify -O`, остаток после §1.5:
**539 runtime-bounds = 233 кадровых + 151 банковых + 155 прочих.**

| Класс | Сайтов | Тип берёт? | Почему |
| --- | --- | --- | --- |
| Банковые ссылки `nns/nfc/nlc[…/65536]` | 151 | **Да** | значение = индекс-или-NONE; narrowing снимает bound |
| Тип-пул `ya/yb[t]` (`ty_a`/`ty_b`, <512) | ~20 (в «прочих») | **Да** | тот же индекс-или-NONE, `idx<512>?` |
| Курсоры `qp`/`nfs` (знак-или-NONE) | ~10–15 (в «прочих») | Частично | `qp` — да; `nfs` (знак свёртки) — семантика иная, спорно |
| Кадровые `fp[nf−1]`/`fnod[nf−1]` | 233 | **Нет** | bound из «nf ≥ 1» (data-flow глубины стека), не сентинель |
| Payload-стеки `vs/vm[depth]` | ~60–80 (в «прочих») | **Нет** | тот же data-flow по счётчику глубины |
| Арифметика `i±j`, вложенные `p[q[..]]` | ~50–60 (в «прочих») | **Нет** | нужен реляционный домен, не сентинель |

**Оценка:**
- **Нижняя граница: ~130** — только чистые банковые ссылки, за вычетом
  сайтов, ко-доминированных другими проверками.
- **Верхняя граница: ~190** — все 151 банковых + ~30–40 индекс-или-NONE
  в «прочих» (тип-пул `ya/yb`, курсоры `qp`).
- **Центр ≈ 160 из 539 (≈ 30 %)** остатка runtime-bounds.

**Сайты, которые тип НЕ берёт (называю явно):** весь кадровый класс 233
(`self.fp[f]`/`self.fnod[f]` при `f = nf−1`, [Parser.eat:309,329,417,
637](../../selfhost/Parser.eat)); payload-стеки `vs/vm`; `i±j` и
вложенная арифметика; `frame_node` **по индексу** (`f < 64` — кадровый),
хотя его **значение** `fnod[f]` — банковое (двойственный сайт
[Parser.eat:314](../../selfhost/Parser.eat)).

**Побочная выгода (не в счёте bounds):** тип **устраняет +14
runtime-assert идиомы**, добавленные §1.5, — они становятся уровнем типа.
И, главное, **фиксирует регрессию**: «отравление» инварианта пула
(единичная запись неограниченного линка) становится непредставимым, а не
«вычищенным вручную и способным вернуться».

**Метод (как §1.4/§1.5):** разметка по исходным строкам живых сайтов
(§7, §10), пробные /tmp-эскизы идиомы `idx<N>?` без правки эталона в
HEAD. Точный счёт снимать реализацией этапа 1 на копии — оценка выше
консервативна.

## 11. Ценник внедрения (правка языка, полное зеркало)

Грубая оценка по стадиям (образец — Option/`str<N>`; ниша-лейаут
дешевле тегового):

| Стадия | Python-эталон | selfhost-зеркало | Риск |
| --- | --- | --- | --- |
| Токен `?` + лексер | `tokens.py`/`lexer.py` +10 | `Tok.eat`/`Lexer.eat` +15 | низкий |
| Узел типа AST + парсинг | `ast_nodes.py`+`parser.py` +18 | `Ast.eat`+`Parser.eat` +30 | низкий |
| Тип в системе (kind 15, intern, resolve, show, compatible) | `types.py`+`typechecker.py` +25 | `Check.eat` +40 | средний (интернирование, «коды не перенумеровывать») |

| tests (verify_suite, дампы lex/parse/ir, verify-кейсы ±) | +150 | (общие) | средний |

**Итого ≈ 1150–1350 строк в ~35 файлах, из них ~25 требуют байт-в-байт
зеркала.** Порядок — как крупный axiom-этап (u64/extern/read_span §3.1),
но **больше** из-за narrowing.

**Точки, где зеркало интрикатно (по образцу §3.1 read_span):**
1. **Флоу-narrowing** — новая фаза потока в тайпчекере обоих
   компиляторов, легко разъезжается; вердикт «сужен/не сужен» обязан
   совпасть байт-в-байт с интервальным narrowing верификатора.
2. **Каст `u32→idx<N>`** — идентичные trap-тексты/коды в двух системах
   кодов (Check `xa` и Ir emit-код, как write_span 110/8 §3.1).
3. **typed-дамп `idx<N>?`** и взаимодействие с тип-интерн-пулом (kind
   15/16, лениво как `u16_ty`, `Check.eat:375`).
4. **Дырка с объявления** в `_ty_iv`/`nty→interval` — новая проекция
   типа в дырочный домен в обоих верификаторах.
5. **Bootstrap-фикспойнт** — самопересборка компилятора с
   мигрированными пулами.

## 12. Вердикт разведки (перф ↔ ценник)

- **Дизайн звучен и элегантен:** `idx<N>?` — ниша-Option, **проецируется
  на уже существующий дырочный домен** верификатора (не дублирует его),
  сентинель совпадает с NONE `lib/Const.eat`, narrowing = усиление
  правила 7. Обхода Power of 10 нет — это ужесточение.
- **Потолок класса реалистичен, но невелик:** ~130–190 из 539
  (~30 % остатка bounds). Кадровый класс 233 и половина «прочих» типу
  недоступны в принципе.
- **Ценник несоразмерен счёту:** ~1200 строк, ~35 файлов, самый дорогой
  и рискованный кусок — **флоу-narrowing**, машинерии которого в языке
  нет; плюс bootstrap-риск и полное зеркало. Для сравнения: **§1.5
  сняла 762 → 539 (−223) вообще без правок компилятора**, чистыми
  `.eat`-идиомами. Как **перф-рычаг тип покупает меньше bounds, чем уже
  снято, дороже на порядок.**
- **Настоящая ценность — не счёт, а свойство:** «непредставимо» вместо
  «недоказуемо» фиксирует регрессию отравления, делает идиому
  структурной, снимает +14 assert §1.5 и переносится на `lib/` (пулы
  Json/Http). Это довод **корректности/эргономики**, не скорости.

**Рекомендация:** держать как готовый дизайн (📐). Внедрять **только
если** появляется мотив помимо счёта bounds:
(а) флоу-narrowing нужен языку и по другим причинам (общеполезная фича);
(б) реальный баг из забытого NONE-assert'а или запрос авторов `lib/` на
типобезопасность пулов;
(в) целевая картина selfhost требует «непредставимости» как гарантии.
**Одной скоростью на сегодня не окупается** — согласуется с выводом
§1.4/§1.5 «остаток `-O` у потолка добросовестных средств».

## 13. Этапы (если вводится — решение пользователя)

Порядок паритет-безопасный, каждый — свой гейт (образец
SELFHOST_VERIFIER, 6 этапов):

0. **Синтаксис/семантика — решение пользователя.** Зафиксировать в
   SPEC §2/§3/§5.3/§6 и POWER_OF_10.md: нотацию, narrowing-правило,
   потолок, каталог ошибок GUIDE §12. **До кода.**
1. **Тип `idx<N>` (без опциональности)** в обоих компиляторах: парсинг,
   kind 15, resolve, дамп, кодоген (= u32 с границей), верификатор
   (`_ty_iv` = `[0,N)`). Гейт полный. Даёт кадровым/дозаписям плотный
   тип уже здесь.
2. **`idx<N>?` + литерал NONE + каст** — ниша, дырка с объявления в
   верификаторе. Гейт.
3. **Флоу-narrowing по `!= NONE`/`== NONE`** (+ `match` при желании) —
   самая дорогая фаза, отдельным этапом. Гейт.
4. **Миграция пулов selfhost** (`nfc/nlc/nns` → `idx<131072>?`, `ya/yb`
   → `idx<512>?`), снятие руко-assert'ов §1.5, замер снятых bounds.
   Bootstrap-фикспойнт. Гейт.
5. **Миграция `lib/`** (Json/Http-пулы) — по желанию, отдельный этап.

Каждый этап — worktree при живом соседе, полный гейт 10 целей зелёный
целиком, замер bounds до/после в FINDINGS, итог по WORKFLOW §3.

## 14. Инварианты (какие обходы НЕ открывает)

Ни кучи (скаляр u32), ни рекурсии, ни цикла без границы, ни мутабельного
глобала, ни молча отброшенного результата. Narrowing-принуждение —
усиление правила 7. Новых чисел §6 не вводит (переиспользует пределы
пулов). Сентинель не пересекается с доменом индексов (§4).

## 16. Открытые вопросы (решение пользователя до кода)

1. **Нотация:** `idx<N>?` (рекомендация) vs `?idx<N>` vs `idx<N>` +
   отдельный опциональный тип.
2. **Narrowing:** флоу по `if != NONE` (рекомендация) vs только `match`
   vs обе формы.
3. **`idx<N>` без `?`** как отдельный публичный тип (плотный индекс) —
   вводить сразу (этап 1) или только опциональный `idx<N>?`.
4. **Вводить ли вообще** — вердикт §12: перфом не окупается; ждём мотива
   корректности/эргономики или общеязыковой потребности во
   флоу-narrowing.
