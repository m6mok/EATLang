- **Блокирует:** —

Дизайн-документ до кода. Проблема — из
[TRACKS.md](TRACKS.md) трек 1: строка — фиксированный агрегат
`{i32, [256 x i8]}` (260 байт), и каждое связывание/передача — memcpy
260 байт независимо от фактической длины. StrBench ~88 M оп/с против
820 M у арифметики; `Rt.write(self)` копирует 260 байт, чтобы вывести 5.

## 0. Как устроено сейчас (карта фактов)

Проверено по коду 2026-07-13; паритет бутстрапа и selfhost дословный.

**Представление.** `STR_CAP = 256`, `STR_LL = {i32, [256 x i8]}` —
`src/eatc/codegen.py:34-40`; в selfhost — `Ir.ty_scalar`
(`selfhost/Ir.eat:687`) и глобалы литералов (`selfhost/Ir.eat:4814`).
Константы 260 в коде нет: размер memcpy берётся трюком `gep(null,1)`
(`codegen.py:251-254`, зеркало — `Ir.e_mcpy`, `selfhost/Ir.eat:1468`).

**Соглашение вызова.** Все агрегаты уже передаются **по указателю**
(`declare_func`, `codegen.py:319-321`; `e_ucall`,
`selfhost/Ir.eat:2092-2096`). Вызывающий копий не делает. Копию
(defensive copy, `materialize` = alloca + memcpy) делает **вызываемый**:

- каждый агрегатный value-параметр — `codegen.py:388-396`,
  `Ir.gen_param` (`selfhost/Ir.eat:4502`);
- получатель `self` без `let` — `codegen.py:379-386`,
  `Ir.gen_prologue` (`selfhost/Ir.eat:4480`); `let self` идёт по
  указателю вызывающего без копии;
- возврат агрегата — sret-слот вызывающего (out-параметр + `void`),
  `codegen.py:314-326`.

**Копии связывания.** `const` — `materialize` (`codegen.py:449-453`),
присваивание — `copy_into`/memcpy (`codegen.py:455-458`); зеркала —
`Ir.copy_into`/`Ir.materialize` (`selfhost/Ir.eat:2106, 2146`).

**Атрибуты (коммит 311f8fc).** sret: `noalias nofree nonnull`;
`self` и агрегатные параметры: `nofree nonnull` — без `noalias`
(комментарий `codegen.py:334-336`: получатель `let self` может
совпасть с аргументом, `s.append_str(s)`) и без `readonly`.
Зеркало — `Ir.p_args` (`selfhost/Ir.eat:4664`).

**Интерпретатор.** Строка — неизменяемый Python `str`; `_copy_value`
для неё no-op (`interpreter.py:82-99`). Наблюдаемая семантика копий
задаётся только правилом «не-`let` параметр неизменяем» — сами копии
в интерпретаторе нематериальны.

**Алиасинг сегодня.** Явного запрета нет ни в typechecker, ни в
checks; `s.append_str(s)` корректен ровно потому, что вызываемый
копирует `o` до мутации `self`.

**Где болит (Rt.eat).** Не-`let` получатели — 7 методов: `write`,
`print`, `eq`, `p_lo`, `p_hi`, `parse_status`, `parse_value`
(копия 260 байт на каждый вызов). Value-параметры `RtStr` — два:
`append_str(o)`, `eq(o)`. Каждый сегмент интерполяции — вызов
`append_str` с копией 260 байт (`gen_strlit`, `codegen.py:719-740`).
В остальном selfhost — ~17 функций с параметрами `str<N>`
(fail/lit-хелперы).

## 1. Семантический инвариант, который нельзя потерять

SPEC §5.4: параметры всегда by-value, `let` разрешён только у `self`,
«других способов разделить состояние нет, поэтому алиасы невозможны».

Из этого следует ключевое наблюдение, на котором стоит весь план:
**единственный канал записи в чужую память — `let self`**. Не-`let`
параметр неизменяем (typechecker), стать получателем мутирующего
метода не может (получатель обязан быть `let`-lvalue). Значит,
опасное перекрытие возможно только внутри одного вызова метода с
`let self`: получатель и агрегатный аргумент указывают на одну память.

## 2. Направление A — read-only параметры по ссылке (рекомендую)

**Идея.** Снять defensive copy у вызываемого: не-`let` `self` и
агрегатные value-параметры связываются прямо по входному указателю.
Наблюдаемая by-value-семантика сохраняется точечной страховкой на
стороне вызывающего.

**Правило перекрытия (на месте вызова).** Только для вызова метода с
`let self`: если корень lvalue-пути агрегатного аргумента совпадает с
корнем пути получателя (`s` и `s`; `s` и `s.f`; `arr[i]` и `arr[j]` —
индексы не сравниваем, консервативно), вызывающий материализует копию
аргумента и передаёт её. Все прочие аргументы — литералы-глобалы,
sret-результаты вызовов, чужие корни — заведомо не перекрываются.
Дизъюнктность `self` ↔ параметры внутри функции держится по индукции:
каждый внешний вызов её обеспечил.

**Что это даёт.**

- Исчезают все 260-байтные копии у 7 не-`let` методов Rt.eat и обеих
  точек `RtStr`-параметров; интерполяция перестаёт копировать сегменты
  (главный вклад в StrBench). То же для всех структур/массивов —
  выигрыш не только строковый (fail-пути, `Tok.same`, лит-хелперы
  эмиттера на горячем пути SelfIr).
- Открываются атрибуты: `readonly` на не-`let` `self` и всех
  агрегатных параметрах; `noalias` становится корректным (записывает
  только `let self`, а его дизъюнктность гарантирует правило
  перекрытия); плюс `dereferenceable(N)` и `align 4`.

**Цена и риски.**

- Правка обоих кодогенов (gen_param/gen_prologue/emit_call + атрибуты)
  и typechecker/checks (пометка «нужна caller-копия» в typed AST);
  интерпретатор не меняется (семантика не наблюдаемо та же).
- SPEC §5.4 — зафиксировать: параметры семантически by-value; копия
  элидируется, когда компилятор доказал отсутствие перекрытия;
  перекрытие с `let self`-получателем покрывается копией на месте
  вызова. Наблюдаемое поведение программ не меняется.
- IR меняется везде → регенерация золотых дампов `tests/ir/`,
  полный паритет и фикспойкт — по чеклисту MODIFYING.md.
- Новые кейсы `tests/verify/`: `s.append_str(s)`, `s.eq(s)`,
  перекрытие через массив/поле, передача литерала.

**Альтернатива внутри A (решение пользователя):** вместо тихой
caller-копии сделать перекрытие **ошибкой компиляции** («аргумент
перекрывается с let self-получателем»). Честнее по духу Power of 10,
но это видимое изменение языка: `s = "{s}ab"` (интерполяция с самим
собой) — легальный и частый паттерн, его пришлось бы специально
разрешать. Рекомендую тихую копию: семантика не меняется вовсе.

## 3. Направление B — копия по фактической длине (второй слой)

**Идея.** В `copy_into` для типа-строки копировать `4 + ln` байт
(длина загружается в рантайме) вместо константных 260. После A
остаются копии `const`/присваиваний, sret-возвратов и caller-копий
перекрытия — их и ужимает B.

**Trade-off.**

- Размер memcpy перестаёт быть константой — LLVM теряет возможность
  раскроить/убрать копию; для коротких строк всё равно выигрыш
  (7 байт вместо 260), для длинных — паритет.
- Хвост буфера у копии остаётся неинициализированным. Сегодня ни одна
  операция Rt.eat не читает `buf` за `ln` — это надо зафиксировать как
  инвариант Rt.eat (и не сравнивать строки memcmp'ом всего буфера
  впредь); литералы-глобалы остаются добитыми нулями.
- Применимо только когда тип — ровно `str`; строка внутри struct
  копируется целым memcpy структуры, как сейчас.
- Семантика не меняется, SPEC не трогается; меняются оба кодогена
  (одна ветка в `copy_into`) + дампы.

Делать после замера остатка от A: возможно, `const`-копии уже не видны
в профиле, и B не окупает потерю константного размера.

## 4. Направление C — настоящий `str<N>` (отложить)

SPEC §6 уже допускает `N ≤ 4096`, но рантайм фиксирован на 256:
layout `RtStr` (`selfhost/Rt.eat:15`), `STR_CAP`, глобалы литералов.
Честная параметризация тянет либо мономорфизацию методов `RtStr` по
N, либо переход на пару (указатель, ёмкость) в рантайме — отдельная
большая тема с решениями по layout. К производительности текущего
кода почти не относится (сегодняшняя цена — копии, а не ёмкость).
Предлагаю отложить до появления потребителя (крипто/МК: SHA-хексы
влезают в 256; МК скорее захочет N < 256, что упирается в тот же
layout).

## 6. Решения пользователя (приняты 2026-07-13)

1. Направление A одобрено; SPEC §5.4 фиксирует элизию копии при
   доказанном отсутствии перекрытия, наблюдаемая by-value-семантика
   неизменна.
2. Перекрытие с `let self`-получателем — **тихая caller-копия**
   (не ошибка компиляции).
3. Атрибуты после A: **`readonly` + `noalias`** на агрегатных
   параметрах и не-`let` `self`.
4. B — **по результатам замера** после A.
5. C — отложено до появления потребителя.
