# План проекта: криптосюита на EATLang (`examples/crypto/`)

- **Статус:** 📋 ЗАПЛАНИРОВАН (готовое ТЗ, реализация не начата)
- **Роль:** активный план — ТЗ примера `examples/crypto/`
- **Правки языка/грамматики:** нет
- **Новые аксиомы ОС (SPEC §7):** нет
- **Зеркало в selfhost:** не требуется (пример на EATLang)
- **Опирается на:** битовые операторы ✅ (реализованы в языке)
- **Блокеры:** нет
- **Блокирует:** —

Задание для агента-исполнителя. Цель — второй крупный пример в
`examples/`, замеряющий пригодность языка для криптографии: битовая
арифметика фиксированной ширины, модульное сложение без кучи и без
типов шире 32 бит, проверяемость через официальные тест-векторы.

Проект **не требует изменений компилятора** — это чистый EATLang поверх
уже добавленных операторов `& | ^ << >>`. Никакой работы в `src/eatc/`
или `selfhost/` быть не должно; если кажется, что нужна — остановись и
сообщи заказчику (это находка про язык, а не задача примера).

Все идиомы ниже **проверены на текущем компиляторе** — числа и
поведение реальны, не додуманы.

## 0. Что уже установлено эмпирически (не переоткрывай)

- **Hex-литералов нет.** Лексер читает только десятичные цифры. Все
  константы — decimal: `0xEDB88320` пиши как `3988292384`, `0x80` как
  `128`. Значения таблиц даны ниже уже в decimal.
- **Индексация строкового литерала работает:** `s[i]` на `str<N>`
  возвращает байт; `u8(s[i])` даёт код. Это упрощает тесты, но для
  сообщений > нескольких байт всё равно удобнее `[u8; N]`-массивы.
- **`u8('a')` тотален** (char↔u8) — короткие сообщения удобно набирать
  символьными литералами.
- **Битового NOT (`~`) нет.** `~x` для u32 — это `x ^ 4294967295`.
- **Нет типа шире u32 и нет беззнакового вычитания с заёмом.**
  `0 - x` при `x>0` — trap переполнения. Отрицание mod 2³² пиши через
  условие, а не `0 - x` (см. CRC ниже).

## 1. Центральная проблема и её решение (ядро всего проекта)

Криптография требует арифметики **mod 2³² с заворотом**. EATLang же
**трапит** и переполнение `+`, и вынос битов из `<<`. Типа `u64` нет,
поэтому «посчитать и потом замаскировать» для сложения не выйдет —
промежуточная сумма трапит раньше.

Решение — два helper'а в модуле `Words.eat`. **Оба проверены:**

### Модульное сложение (32-битное, через 16-битные половины)

```eat
func add32(a: u32, b: u32) -> u32
    requires true
    ensures true
{
    const lo: u32 = (a & 65535) + (b & 65535)
    const hi: u32 = (a >> 16) + (b >> 16) + (lo >> 16)
    return ((hi & 65535) << 16) | (lo & 65535)
}
```

Проверено: даёт точный результat mod 2³² (`add32(4000000000,
4000000000) == 3705032704`) и **верификатор доказывает его полностью
свободным от trap'ов** — ни одной остаточной рантайм-проверки. Каждое
модульное сложение в SHA/ChaCha идёт через `add32`. Вложенность
`add32(add32(add32(...)))` укладывается в лимит глубины выражений (32).

### Ротации (циклический сдвиг)

Прямой `x << k` для u32 с установленными старшими битами **трапит вынос
битов** — поэтому ротацию нельзя писать «в лоб». Маскируй операнд до
сдвига:

```eat
# Циклический сдвиг влево на константу (пример: 16).
func rotl16(x: u32) -> u32
    requires true
    ensures true
{
    return ((x & 65535) << 16) | (x >> 16)
}

# Общий вариант (величина — параметр).
func rotl(x: u32, n: u32) -> u32
    requires n >= 1 and n <= 31
    ensures true
{
    const one: u32 = 1
    const k: u32 = 32 - n
    const mask: u32 = (one << k) - one
    return ((x & mask) << n) | (x >> k)
}
```

`rotr(x, n)` симметрично: `((x & maskLow_n) << (32 - n)) | (x >> n)`, или
просто `rotl(x, 32 - n)`.

**Замер (проверен):** константные ротации (`rotl16` и т.п.) доказываются
верификатором полностью. Общий `rotl` с переменным `n` оставляет **ровно
одну** рантайм-проверку переполнения на `<< n` — верификатор не
связывает ширину маски `32−n` с величиной сдвига `n`. Корректности это
не нарушает (значения всегда влезают), но снижает процент доказанного.

**Рекомендация:** величины ротаций в SHA-256 и ChaCha20 —
**компайл-тайм константы**, поэтому реализуй ротации специализированными
функциями под каждую константу (набор мал, см. §3–§4). Это даёт ~100 %
доказанных сдвигов и переполнений. Общий `rotl`/`rotr` оставь в
`Words.eat` для полноты и как демонстрацию, но в горячем коде не
используй. В комментарии к общей версии зафиксируй этот trade-off — это
одна из измеряемых величин примера.

### Прочие идиомы Words.eat

- `xor3(a,b,c) -> u32` = `a ^ b ^ c` (тройной XOR для Σ/σ — читаемость).
- Загрузка/выгрузка big-endian:
  `load_be32(b0,b1,b2,b3) = (b0 << 24) | (b1 << 16) | (b2 << 8) | b3`
  (все операнды u32, `b0` ≤ 255 → `b0 << 24` ≤ 0xFF000000, доказуемо
  без переполнения). Выгрузка: `(w >> 24) & 255`, `(w >> 16) & 255`, …

## 2. Ограничения языка, влияющие на структуру (учти заранее)

- **≤ 60 statements на функцию.** Таблицы из 64 записей в один
  if/elif-каскад не влезут — но 64-элементный **литерал массива —
  это один statement**: `const k: [u32; 64] = [1116352408, …]`. Именно так
  задавай таблицы K (SHA) и раунд-константы.
- **≤ 6 параметров.** Восемь рабочих переменных SHA (a..h) нельзя
  передать в отдельную функцию раунда. Держи их **локальными `let` в
  функции компрессии** (8 переменных) и обновляй прямо в теле цикла
  `for t in 0..64`, либо в `[u32; 8]`. Функции Σ/Ch/Maj/σ принимают
  1–3 аргумента — они помещаются.
- **`for a..b` — только константные границы**; `for x in arr` — по
  массиву. Оба цикла ограничены — это ровно то, что нужно (64 раунда,
  16 слов расписания и т.п.).
- **`loop` — только в `main`, один.** Потоковое чтение stdin —
  единственный `loop` в `Main.eat`.
- **Рекурсии нет** — граф вызовов должен быть DAG. Все алгоритмы
  итеративны, проблем нет.
- **Контракты опциональны** (правило 5, форма переопределена — DX_PLAN
  §6.5): пропущенная клауза ≡ `true`. Где инвариант невыразим — просто
  опускай. Где выразим (`n <= 31`,
  `idx < 64`) — записывай: это повышает процент доказанного и служит
  документацией.

## 3. Модуль SHA-256 (`Sha256.eat`)

Потоковый хешер как struct — не ограничен `str<256>`, хеширует ввод
любой длины поблочно.

```
struct Sha256 {
    h: [u32; 8]        # текущее состояние хеша
    block: [u8; 64]    # буфер текущего 512-битного блока
    fill: u32          # сколько байт в block
    total: u32         # всего обработано байт (для padding; младшие 32 бита)
    k: [u32; 64]       # константы раундов
}
```

Функции:
- `sha256_new() -> Sha256` — инициализирует `h` начальными значениями
  (ниже), `k` — 64-элементным литералом, счётчики в 0.
- методы `update(let self, byte: u32)` — кладёт байт в `block`, при
  заполнении 64 вызывает `compress`; `total += 1`.
- `compress(let self)` — ядро; см. ниже.
- `finish(let self)` — добавляет `0x80`, паддинг нулями, 64-битную длину
  в битах (big-endian; старшие 32 бита = 0 для сообщений < 2²⁹ байт,
  младшие = `total * 8` mod 2³², считать через `add32`/сдвиги),
  вызывает `compress` на последнем блоке(ах).
- `digest_byte(self, i: u32) -> u8 requires i < 32` — i-й байт итогового
  хеша: `(self.h[i/4] >> (24 - 8*(i%4))) & 255`.

Логические функции (специализированные ротации-константы):
- `ch(e,f,g) = (e & f) ^ ((e ^ 4294967295) & g)`
- `maj(a,b,c) = (a & b) ^ (a & c) ^ (b & c)`
- `bsig0(a) = rotr2(a) ^ rotr13(a) ^ rotr22(a)`
- `bsig1(e) = rotr6(e) ^ rotr11(e) ^ rotr25(e)`
- `ssig0(x) = rotr7(x) ^ rotr18(x) ^ (x >> 3)`
- `ssig1(x) = rotr17(x) ^ rotr19(x) ^ (x >> 10)`

Нужный набор `rotrN` (каждый — 2 строки, как `rotl16` из §1, но
`((x & maskLow_N) << (32-N)) | (x >> N)`):
N ∈ {2, 6, 7, 11, 13, 17, 18, 19, 22, 25}. `x >> 3` и `x >> 10` —
прямые правые сдвиги (тотальны).

Компрессия (`for t in 0..64`), всё сложение — `add32`:
```
t1 = add32(add32(add32(add32(h, bsig1(e)), ch(e,f,g)), self.k[t]), w[t])
t2 = add32(bsig0(a), maj(a,b,c))
h=g; g=f; f=e; e=add32(d,t1); d=c; c=b; b=a; a=add32(t1,t2)
```
Расписание `w: [u32; 64]`: первые 16 слов — из `block` big-endian
(`load_be32`); для `t in 16..64`:
`w[t] = add32(add32(add32(ssig1(w[t-2]), w[t-7]), ssig0(w[t-15])), w[t-16])`.
Следи за лимитом 60 statements: вынеси расписание в отдельный метод
`schedule(let self, ...)` если тело компрессии распухнет.

## 4. Остальные модули

### `Chacha20.eat` (зависит от `Words.eat`)
Блочная функция ChaCha20 (RFC 8439 §2.3). Состояние — `[u32; 16]`:
константы `"expand 32-byte k"` (1634760805, 857760878, 2036477234,
1797285236), 8 слов ключа, счётчик, 3 слова nonce. 20 раундов =
10×(4 столбцовых + 4 диагональных) quarter-round. Quarter-round:
```
a = add32(a,b); d = d ^ a; d = rotl16(d)
c = add32(c,d); b = b ^ c; b = rotl12(b)
a = add32(a,b); d = d ^ a; d = rotl8(d)
c = add32(c,d); b = b ^ c; b = rotl7(b)
```
Ротации только константные — специализируй `rotl16/12/8/7`. На выходе
блока — `add32` исходного состояния с рабочим (16 слов).
**Тест-вектор:** RFC 8439 §2.3.2 (key=00..1f, nonce=00000009 0000004a
00000000, counter=1) → известный keystream-блок из 16 u32. Сверяй слова.

### `Crc32.eat` (независим)
CRC-32/IEEE, table-driven. Таблица `[u32; 256]` строится в `crc_new`
двойным циклом. **Внимание — отрицание mod 2³²:** классический трюк
`poly & -(crc & 1)` использует беззнаковое `-`, которого нет. Пиши
через условие:
```
if (crc & 1) == 1 {
    crc = (crc >> 1) ^ 3988292384    # 0xEDB88320
} else {
    crc = crc >> 1
}
```
Итог: `crc = crc_final ^ 4294967295` (финальная инверсия через XOR),
начальное `crc = 4294967295`.
**Тест-векторы:** `""` → 0; `"123456789"` → 3421780262; `"The quick
brown fox jumps over the lazy dog"` → 1095738169.

### `Base64.eat` (независим)
Кодирование: 3 байта → 4 сектета (`b0 >> 2`, `((b0 & 3) << 4) | (b1 >> 4)`,
…), алфавит A–Za–z0–9+/ через `char(u8(...))`, паддинг `'='`. Декод —
обратно, с валидацией. Работает на u8/u32 и битах.
**Тест-векторы:** `""`→`""`, `"f"`→`"Zg=="`, `"fo"`→`"Zm8="`,
`"foo"`→`"Zm9v"`, `"foobar"`→`"Zm9vYmFy"`.

### `Hmac.eat` (зависит от `Sha256.eat`)
HMAC-SHA256, размер блока 64. Ключ > 64 байт → сначала хешируется;
`ipad`=0x36 (54), `opad`=0x5c (92). `HMAC = H((K⊕opad) ∥ H((K⊕ipad) ∥ m))`.
**Тест-вектор (RFC 4231 case 1):** key = 20 байт 0x0b, data = `"Hi There"`
→ `b0344c61d8db38535ca8afceaf0bf12b881dc200c9833da726e9376c2e32cff7`.

## 5. `Main.eat` — точка входа

`sha256sum`-подобная утилита: читает **все** байты stdin потоково,
печатает строчный hex SHA-256 (32 байта → 64 hex-символа).

```
loop {
    match read_byte() {
        Ok(b) { h.update(u32(b)) }
        Err(_) { break }
    }
}
h.finish()
# печать 64 hex-символов через hex_digit/write_hex8
```
Функции `hex_digit`/`write_hex8` возьми из `examples/mos6502/Main.eat`
(там уже есть готовый образец печати hex).

## 6. Раскладка и сборка

```
examples/crypto/
  Words.eat      # add32, rotl/rotr, load/store be32, xor3
  Sha256.eat     # struct Sha256, compress, update/finish
  Hmac.eat
  Crc32.eat
  Base64.eat
  Chacha20.eat
  Tests.eat      # все тест-векторы в test-блоках
  Main.eat       # sha256 stdin -> hex
  README.md
```
Порядок модулей при компиляции: `Rt.eat` первым, затем зависимости
раньше зависимых (`Words` → `Sha256` → `Hmac`), `Main` — последним.

Makefile (по образцу `run_mos6502` и блока `verify`):
```make
CRYPTO = $(RT) examples/crypto/Words.eat examples/crypto/Sha256.eat \
    examples/crypto/Chacha20.eat examples/crypto/Crc32.eat \
    examples/crypto/Base64.eat examples/crypto/Hmac.eat \
    examples/crypto/Tests.eat examples/crypto/Main.eat

run_crypto:
	printf 'abc' | $(EATC) run $(CRYPTO)
```
И добавь в цель `verify` сверку interp == native, и в неё же —
эталонную проверку против системного `sha256sum`:
```sh
printf 'The quick brown fox' | ./build/Main
printf 'The quick brown fox' | sha256sum   # должны совпасть
```

## 7. Критерии приёмки

1. `make check` (или `eatc check` по всем модулям) — все `test`-блоки
   зелёные: SHA-256 (3 вектора), HMAC (1), CRC32 (3), Base64 (5),
   ChaCha20 (1).
2. `eatc build … -o build/Crypto` — собирается; вывод бинарника на
   стороннем вводе совпадает с `sha256sum` и с интерпретатором
   (`interp == native`).
3. Никаких правок в `src/eatc/` и `selfhost/`.
4. В `README.md` примера — раздел «что показал про язык», по образцу
   `examples/mos6502/README.md`: обязательно зафиксируй (а) как решались
   модульное сложение и ротации без u64, (б) итоговый процент
   доказанного верификатором с разбивкой (`eatc build` печатает его),
   (в) любые упёртые ограничения (кандидаты: отсутствие `~`,
   беззнакового вычитания, hex-литералов, типа шире 32 бит).
5. Прогони полный `make verify_suite verify_selfhost verify_bootstrap` —
   они не должны сломаться (пример их не трогает, но это страховка, что
   добавленные `.eat` не ломают дифф-сверку самого компилятора: harness
   гоняет self-hosted фронтенд по всем `.eat` репозитория).

## 8. Порядок работ (рекомендуемый)

1. `Words.eat` + микротест add32/rotl на `/tmp` (значения — из §1).
2. `Sha256.eat` + `Tests.eat` (только SHA) → зелёные 3 вектора. Это 80 %
   ценности; остановись и покажи заказчику промежуточный результат.
3. `Main.eat`, сверка с `sha256sum`.
4. `Hmac.eat` (переиспользует Sha256), `Crc32.eat`, `Base64.eat`,
   `Chacha20.eat` — каждый со своими тестами.
5. Makefile-таргеты, README, финальный прогон всех сьютов.
6. Коммиты по модульным вехам (SHA-ядро отдельно от остального), стиль
   сообщений — как в истории репозитория (snake_case заголовок).
