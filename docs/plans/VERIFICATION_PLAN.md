## Ключевой принцип: модульные контракты

Вызывающий **всегда** вправе предполагать `ensures` вызванной функции,
даже если тот не доказан статически: недоказанный `ensures`
компилируется в runtime-проверку внутри функции, поэтому любой
завершившийся вызов ей удовлетворяет. Если функция с багом нарушает
контракт — она trap'ается до возврата, и предположение вызывающего
не наблюдается ложным. Это даёт SPARK-стиль модульного рассуждения:
функции доказываются по одной, композиция — через контракты.

Симметрично `requires`: внутри функции предпосылка — факт (проверена
вызывающим или trap'ом на входе).

**Граница модулей (MODULES_PLAN §6, этап 2)**: внутри одного модуля
вызов дополнительно пропускает body-summary (интервал результата или
тождество параметру, выведенные из тела). Через границу модулей —
только объявленный контракт: интервал результата = диапазон типа,
обрезанный конъюнктами `ensures` (`_contract_iv`). Интерфейс модуля =
его контракты; реализация меняется свободно, доказательства клиентов
не зависят от тел. Кейсы 33/34. Ограничение: дизъюнктивный `ensures`
(`result <= 15 or result == 99`) не конъюнкт — интервал через границу
не режет; лечится усилением контракта у экспортёра.

## Арсенал верификатора (реализовано)

1. Интервалы путей (`имя`, `self.поле`, `var.поле`) с уточнением по
   requires, условиям ветвей, assert и `!= const` (подрезка границ,
   nz-факты для делителей).
2. Разностные ограничения путей `p <= q + d` (Флойд–Уоршелл):
   условия ветвей и requires (в т.ч. со смещением, `i + 1 < n`),
   присваивания выражений (`let j = i + 1` → факт `j == i + 1`);
   символический `result`; разложение `p ± c` (off-by-one);
   реляционный пол вычитания (`a - b >= 0` при `b <= a`).
3. Структурное равенство выражений (коммутативность `+`/`*`),
   решает `==`/`<=`/`>=` для `result` против выражения return.
4. Ускорение циклов: единственное обновление `p = p ± const` даёт
   точный интервал `v0 + k·d` в теле и после цикла; охрана `p < C`
   ограничивает результат даже при избытке итераций; счётчик
   с шагом +1 идёт в ногу с переменной цикла (`count == i`,
   у условного — `count <= i`).
5. Модульные контракты: ensures вызова — факты вызывающего,
   включая арифметику сторон (`result <= pos + width` при вызове
   `advance(p, 5)` даёт `next <= p + 5`); summary функций
   (интервал или тождество параметру).
6. Границы у целей присваивания (`pool[i] = ...`) — проверяются
   и доказываются наравне с чтением.
7. `char` как байт: интервал [0, 255], литерал — точное значение;
   касты `char` ↔ `u8` тотальны (обязательств не порождают) и
   прозрачны — пути, разложения и структурное равенство проходят
   сквозь каст.

## Чего не хватает до self-host

Верификатор: запланированные механизмы реализованы (кейсы 19–21):

- [x] отношения через присваивание выражений (`let j = i + 1` →
  факт `j == i + 1`) — кейс 19;
- [x] инвариант связи двух счётчиков в одном цикле (`i == count`) —
  кейс 20;
- [x] ensures с арифметикой в модульных фактах
  (`result <= pos + width`) — кейс 21.

Язык (кирпичи, без которых компилятор не написать, — верификация
к ним готова):

- [x] `char` ↔ `u8` (классификация символов лексером) — касты
  `u8(c)`/`char(b)` тотальны (один байт, no-op над i8) и прозрачны
  для верификатора: уточнение по `requires u8(c) >= 48` попадает
  на путь `c` — кейс 22;
- [x] `enum` с полезной нагрузкой (узлы AST) — вариант несёт 0/1
  значение (несколько — struct), конструктор `Node.Num(42)`,
  разбор — исчерпывающий `match` с привязкой; lowering
  `{tag, слот на вариант}`; тайпчекер отклоняет типы, содержащие
  себя по значению, — кейс 23;
- [x] байтовый I/O через `Result` — `read_byte()`/`write(s)`:
  компилятор работает фильтром stdin → stdout, файловых
  дескрипторов (ресурсов-хэндлов) в языке нет — кейс 24;
- [x] массивы struct'ов как пулы + `while`-эквивалент по префиксу
  (выражается `for` + охрана и `loop` + `break`, кейсы 04/05/24/25);
- [x] `var self` — мутирующие методы (пул токенов/AST как поле struct,
  `push` двигает курсор): получатель обязан быть `var`-переменной,
  алиасов нет (всё остальное by-value); верификатор сбрасывает факты
  о получателе и принимает ensures метода — кейс 25;
- [x] литерал заполнения массива `[0; 4096]` — пулы объявляются
  одной строкой (значение вычисляется один раз, заполнение циклом);
  ёмкость строк выровнена с рантаймом (`str<N>`, N ≤ 256);
- [x] модули (компилятор в нескольких файлах) — программа =
  упорядоченный список файлов с единым пространством имён
  (`eatc run/build a.eat b.eat main.eat`); ошибки атрибутируются
  файлу-источнику; пример `examples/modules/`, сверка в
  `make verify`.

Критерий готовности: каждый паттерн из будущего кода компилятора
имеет кейс в `tests/verify/` со статусом ✅, отрицательные кейсы
не деградируют. Набор обязателен к прогону перед каждым коммитом
в верификатор.

## Готовность к self-host: проба и остаточные риски

Проба — `examples/lexer/` (LexUtil + LexMain): фрагмент лексера на
EATLang, задействующий все кирпичи разом (read_byte-цикл до Eof,
классификация через `u8(c)`, enum + match, пул в struct c `var self`,
`[0; 1024]`, два модуля). Разбирает собственный исходник, 11/12
проверок доказано (в рантайме — только инкремент позиции в
неограниченном цикле), интерпретатор == бинарник (`make verify`,
шаг VERIFIED LexerProbe).

Вывод: кирпичи для полной имплементации self-hosting собраны.
Реализация начата: фаза 1 (полный лексер, `selfhost/`) готова и
сверяется с эталоном на каждом `.eat` репозитория —
см. `docs/SELFHOST.md`, `make verify_selfhost`.

Остаточные риски (не блокеры, решаются по ходу):

- битовые операции отсутствуют — эмиссия текстового LLVM IR их
  не требует;
- вывод идентификаторов из пула — посимвольный (`write("{char(b)}")`);
  медленно, но корректно; при нужде — builtin для среза пула;
- пулы компилятора (131 072 токена, 65 536 узлов) дают кадры в десятки мегабайт —
  следить за отчётом §8, при нужде поднять stack size линкером;
- рекурсии нет (правило 1) — парсер пишется с явным стеком
  (пул + loop), паттерн покрыт кейсами 04/05/25.

## Карта остатка и потолок интервального анализа (2026-07-14, разбор)

Разбор непроказанного по форме (bootstrap-вход, доказано 5456/6945)
показал: **оставшиеся классы — за пределом интервального/дырочного
анализа, быстрых звуковых побед нет.** Подтверждено экспериментами.

**bounds — 921** (из них Check.eat 471). Разбор по форме индекса:

- **259 банкованных `a / 65536`** (`x[a/65536][a%65536]`). Половина
  `a % 65536` уже доказана безусловно (`< 65536` = размер банка).
  Провал — в `a / 65536 < 2`, когда `a` — узловой индекс из пула
  `[0,131071] ∪ {NONE}` без охраны `!= NONE`: `NONE/65536 = 65535 ≥ 2`
  — реально вне границ, доказать **нельзя** (не пропагация — семантика
  «этот линк здесь не NONE», интервалам недоступна). Проверено:
  список-обход с явным guard'ом и `c` из пула доказывается **6/6**
  (§1.1 работает как задумано); валят только вложенные промежуточные
  чтения `nns[nfc[c]...]`, где промежуток не охранён. Снятие
  NONE-сентинела дало бы +82 bounds, но это **несостоятельно**
  (a==NONE — настоящий выход за границы);
- прочее (`f`, `sp`, `self.tres`…) — единичные локальные индексы без
  выводимого верхнего предела.

**assert — 0/103.** Гипотеза «суммы полей» (§1.2 OPTIMIZATIONS_PLAN)
**опровергнута разбором**: доминируют `assert <вызов>(...)` в
test-блоках (`is_space(32)`, `hex_val('9')==9`, `same(a,b)`,
`parse_status(...)==N`) — верификатор бракует любой assert с вызовом
(`_has_call`). Сумм полей (`self.ln + n <= 256`) ~4 штуки. Доказать
основную массу = **вычислить вызов на компиляции** (comptime, §5
COMPTIME_PLAN), а не разностные инварианты полей. Направление §1.2
переоценено: не окупается интервалами.

**Вывод.** Верификатор у звукового потолка интервального анализа для
этого кода: дальше — либо comptime (assert-с-вызовом), либо
семантическое моделирование не-NONE линков дерева (research-grade,
или охраны в самом selfhost — большой паритет-этап). Быстрых
verifier-only побед не осталось.
