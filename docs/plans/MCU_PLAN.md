- **Зеркало в selfhost:** `extern` уже зеркалён; §5–6 компиляторы не трогали
- **Опирается на:** языковой этап МК (✅, ранняя фаза этого же плана)
- **Блокеры:** —
- **Блокирует:** —

Цель: на EATLang можно вести реальный embedded-проект — логика на
языке со всеми гарантиями (тотальность, Power of 10, §8), граница с
железом узкая и явная, распространённые платы ARM Cortex-M
поддерживаются из коробки, сборка и сверка — одной командой.

## §1. Решения пользователя (2026-07-13)

1. **Периферия — через extern-функции**: язык получает объявление
   внешних C-функций (расширяемые аксиомы). Драйверы — на C рядом с
   проектом, логика — на EATLang. MMIO-встроенные — возможный
   следующий шаг, отдельным решением.
2. **Железо — охват распространённых моделей** ARM Cortex-M:
   RP2040 (M0+), STM32F1 (M3), STM32F4 (M4), nRF52 (M4).
   ESP32 — не ARM (Xtensa/RISC-V), вне этапа.
3. **Модель исполнения — super-loop**; прерывания — в C-шиме
   (анализ в §2: ISR на EATLang ломает «нет глобалов» и верификатор).
4. **Живой UART-ввод — через extern** (`uart_poll()` и т.п.);
   протокол конца потока для аксиомы `read_byte` не нужен,
   прошиваемый вход остаётся для batch-сверки.
5. **Синтаксис**: `extern func имя(арг: тип, …) [-> тип]` без тела,
   верхний уровень модуля (методов-extern нет).
6. **Типы на границе**: скаляры (u8/u32/i32/bool/char) и массивы
   `[u8; N]` (по указателю — батчевые обмены SPI/DMA).
7. **Контракты**: `requires` проверяется у вызывающего (верификатор
   и рантайм как обычно); `ensures` принимается на веру и даёт
   факты верификатору — ровно как у аксиом (`read_byte <= 255`).
8. **Интерпретатор**: вызов extern — чёткая ошибка («extern доступен
   только в бинарнике»); сверка МК-программ — через QEMU;
   test-блоки без extern-вызовов работают как раньше.
9. **Типы (решения 2026-07-13, обсуждение extern)**: добавить **u16**
   (повторяет машинерию u8: casts, overflow-trap, интервалы
   0..65535); на границе extern разрешить массивы всех скаляров
   (`[u8|u16|u32; N]` — ADC/DMA-буферы, таблицы регистров).
   **u64 — отдельной фазой после extern** (время без заворота,
   задел SHA-512 трека 5); float/структуры на границе/opaque-хэндлы —
   не делаем (fixed-point поверх целых, обёртки, u32-индексы).
10. **hex-литералы (`0x…`) и битовое NOT (`~x`) — в этот этап**
   (закрывают два пункта трека 5; в драйверах маски и адреса —
   каждая вторая строка).

## §2. Прерывания и 10 правил (анализ)

ISR на EATLang невозможен без слома базы: у обработчика нет
вызывающего, единственный канал связи с `main` — разделяемое
состояние вне стека, а в EATLang вся память живёт в кадрах —
«нет глобалов» (правило 6/Power of 10). Открыть глобалы = обход
правила 4 AGENTS.md. Вдобавок доказательства верификатора стоят на
однопоточности: асинхронная запись убивает факты в каждой точке.

Решение: **прерывание — деталь драйвера**. ISR на C кладёт
байты/события в кольцевой буфер шима (single-producer/
single-consumer), EATLang-суперцикл забирает их через extern.
Латентность даёт ISR, гарантии языка целы. ISR на EATLang —
исследовательский трек (§7): потребует разделяемых объектов с
атомарной семантикой и правил переноса фактов через точки
прерывания.

### Фаза extern

Самая инвазивная часть — оба компилятора, полный паритет.

- SPEC: §4 грамматика (`extern func` без тела), §5 семантика
  (граница доверия: тотальность extern — на вере, как у аксиом;
  результат нельзя молча отбросить), §7 (связь с аксиомами).
- Символ = имя как написано (без `eat_`-префикса); checks запрещают
  коллизии со встроенными, аксиомами (`eat_*`), `main`.
- Стек §8: extern — лист DAG с нулевым кадром, в отчёте пометка
  (как аксиомы сейчас).
- Python-бутстрап: lexer (ключевое слово) → parser → checks →
  typechecker → verifier (requires у вызывающего, ensures — факты)
  → interpreter (ошибка при вызове) → codegen (declare + call).
- Зеркало selfhost: Tok/Lexer/Ast/Parser/Check/Ir — байт-в-байт
  по всем дампам; порядок и кодировки — docs/MODIFYING.md.
- Тесты: lex/parse/sig/typed/ir-дампы c extern; verify_suite —
  кейсы requires/ensures на extern; негативные (тело у extern,
  extern в интерпретаторе, коллизия имён, str в границе).

## §4. Фаза 2: структура портов mcu/ — **сделано 2026-07-14**

- `mcu/boards/<board>/`: `board.ld` (память чипа), `board.c`
  (UART/периферия платы, ISR-буферы), `board.mk` (--target, -mcpu,
  QEMU-машина или команда прошивки).
- Общий `mcu/startup.c` (вектора, .data/.bss, вызов main) и
  `mcu/eabi.c` (__aeabi_*) — выделить из runtime_mcu.c.
- `make mcu BOARD=<board> SRC="…" [EXTERN="drv.c"]` — сборка
  прошивки из любых модулей с пользовательскими драйверами;
  `make mcu_run BOARD=… `/`make mcu_verify BOARD=…` для QEMU-плат.
- Автосверка §8 на сборке: стек из отчёта + .data + .bss против
  RAM платы из board.mk — переполнение = ошибка сборки, а не
  сюрприз на железе.

## §6. Фазы 4–5: периферийный шим и флагманский пример — **сделано 2026-07-16**

- Шим платы: `gpio_set/gpio_get`, `systick_ms`, `uart_poll`
  (ISR → кольцо → extern), при надобности SPI/I2C — контракты в
  EATLang-обёртках (`mcu/Mcu.eat`: requires на номера линий,
  Result для опросов). Реализация: `mcu/common/shim.c` (слабые
  символы поверх хуков платы, SysTick с накоплением тактов в u64,
  кольцо SPSC), `mcu/common/startup.c` — полная таблица векторов и
  диспетчер `board_irq(irq)` по IPSR; `poll_byte() -> Option<u8>`
  прячет сентинел 256. Аксиомы SPEC §7 не понадобились — граница
  целиком на `extern` (решение §1.1).
- Флагманский пример `examples/blinky_cli/`: суперцикл `loop`
  (SPEC §5.1) — мигание по SysTick, кнопка, UART-команды
  (o/f/b/s/t/i/q); один исходник собирается на все 8 портов;
  на 4 QEMU-платах — автосверка вывода байт-в-байт с хостовым
  эталоном (`host_driver.c`), вывод зависит только от команд.
- Числа в FINDINGS (итерация 34): флеш 7.5–8.4 КБ и RAM ~1.5 КБ по
  8 платам, латентность такта ~1.1–1.7 мкс (QEMU).

## §7. Исследовательские хвосты (вне этапа)

- ISR на EATLang — см. §2, нужны разделяемые объекты в SPEC.
- hex-литералы и `~x` (трек 5) — станут заметно нужнее в драйверах.
